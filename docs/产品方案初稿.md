# Drama Pulse 产品方案初稿

## 1. 文档目的

本文档用于沉淀 `Drama Pulse` 比赛项目的第一版正式产品方案，作为后续功能设计、技术拆解、开发排期、答辩表达的统一依据。

当前版本重点解决以下问题：

- 明确产品定位和比赛目标
- 冻结 MVP 范围和主闭环
- 明确高光互动与尾集分支两条产品主线
- 统一客户端、服务端、AI 处理链路的职责分工
- 为后续技术设计文档和排期文档提供输入

## 2. 项目背景理解

根据比赛需求，项目核心不是做一个普通短剧播放器，而是围绕短剧观看过程中的情绪高峰和剧情参与感，设计出一套即时互动产品机制。

需求文档强调了三个核心方向：

- 围绕剧情高光点设计即时互动能力
- 围绕剧情尾部设计剧情分支或拓展能力
- 完成客户端与服务端的业务闭环，并能够稳定演示

因此，产品目标不能只停留在“能播放视频”和“能调模型”，而应该让评委感受到：

- 用户在看剧时真的更容易表达情绪
- 用户和内容之间真的有更深的参与感
- 项目具备完整的前后端闭环与可扩展空间

## 2.1 互动激发定义

本项目中的“互动激发”不是简单地给用户一个评论框或弹幕输入框，而是指系统主动识别剧情高光，在用户最有表达欲的时间点提供低成本互动入口，从而激发用户完成情绪表达和剧情参与。

该定义对应题目中的两个核心问题：

- 传统评论和弹幕输入成本高
- 文本输入会中断短剧观看节奏

因此，Drama Pulse 的产品机会点可以概括为：

`不替代评论区，而是补足剧情瞬间情绪表达的空白`

## 2.2 用户与场景分析

### 目标用户

主要目标用户：

- 高频观看短剧的用户
- 偏好爽点、反转、撒糖、复仇、悬念内容的用户
- 有即时情绪表达欲，但不愿意频繁打字的用户
- 喜欢围观他人反应、获得陪伴感的用户

次要目标用户：

- 喜欢站队、玩梗、参与剧情判断的用户
- 对剧情续写、二创和 AIGC 内容感兴趣的用户
- 愿意在尾集后继续消费衍生内容的用户

### 使用场景

场景一：剧情高光瞬间

- 用户看到打脸、反转、身份曝光、甜蜜撒糖时，情绪瞬间被点燃
- 用户想表达，但不想暂停或打字
- 产品机会是提供一键互动按钮，在最短路径内完成情绪释放

场景二：剧情冲突与站队

- 用户看到男女主误会、反派挑衅、家庭冲突时，会天然形成立场
- 产品机会是提供轻量站队或助攻型互动，让用户表达判断而非输入长评论

场景三：追到尾集后意犹未尽

- 用户看完尾集后，往往会继续想象“如果换一种发展会怎样”
- 产品机会是提供固定分支和自定义分支，让用户继续参与剧情延展

场景四：二刷或后续观看

- 用户通过历史弹幕和热度表现，感知“别人也在这里上头过”
- 产品机会是通过历史互动回流制造陪伴感和群体情绪共振

## 2.3 现有表达方式分析

短剧用户当前的表达方式主要有弹幕、评论、点赞、分享和投票，但它们都不能完整覆盖“剧情瞬间情绪表达”这一需求。

| 表达方式 | 优点 | 问题 | Drama Pulse 的改进 |
| --- | --- | --- | --- |
| 弹幕 | 即时、有陪伴感 | 需要输入文字，容易遮挡画面，质量不稳定 | 通过预设互动降低输入成本，并控制展示节奏 |
| 评论 | 能承载深度观点 | 多发生在观看后，与剧情时间点脱节 | 将互动绑定到具体高光点 |
| 点赞 | 操作简单 | 表达单一，无法体现“为什么点” | 按高光类型提供差异化情绪表达 |
| 分享 | 有传播价值 | 操作链路长，不适合即时表达 | 在播放中完成轻量情绪反馈 |
| 投票 | 有参与感 | 往往与播放流程割裂 | 融入高光点或尾集分支节点 |

因此，Drama Pulse 的产品价值主张不是“做一个更花哨的弹幕系统”，而是：

- 用剧情语义驱动互动入口
- 用低输入门槛承接即时情绪
- 用历史互动回流放大群体共鸣

## 3. 产品定位

`Drama Pulse` 是一款面向短剧观看场景的 Android 互动播放器。产品通过 AI 辅助识别剧情高光，并在播放过程中提供低打断、高反馈、强视觉冲击的互动能力；在尾集结束后，通过固定分支和自定义 AIGC 分支，让用户继续参与剧情延展。

产品定位关键词：

- 快节奏
- 强反馈
- 低输入门槛
- 不打断观看
- 有情绪爆发感
- 有剧情参与感

## 4. 产品目标

### 4.1 比赛目标

- 满足需求文档中的必选功能要求
- 在完整性、技术实现、创新性三个主要评分项中形成稳定优势
- 让 Demo 具有较强的现场感和演示张力

### 4.2 用户目标

- 用户无需打字，也能在关键剧情点快速表达情绪
- 用户能够感知“别人也在这个地方上头”，形成陪伴感
- 用户在尾集后还能继续“玩剧情”，增强停留和记忆点

### 4.3 产品策略目标

- 主闭环优先稳定
- 互动体验优先强感知
- AI 能力优先服务闭环，不喧宾夺主
- 固定内容保底，自定义生成加分

## 5. 最终产品形态

当前确认的最终产品形态如下：

- 客户端平台：`Android`
- 终端形态：`仅竖屏`
- 服务端形式：`本地部署 Local Server`
- 内容互动：`高光剧情互动` 和 `尾集分支互动` 两条主线同时实现
- 主打剧：`荒年全村啃树皮，我有系统满仓肉`
- 备选剧：`撕夜`
- 高光来源：`AI 离线提取 + 人工校准`
- 触发机制：`基于时间轴触发`
- 互动规则：`每个高光点 1 个主互动 + 1 个辅助反馈`
- 用户互动展示：`历史互动沉淀展示，不做强实时多人`
- 尾集分支：`固定分支视频保底 + 自定义异步生成`
- 分支结果互动：`点赞和评论均支持`
- 管理能力：`完整功能的本地 Web 管理后台`

## 6. 核心产品闭环

产品主闭环定义为：

`短剧列表 -> 剧集播放 -> 高光点触发互动 -> 互动结果上报 -> 历史互动沉淀展示 -> 尾集分支入口 -> 固定/自定义分支结果回看`

该闭环覆盖了需求文档中的以下内容：

- 短剧列表和播放基础能力
- 高光点打标、存储、下发
- 高光互动能力
- 用户互动沉淀展示
- 尾集剧情分支或拓展能力
- 本地服务端部署

## 6.3 产品边界说明

第一版只做尾集入口，意思是：

- 只有当一部短剧播放到最后一集时，才展示分支入口
- 暂不在中间剧情的关键节点插入分支入口
- 先把“尾集看完后的延展体验”做完整，避免入口过多导致实现和演示复杂

后续版本如果需要，可以再扩展到“关键剧情节点分支”。

## 6.1 用户核心路径

### 路径一：高光互动路径

`打开短剧列表 -> 选择短剧 -> 进入播放页 -> 命中剧情高光 -> 弹出轻量互动组件 -> 用户点击情绪按钮 -> 获得视觉反馈 -> 看到历史观众反应 -> 继续观看`

对应用户感受：

- 我刚好想表达的时候，系统给了我一个不需要打字的入口
- 我点完以后有明显反馈，感觉这个高光被“放大”了
- 我还能看到别人也在这里有类似反应，有一种一起看剧的感觉

### 路径二：尾集分支路径

`用户看完尾集 -> 系统展示分支入口 -> 用户选择固定方向 -> 查看预置分支结果 -> 或输入自己的剧情想法 -> 系统创建自定义任务 -> AI 生成剧情文本/分镜结果 -> 用户回看、点赞或评论`

对应用户感受：

- 我不只是看完了这个故事，还可以继续参与这个故事的后续走向
- 固定分支能立即满足好奇心，自定义分支则提供更强的参与感

## 6.2 Demo 演示路径

为了确保答辩演示有节奏、能稳定展示亮点，建议 Demo 路径固定为：

1. 从短剧列表进入主打剧集
2. 演示基础播放与高光时间轴标记
3. 在一个强高光点触发互动，展示按钮、特效、历史弹幕回流
4. 再演示一个不同类型高光，体现不同类别组件差异
5. 播放到最后一集结束，展示固定分支选项
6. 演示固定分支结果
7. 演示自定义分支输入、任务创建与结果回看
8. 最后切到本地 Web 管理后台，展示高光标签、互动统计与任务状态

## 9. 高光互动产品方案

## 9.1 高光能力定位

高光互动是本项目的主线功能，也是需求文档中最明确的必选方向之一。

它承担三个任务：

- 将剧情高光识别为可消费的互动时刻
- 让用户快速表达情绪
- 通过视觉反馈放大情绪感受

## 9.2 高光点来源

高光点不采用纯人工逐点配置，而采用如下处理链路：

1. 提取视频音频内容
2. 进行 ASR 识别得到台词文本
3. 对文本进行结构优化，包括人物、段落、情绪线索、冲突段识别
4. 通过大模型识别高光点、反转点、名场面、甜蜜场景等
5. 为高光点打上时间戳和类型标签
6. 人工对关键标签进行校准
7. 将最终标签存储到服务端，供播放时下发

该方案的优点：

- 比纯人工标注更省时间
- 比纯模型自动下发更可控
- 便于在答辩中体现 AI 在内容理解链路中的价值

## 9.3 高光标签类型

当前建议固定以下高光标签类型：

- `爽点`
- `反转`
- `搞笑`
- `甜蜜`
- `冲突`
- `悬念`
- `情绪爆发`

MVP 阶段无需过多扩展标签，否则前端类别表现和触发策略会变复杂。

## 9.4 高光标签字段建议

当前建议标签结构至少包含：

- 剧 ID
- 集 ID
- 开始时间
- 结束时间
- 交互窗口开始时间
- 交互真正出现时间
- 交互窗口结束时间
- 高光类型
- 标题文案
- 强度等级
- 互动类别与视觉表现 ID
- 来源类型
- 置信度

这套字段足够支撑 MVP 的触发、展示、统计与修正。

## 9.5 高光触发规则

当前产品规则已经确认：

- 高光以时间轴为触发基础
- 每个高光点只允许 `1 个主互动 + 1 个辅助反馈`

这样定义的原因：

- 避免同屏出现多个互动组件，破坏观看体验
- 降低客户端状态管理复杂度
- 让每个高光点的产品意图更明确

## 9.6 高光互动形式

主互动建议包括：

- 情绪按钮
- 站队按钮
- 一键助攻按钮
- 爽感触发按钮

辅助反馈建议包括：

- 粒子爆发
- 屏幕边缘光效
- 贴纸飞出
- 局部震动
- 弹幕短时增幅

主互动负责表达，辅助反馈负责放大情绪。

## 9.7 高光触发详细规则

触发时机：

- 当播放进度到达 `interaction_appear_ms` 时露出互动组件
- 用户实际可点击窗口为 `interaction_start_ms` 到 `interaction_end_ms`
- `start_time_ms` 到 `end_time_ms` 表示高光内容本体，不再和组件露出时间混用
- 若用户正在拖动进度条，不触发主互动
- 若视频处于暂停状态，不触发新的高光互动

触发频率：

- 同一播放会话中，同一高光点只触发一次主互动
- 用户手动回放时，可以展示历史互动，但不重复弹出强打断组件
- 两个高光点间隔过近时，两个点都保留，但后一个仅展示弱提示或轻反馈，不重复弹完整主互动组件
- 同屏最多存在一个主互动组件

展示时长：

- 主互动组件展示 `3 到 5 秒`
- 用户点击后立即收起主组件
- 特效持续 `1 到 2 秒`
- 历史互动提示持续 `2 到 4 秒`
- 未点击时自动淡出，不影响继续观看

选项设计：

- 每个高光点建议提供 `2 到 4 个` 互动选项
- 单个选项文案优先控制在 `4 到 6 个字`
- 情绪表达优先，不做复杂解释性文案

示例：

- 爽点：`爽了` / `打脸` / `继续虐他`
- 反转：`啊？` / `没想到` / `细思极恐`
- 甜蜜：`磕到了` / `锁死` / `太甜了`
- 冲突：`开怼` / `别忍` / `站女主`
- 悬念：`不对劲` / `有坑` / `继续看`

## 9.8 高光强度等级

为了让不同高光点的表现强度可配置，建议引入统一强度等级：

| 等级 | 定义 | 主互动 | 特效 | 弹幕密度 |
| --- | --- | --- | --- | --- |
| 1 | 轻微情绪点 | 不展示或弱展示 | 无 | 低 |
| 2 | 普通高光 | 展示 | 轻 | 低 |
| 3 | 明显高光 | 展示 | 中 | 中 |
| 4 | 强高光 | 展示 | 强 | 中高 |
| 5 | 爆点 | 展示 | 强爆发 | 高 |

强度等级的意义：

- 让高光点不再“一刀切”
- 便于客户端按配置切换动效和弹幕密度
- 便于后续运营调优和答辩解释

## 9.9 高光类别与交互表现

第一版不再让运营或前端直接选择“模板”，而是冻结为“类别一一对应表现”：

- `feel_good`
  - 对应 `爽` 字组件
- `reversal`
  - 对应 `卧槽` 字组件
- `conflict`
  - 对应火焰燃烧组件
- `sweet`
  - 对应爱心组件

强度规则：

- `intensity 1-2`
  - 轻量选项态或浮窗弹幕态
- `intensity 3`
  - 标准组件态
- `intensity 4-5`
  - 强视觉组件态

## 9.10 不打断观看的设计原则

- 高光互动组件不遮挡人物脸部和字幕核心区域
- 主互动展示时长有限，超时自动收起
- 用户不点击也不影响视频继续播放
- 互动反馈以轻量动画为主，不强制暂停视频
- 尾集分支只在最后一集结束后出现，不在普通播放中频繁插入
- 弹幕密度可控，避免覆盖核心画面
- 所有互动失败都不阻塞播放主流程

## 10. 历史互动沉淀方案

## 10.1 能力定位

本项目不做真正的实时多人联机互动，而是通过记录历史用户在高光点上的互动行为，并在后续观看中回流展示，制造“大家都在这个地方有反应”的群体感。

这比强实时方案更适合单人项目，且更稳。

## 10.2 数据流逻辑

1. 用户在高光点触发互动
2. 客户端将互动行为上报服务端
3. 服务端按剧集、时间点、高光点进行存储聚合
4. 后续用户播放到相同高光点时，下发历史互动数据
5. 客户端以弹幕、贴纸、热度表现等形式展示

## 10.3 展示形式

建议展示内容包括：

- 历史弹幕回流
- 热门情绪标签数量
- 历史高频贴纸
- 热度感知提示

展示策略建议：

- 高光到来时先弱展示历史热度和弹幕氛围
- 用户点击后再强化展示统计结果和群体反馈
- 历史互动存在感强于普通时段，但弱于主互动组件本身

产品价值：

- 提升陪伴感
- 放大高光点的情绪感染力
- 让互动不再是“一次性点击”

## 10.4 高光候选池回流

可以加入一条扩展逻辑：

- 某个非高光时间点若积累了大量用户互动
- 服务端可将其加入高光候选池
- 后续由人工或 AI 复核，决定是否纳入正式高光标签

该机制建议作为增强项，不建议在播放链路中动态改标签。

## 10.5 历史互动数据策略

历史互动数据来源建议明确区分为两类：

- `demoSeedData`：为演示准备的预置匿名数据
- `userGeneratedData`：真实用户互动上报数据

MVP 阶段允许使用预置演示数据模拟“历史沉淀”效果，但不应让 AI 生成的假弹幕冒充真实用户数据。

历史弹幕表达形式建议固定为：

- 基于互动选项映射出的历史互动短文案
- 不开放用户自由输入弹幕

这样可以更好控制质量、节奏和画面密度。

展示排序建议：

- 高频选项优先
- 最近互动优先
- 与当前高光类型更相关的内容优先
- 避免重复文案连续出现

## 10.6 热度等级与去重策略

建议将高光热度抽象为 `heat_level`：

- `0`：无明显互动
- `1`：少量互动
- `2`：普通热度
- `3`：高热度
- `4`：爆点

客户端可以根据 `heat_level` 动态调整：

- 弹幕密度
- 弹幕速度
- 特效强度
- 热门提示文案

基础去重和反垃圾规则建议包括：

- 同一设备在同一高光点可重复点击同一选项，但主统计只计一次
- 短时间内重复上报需要去重
- 异常高频点击不进入热度统计
- 评论类或自由文本内容需要敏感词过滤

## 11. 尾集分支 AIGC 方案

## 11.1 能力定位

尾集分支是本项目的创新支线，主要承担“剧情延展”和“互动想象空间”这两个目标。

它的定位不是取代主视频，而是在尾集结束后继续激发用户参与感。

## 11.2 入口时机

尾集分支入口建议只在以下场景出现：

- 当前剧集最后一集播放结束

避免在普通剧情中频繁打断。

## 11.3 分支模式定义

尾集分支采用双轨制：

- `固定分支`
- `自定义分支`

### 固定分支

产品目标：

- 稳定展示
- 确保答辩成功率

形式建议：

- 提供 2 个固定方向选项
- 每个方向对应一段预生成好的短视频结果

示例方向：

- 复仇反杀
- 隐藏真相

### 自定义分支

产品目标：

- 提供创新感和个性化能力
- 展示 AIGC 应用能力

形式建议：

- 用户输入一句想法
- 后端异步执行剧情扩写
- 生成分支剧情文本与分镜结果，后续可扩展到视频任务
- 结果完成后供用户回看

## 11.4 自定义生成策略

MVP 阶段不建议追求长视频全自动生成。

推荐定义为：

- 短分支
- 短片段
- 轻量结果

这样可以显著降低失败率，并提高答辩时的可控性。

## 11.5 自定义输入规则

为了保证生成质量和内容稳定性，建议自定义输入采取轻约束设计：

- 限制字数在 `20 到 60 字`
- 引导用户围绕“尾集后关键走向”进行输入
- 给出输入占位提示

示例提示语：

- 如果主角没有离开，之后会发生什么
- 如果反派反转获胜，结局会怎样
- 如果男女主再次相遇，会出现什么新冲突

## 11.6 异步任务机制

自定义分支必须采用异步机制：

1. 用户提交分支想法
2. 前端提示任务已创建
3. 后端异步处理剧情扩写和生成任务
4. 前端不阻塞主流程
5. 结果完成后进入历史分支查看

该机制可以避免生成任务影响播放体验和主流程稳定性。

## 11.7 失败兜底

自定义生成必须具备失败兜底：

- 生成超时
- 模型失败
- 结果不合规

兜底方式建议：

- 提示稍后再试
- 保留固定分支入口
- 保留已成功的历史分支记录

## 11.8 分支类型定义

为了让固定分支和自定义分支有更稳定的方向约束，建议将尾集分支抽象为以下几类：

`结局改写型`

- 用户想看到不同结局
- 例如：女主没有离开、男主提前知道真相、反派赢了

`爽点加倍型`

- 用户想把原剧中的反击或打脸继续放大
- 例如：主角彻底反杀、反派当众翻车、隐藏身份直接曝光

`情感补完型`

- 用户想补足原剧未充分展开的情感线
- 例如：误会解除后的生活、多年后的重逢、再次相遇后的冲突

`悬念延展型`

- 用户想继续探索未解释清楚的谜团
- 例如：幕后黑手是谁、那封信写了什么、身世秘密还有什么后续

## 11.9 自定义分支结果规范

自定义分支生成结果不应只是一段随意文本，而应统一为结构化结果，方便客户端展示和后续互动复用。

建议至少包含：

- 分支标题
- 一句话看点
- 剧情正文
- 分镜卡片
- 情绪标签
- 推荐互动按钮

结构示意：

```json
{
  "title": "她没有离开，而是回来了",
  "hook": "三年后，她以新身份重回宴会现场。",
  "story": "……",
  "storyboards": [
    {
      "scene": 1,
      "visual": "女主站在雨夜的车前，抬头看向灯火通明的酒店。",
      "dialogue": "这一次，该轮到我了。"
    }
  ],
  "tags": ["复仇", "反转", "爽点"],
  "interactionOptions": ["爽了", "继续反杀", "别放过他"]
}
```

该结构的意义在于：

- 结果更像“可展示内容”而不是调试输出
- 可反向驱动后续互动按钮和视觉表现
- 更方便答辩时演示 AI 生成的结构化价值

## 12. 客户端体验定义

## 12.1 体验关键词

短剧产品节奏快，互动体验必须符合短剧用户心理预期，不能做得像传统长视频社区。

体验关键词建议统一为：

- 快
- 爽
- 脆
- 炸
- 有存在感

## 12.2 弹幕定义

弹幕速度建议：

- 默认中高速
- 单条停留时间控制在 `1.8 到 2.5 秒`
- 高光时刻允许短时提速

弹幕密度建议：

- 普通时段低密度
- 高光时段中高密度
- 单屏同时可见建议 `6 到 10 条`

动画样式建议：

- 快速滑入
- 轻微弹射
- 爆发式出现
- 避免慢速漂浮

目标是让用户感知到“群体情绪爆发”，而不是传统直播弹幕的均匀飘屏。

## 12.3 贴纸和特效定义

不同高光类型建议绑定不同视觉风格：

- `爽点`：爆裂、火花、冲击光效
- `甜蜜`：心形、柔光、漂浮贴纸
- `反转`：翻牌、裂屏、反向冲击
- `搞笑`：夸张表情、抖动文字、拟声贴图
- `冲突`：红色脉冲、震动、碎片感
- `悬念`：暗角、聚焦、短暂停顿感

总体原则：

- 特效数量不宜太多
- 类别映射固定，但感知要强
- 优先保性能，再追求复杂炫技

## 12.4 文案风格

产品文案应统一为：

- 短
- 直接
- 不说教
- 不抢戏

示例：

- `这段太炸了`
- `历史观众在这里反应很大`
- `分支正在生成，稍后回来看看`
- `这集暂未识别到高光，先沉浸看片`

## 13.8 关键状态流转

`未创建 -> pending -> running -> success -> 结果展示`

异常分支：

- `running -> failed -> 展示失败兜底`
- `running -> timeout -> 允许重新生成`

## 15. 风险与控制策略

### 15.1 高光识别不准

风险：

- 模型输出高光点偏差较大

策略：

- 保留人工复核和修正机制

### 15.2 自定义分支生成不稳定

风险：

- 生成慢
- 生成失败
- 成本不稳定

策略：

- 固定分支保底

- 输出结果长度受控

### 15.3 客户端渲染压力

风险：

- Android 端视频播放和特效叠加导致卡顿

策略：

- 控制同屏元素数量
- 固定特效类别数量
- 优先保证视频播放稳定

### 15.4 本地部署演示不稳定

风险：

- 局域网环境变化
- 本机服务异常

策略：

- 提前准备标准演示机
- 本地服务单机可跑通
- 关键资源预缓存

- 仍失败则返回兜底表现

## 18.4 AI 质量评估

高光识别评估维度：

- 时间点准确性
- 类型准确性
- 强度等级合理性
- 推荐按钮贴合度
- 漏检率
- 误判率

分支生成评估维度：

- 是否符合原剧情人设
- 是否符合短剧节奏
- 是否有冲突、爽点或情绪推进
- 是否内容安全
- 是否具有继续观看欲望
- 是否长度适中

建议量化指标：

- `高光候选可用率 = 人工确认高光数 / 模型候选高光数`
- `分支生成成功率 = 成功生成任务数 / 总任务数`
- `JSON 解析成功率 = 合法结构化输出数 / 模型调用总数`
- `平均生成耗时 = 总生成耗时 / 生成任务数`

## 19. 异常状态与兜底策略

`视频加载失败`

- 文案：`视频加载失败，请检查本地服务或资源路径`
- 操作：重试、返回列表

`高光点为空`

- 文案：`这集暂未识别到高光，先沉浸看片`
- 处理：不展示互动层，保留基础播放能力

`高光接口失败`

- 处理：播放器继续播放，不阻塞主流程
- 若本地存在缓存高光点，优先使用缓存

`互动上报失败`

- 处理：本地先展示动画，再异步重试上报
- 不影响用户观看

`AI 分支生成失败`

- 文案：`这次脑洞生成失败了，可以换个方向再试试`
- 处理：保留固定分支入口，支持重新提交，记录失败原因

`模型输出格式错误`

- 处理：服务端尝试修复 JSON
- 修复失败后自动重试一次
- 仍失败则返回兜底表现

## 21. 内容安全、版权与隐私边界

短剧素材使用边界：

- 仅使用比赛指定范围内的短剧内容
- 遵守素材平台使用规则
- 仓库中不直接提交大体积版权视频资源
- 通过本地资源路径配置或单独说明完成内容接入

用户输入安全边界：

- 限制输入长度
- 过滤明显违规内容
- 对敏感输入返回友好提示
- 不将用户输入直接拼接到高风险系统指令中

模型输出安全边界：

- 限制输出长度
- 对结果进行敏感内容检查
- 不生成违法、血腥、仇恨等违规内容

密钥与隐私边界：

- 模型密钥不写入代码仓库
- 使用环境变量管理
- 提交 `.env.example`
- 在 `.gitignore` 中忽略 `.env`
- 录屏、截图和演示中避免展示真实密钥

## 22. 答辩表达主线

建议答辩统一沿以下主线展开：

1. 痛点
短剧高光密集，用户情绪强，但传统弹幕和评论需要输入文字，表达成本高，也容易打断观看。

2. 方案
Drama Pulse 通过 AI 辅助高光识别，在关键剧情点触发轻量互动组件，让用户一键表达情绪，并通过历史互动回流形成群体共鸣。

3. 闭环
系统完成了从短剧内容分析、高光点入库、客户端播放触发、用户互动上报、历史互动展示、尾集分支生成的完整闭环。

4. 创新

- 剧情语义驱动互动
- 低输入成本情绪表达
- 历史互动沉淀回流
- 尾集 AIGC 分支延展

5. 工程
Android 客户端 + 本地 Fastify 服务端 + 本地 Web 管理后台 + AI Orchestrator，模块边界清晰，支持后续扩展。

6. 展望
后续可扩展到更精细的视频理解、多用户实时互动、个性化互动推荐和更丰富的 AIGC 结果形态。
