# Drama Pulse 开发顺序与任务拆解初稿

## 1. 文档目的

本文档用于说明 Drama Pulse 项目后续应如何按模块推进开发，包括：

- 推荐开发前后顺序
- 每个阶段先做什么
- 每个模块包含哪些核心任务
- 每个模块完成后应产出什么结果
- 哪些模块可以并行，哪些模块存在依赖关系

本文档的目标不是给出严格时间排期，而是给出一条适合当前项目的开发推进路径。

## 2. 开发推进原则

当前项目建议遵循以下开发原则：

### 2.1 先基础支撑，再用户主链路，最后补增强项

必须先把资源、数据、服务层打稳，再去做客户端互动体验。否则后期会出现大量返工。

### 2.2 先做完整闭环，再做强效果

优先确保：

- 能播
- 能触发
- 能上报
- 能统计
- 能分支

在此基础上再去增强视觉效果和 AI 能力。

### 2.3 每一阶段都要有清晰产出

不是“写完一堆代码”算完成，而是必须形成可验证的结果，例如：

- 服务端接口可调通
- 客户端可播放
- 高光可以触发
- 分支结果可以展示

## 3. 推荐开发总顺序

建议整体按以下顺序推进：

1. 内容资源模块
2. 数据与配置模块
3. 本地服务端模块
4. 本地 Web 管理后台模块
5. Android 播放与基础体验模块
6. 高光互动模块
7. 历史互动沉淀模块
8. 尾集分支模块
9. AI 与离线脚本模块
10. 联调与验收模块

这个顺序的核心逻辑是：

- 先有资源和数据基础
- 再有后端中枢
- 再有管理能力
- 再接客户端主链路
- 最后补创新链路与 AI 支撑

## 4. 分阶段拆解

## 4.1 第一阶段：内容资源模块

### 目标

整理所有视频和基础素材，确保项目播放内容、固定分支结果和封面信息具备统一结构。

### 主要任务

- 整理主打剧 `荒年全村啃树皮，我有系统满仓肉`
- 整理备选剧 `撕夜`
- 统一剧名和集数命名
- 整理本地目录结构
- 准备封面图和简介信息
- 准备 2 个固定分支结果素材

### 阶段产出

- 本地视频目录可被服务端读取
- 封面和剧集元信息整理完成
- 固定分支资源准备完成

### 依赖关系

- 无前置依赖
- 后续服务端和后台直接依赖此阶段结果

## 4.2 第二阶段：数据与配置模块

### 目标

冻结系统的核心数据结构和配置方式，作为前后端和脚本层的统一依据。

### 主要任务

- 定义 Drama 数据结构
- 定义 Episode 数据结构
- 定义 Highlight 数据结构
- 定义 InteractionEvent 数据结构
- 定义 HighlightStats 数据结构
- 定义 BranchTask、BranchComment、BranchLike 数据结构
- 定义 WatchProgress 数据结构
- 定义主外键关系、唯一约束和索引
- 定义数据库 `path` 与接口 `url` 映射规则
- 定义状态枚举、模板枚举和热度枚举
- 定义本地配置文件结构
- 定义环境变量模板

### 阶段产出

- 数据模型说明
- 数据库 schema 依据
- 接口字段口径
- 配置项口径

### 依赖关系

- 可以与内容资源模块部分并行
- 后续服务端、后台、客户端都依赖此阶段结果

## 4.3 第三阶段：本地服务端模块

详细方案见 [本地服务端技术实现方案.md](/Users/a0000/Desktop/项目文件/drama-pulse/docs/本地服务端技术实现方案.md)

### 目标

建立整个项目的数据中枢和接口中枢，承接客户端和后台的所有核心业务能力。

### 主要任务

- 初始化服务端工程
- 接入 SQLite 与 ORM
- 建立核心数据表
- 提供短剧列表接口
- 提供剧集接口
- 提供高光列表接口
- 提供互动上报接口
- 提供高光统计接口
- 提供固定分支接口
- 提供自定义分支任务创建与查询接口
- 提供点赞和评论接口
- 提供继续观看进度接口
- 提供静态资源访问与 `path -> url` 转换

### 阶段产出

- 本地 API 服务可运行
- 核心业务接口可调试
- 基础数据可写入数据库

### 依赖关系

- 依赖内容资源模块与数据与配置模块
- 后续后台和客户端直接依赖此阶段结果

## 4.4 第四阶段：本地 Web 管理后台模块

详细方案见 [Web管理后台技术实现方案.md](/Users/a0000/Desktop/项目文件/drama-pulse/docs/Web管理后台技术实现方案.md)

### 目标

建立后台管理能力，承接内容配置、标签修正、数据查看和演示数据初始化工作。

### 主要任务

- 初始化后台工程
- 做剧集资源配置页
- 做高光标签管理页
- 做互动数据查看页
- 做分支任务管理页
- 做演示数据初始化能力
- 做标签启停、模板调整、任务重试
- 做后台鉴权和本地访问控制

### 阶段产出

- 可用的本地 Web 管理后台
- 不再依赖手工改数据库或脚本来配置业务

### 依赖关系

- 依赖本地服务端模块
- 可与客户端开发部分并行推进

## 4.5 第五阶段：Android 播放与基础体验模块

详细方案见 [Android端技术实现方案.md](/Users/a0000/Desktop/项目文件/drama-pulse/docs/Android端技术实现方案.md)

### 目标

先把“看剧”这件事跑通，保证列表、播放、选集和最后一集结束检测具备稳定能力。

### 主要任务

- 搭客户端包结构
- 做短剧列表页
- 做播放页
- 接入视频播放器
- 实现播放、暂停、进度条
- 实现播放页内选集
- 实现最后一集播放完成检测
- 建立基础网络请求与缓存能力

### 阶段产出

- 用户可在客户端稳定播放主打剧
- 可切换剧集
- 可检测最后一集结束事件

### 依赖关系

- 依赖服务端基础接口
- 是高光互动模块和尾集分支模块的承载基础

## 4.6 第六阶段：高光互动模块

### 目标

完成项目最核心的即时互动能力，让用户在高光点看到互动组件并完成点击反馈。

### 主要任务

- 监听播放时间轴
- 命中高光点触发互动
- 实现一主一辅互动机制
- 实现高光点过近的弱化规则
- 实现互动按钮、站队、助攻等模板能力
- 实现点击后的基础动效
- 完成互动上报

### 阶段产出

- 高光点可按时触发
- 用户点击后有明确视觉反馈
- 服务端可接收到互动事件

### 依赖关系

- 依赖 Android 播放与基础体验模块
- 依赖服务端高光下发能力

## 4.7 第七阶段：历史互动沉淀模块

### 目标

把用户互动数据沉淀为历史热度、弹幕和群体反馈，建立“陪伴感”与“高光氛围”。

### 主要任务

- 服务端做互动统计聚合
- 计算高光热度等级
- 生成模板化历史弹幕
- 播放页展示历史热度和回流弹幕
- 做重复点击去重和基础反垃圾策略

### 阶段产出

- 高光点能展示历史互动痕迹
- 热度分级机制可生效
- 播放页具备群体反馈感

### 依赖关系

- 依赖高光互动模块提供互动事件
- 依赖服务端统计能力

## 4.8 第八阶段：尾集分支模块

### 目标

完成最后一集结束后的分支体验，形成项目的创新亮点。

### 主要任务

- 最后一集结束后展示分支入口
- 实现 2 个固定分支展示
- 接入固定分支结果素材
- 实现自定义 Prompt 输入
- 实现自定义分支任务创建
- 实现任务状态轮询
- 实现分支结果页
- 实现点赞与评论
- 落地自定义分支结果的文本剧情与分镜展示

### 阶段产出

- 尾集结束后有分支入口
- 固定分支稳定可演示
- 自定义分支形成完整任务闭环
- 自定义分支结果以结构化文本和分镜为主，不依赖实时视频生成

### 依赖关系

- 依赖播放模块的最后一集结束检测
- 依赖服务端分支任务接口
- 依赖后台与 AI 链路提供配置和结果支持

## 4.9 第九阶段：AI 与离线脚本模块

### 目标

让高光识别和分支生成真正具备 AI 支撑，而不是只靠手工配置。

### 主要任务

- 音频抽取
- ASR 转写
- 文本结构化清洗
- 高光识别输出 JSON
- 高光导入服务端
- 分支剧情扩写
- 结构化分支结果生成
- AI 输出结构校验与落库
- 后台辅助复核与任务重试联动

### 阶段产出

- 至少一部剧能离线产出高光标签
- 自定义分支可返回结构化 AI 结果

### 依赖关系

- 可相对后置
- 不阻塞前期服务端和客户端闭环

## 4.10 第十阶段：联调与验收模块

### 目标

将各模块整合成一条稳定可演示的闭环，并完成演示前校验。

### 主要任务

- 客户端联服务端
- 后台联服务端
- 校验高光触发与互动上报
- 校验历史互动回流
- 校验固定分支与自定义分支
- 校验评论和点赞
- 准备演示数据
- 验证主打剧完整演示路径

### 阶段产出

- 一条完整的 Demo 演示闭环
- 可用于录屏与答辩的稳定版本

### 依赖关系

- 依赖所有核心模块基本完成

## 5. 可以并行推进的部分

以下模块可以部分并行：

- 内容资源模块 和 数据与配置模块
- 本地服务端模块 和 本地 Web 管理后台模块
- 本地 Web 管理后台模块 和 Android 播放与基础体验模块
- AI 与离线脚本模块可以相对后置独立推进

## 6. 关键依赖关系

以下依赖关系必须明确：

- 没有内容资源模块，服务端无法稳定提供剧集内容
- 没有数据与配置模块，前后端字段会反复变更
- 没有本地服务端模块，客户端和后台无法形成闭环
- 没有 Android 播放与基础体验模块，高光互动和分支入口无承载页面
- 没有高光互动模块，历史互动沉淀没有真实事件来源
- 没有最后一集结束检测，尾集分支模块无法成立

## 7. 开发包分组建议

为了便于按块推进开发，建议将这些模块进一步打包为 5 个开发包。

### 7.1 基础包

包含：

- 内容资源模块
- 数据与配置模块

目标：

- 先把内容和数据口径打稳

### 7.2 服务包

包含：

- 本地服务端模块

目标：

- 建立项目的数据中枢和业务接口能力

### 7.3 后台包

包含：

- 本地 Web 管理后台模块

目标：

- 建立内容配置和数据管理能力

### 7.4 客户端主链路包

包含：

- Android 播放与基础体验模块
- 高光互动模块

目标：

- 打通最核心的用户观看和互动链路

### 7.5 创新包

包含：

- 历史互动沉淀模块
- 尾集分支模块
- AI 与离线脚本模块

目标：

- 做出真正的差异化体验和创新亮点

## 8. 推荐实际推进顺序

如果按开发包来推进，建议顺序如下：

1. 基础包
2. 服务包
3. 后台包
4. 客户端主链路包
5. 创新包

这个顺序的优点是：

- 前面几包是后面几包的基础
- 可逐步形成闭环
- 不容易因为前置不稳造成返工

## 9. 当前建议

如果现在要开始正式开发，建议最先启动的不是客户端花哨效果，而是：

- 内容资源模块
- 数据与配置模块
- 本地服务端模块

原因是：

- 这三块决定了后面后台和客户端是否能共享统一的数据基础
- 它们是所有业务闭环的最早依赖
- 先把这三块打稳，后续开发会顺很多
