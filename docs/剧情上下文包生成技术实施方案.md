# Drama Pulse 剧情上下文包生成技术实施方案

> 说明：本文档聚焦 `story_context_package` 的生成链路，服务于尾集分支生成前置能力建设。若与 [技术方案初稿.md](/Users/a0000/Desktop/项目文件/drama-pulse/docs/技术方案初稿.md) 存在冲突，以本文对 `剧情上下文包工作流` 的实现定义为准。

## 1. 文档目标

本文档只解决一件事：

`如何把一部短剧从“原始视频/字幕素材”加工为可复用、可版本化、可供 BranchTask 消费的剧情上下文包。`

本文档用于指导以下工作：

- 剧情理解离线脚本设计
- 剧集级上下文任务编排
- 中间产物目录与文件格式定义
- `story_context_package` 数据结构设计
- `BranchTask` 如何挂接上下文包
- 后续人工修订与局部重跑策略

## 2. 设计边界

### 2.1 本方案覆盖

- 单集台词文本化
- 分集层剧情摘要生成
- 关键事件时间线生成
- 全剧层总览生成
- 尾集状态快照生成
- 总包组装与版本化
- 与尾集分支生成任务的衔接方式

### 2.2 本方案不覆盖

- 固定分支内容制作
- 视频生成供应商接入
- 人物参考图自动抽取
- 剧集内容审核系统
- Android / Web 端展示细节

### 2.3 实施原则

- 一部剧的上下文包尽量只生成一次，多次复用
- 上下文生成链路与 `BranchTask` 解耦
- 所有中间产物都可落文件、可人工检查
- 允许局部重跑，不要求全链路每次重做
- 结构化 JSON 优先，不依赖长篇自然语言结果

## 3. 为什么必须有剧情上下文包

尾集分支生成不是普通问答，而是“受约束的剧情续写”。

如果只给模型一段用户提示词和尾集内容，通常会出现这些问题：

- 忘记前面发生过什么
- 人物关系理解错位
- 关键真相和因果链断裂
- 角色性格跑偏
- 同一部剧多次生成结果风格不稳定

所以需要先把整部剧压缩成一个可复用的 `story_context_package`，再让分支任务在这个上下文包上生成。

## 4. 目标产物

最终目标文件：

`story_context_package.json`

它不是单段摘要，而是由多层结构组成的标准包。

推荐结构：

```json
{
  "drama_id": "drama_001",
  "title": "示例剧名",
  "version": "2026-05-31T20:00:00+08:00",
  "series_overview": {},
  "character_bible": [],
  "episode_summaries": [],
  "event_timeline": [],
  "tail_state_snapshot": {},
  "canon_constraints": []
}
```

## 5. 五层处理架构

推荐固定为五层，不建议跳层直接生成：

1. `transcript layer`
2. `episode layer`
3. `timeline layer`
4. `series layer`
5. `tail snapshot layer`

处理顺序：

```text
原始视频 / 字幕
  -> 单集台词文本化
  -> 分集层摘要
  -> 时间线层
  -> 全剧层
  -> 尾集快照层
  -> 组装总包
```

## 6. 各层设计

### 6.1 Transcript Layer

目标：

- 把每一集视频内容转成带时间戳的文本片段

输入：

- 剧集视频文件
- 可选外挂字幕
- 可选 OCR 字幕结果
- 可选角色名单

输出文件：

`episode_transcript.json`

建议字段：

```json
{
  "episode_id": "ep_001_01",
  "episode_no": 1,
  "source_type": "mixed",
  "language": "zh-CN",
  "segments": [
    {
      "segment_id": "seg_0001",
      "start_ms": 1200,
      "end_ms": 3800,
      "speaker_guess": "女主",
      "text": "你终于回来了。",
      "confidence": 0.94
    }
  ]
}
```

要求：

- 只做文本抽取和基础清洗
- 不在这一层做剧情归纳
- 如已有稳定字幕，优先字幕；ASR 只做补全

### 6.2 Episode Layer

目标：

- 回答“这一集发生了什么”

输入：

- `episode_transcript.json`

输出文件：

`episode_summary.json`

建议字段：

```json
{
  "episode_id": "ep_001_01",
  "episode_no": 1,
  "summary": "女主在多年后回到故乡，发现男主身份已经改变。",
  "key_events": [],
  "character_changes": [],
  "new_reveals": [],
  "open_questions": [],
  "episode_ending_state": {}
}
```

字段建议：

- `summary`
- `key_events`
- `character_changes`
- `new_reveals`
- `open_questions`
- `episode_ending_state`

要求：

- 必须结构化 JSON
- 单集独立生成，支持局部重跑

### 6.3 Timeline Layer

目标：

- 回答“这些事情按什么顺序发生，彼此因果是什么”

输入：

- 全部 `episode_summary.json`
- 必要时回看 `episode_transcript.json`

输出文件：

`event_timeline.json`

建议字段：

```json
[
  {
    "event_id": "evt_0001",
    "episode_id": "ep_001_01",
    "episode_no": 1,
    "order_in_episode": 1,
    "actors": ["女主", "男主"],
    "event": "女主回乡与男主重逢",
    "cause": "多年失联后再次相遇",
    "impact": "主线关系重新启动",
    "resolved": false
  }
]
```

要求：

- 必须可排序
- `cause` 和 `impact` 尽量显式
- 第一版可以只覆盖主线关键事件

### 6.4 Series Layer

目标：

- 回答“这部剧整体讲什么、有哪些不能改的设定”

输入：

- 全部 `episode_summary.json`
- `event_timeline.json`

输出文件：

`series_overview.json`

建议字段：

- `series_premise`
- `genre`
- `main_conflict`
- `tone`
- `character_bible`
- `relationship_graph`
- `canon_constraints`

其中 `character_bible` 建议至少包含：

- `character_name`
- `identity`
- `personality`
- `motivation`
- `relationship_summary`
- `do_not_change`

其中 `canon_constraints` 建议至少包含：

- 已发生且不能推翻的事实
- 角色不能偏移的人设边界
- 不能越界的题材规则

### 6.5 Tail Snapshot Layer

目标：

- 回答“尾集结束这一刻，世界是什么状态”

输入：

- 尾集 `episode_transcript.json`
- 尾集 `episode_summary.json`
- `event_timeline.json`
- `series_overview.json`

输出文件：

`tail_state_snapshot.json`

建议字段：

```json
{
  "current_conflict": "女主已经得知真相，但男主仍选择隐瞒关键证据",
  "alive_status": {},
  "who_knows_what": {},
  "relationship_states": {},
  "unresolved_questions": [],
  "branch_entry_points": [],
  "hard_constraints": []
}
```

要求：

- 这层优先级高于全剧摘要
- 后续分支生成应优先使用这层作为续写起点

## 7. 总包组装规则

最终由 `series_overview.json`、`event_timeline.json`、`tail_state_snapshot.json`、全部 `episode_summary.json` 聚合为：

`story_context_package.json`

组装时建议增加：

- `drama_id`
- `title`
- `version`
- `generated_at`
- `source_manifest_version`

版本规则建议：

- 默认使用生成时间戳
- 允许人工修订后产生新版本
- `BranchTask` 只记录自己命中的版本，不跟随最新版本自动变动

## 8. 脚本拆分方案

建议目录：

```text
scripts/story_context/
  01_extract_transcript.py
  02_generate_episode_summary.py
  03_build_event_timeline.py
  04_build_series_overview.py
  05_build_tail_snapshot.py
  06_assemble_story_context.py
  run_story_context_pipeline.py
```

### 8.1 `01_extract_transcript.py`

职责：

- 输入视频或字幕
- 输出 `episode_transcript.json`

建议参数：

- `--drama-id`
- `--episode-id`
- `--video-path`
- `--subtitle-path`
- `--output-dir`

### 8.2 `02_generate_episode_summary.py`

职责：

- 输入单集 transcript
- 输出单集摘要

建议参数：

- `--drama-id`
- `--episode-id`
- `--transcript-path`
- `--output-dir`
- `--model`

### 8.3 `03_build_event_timeline.py`

职责：

- 聚合全部分集摘要
- 输出关键事件时间线

建议参数：

- `--drama-id`
- `--episode-summary-dir`
- `--output-dir`
- `--model`

### 8.4 `04_build_series_overview.py`

职责：

- 聚合分集摘要和时间线
- 输出全剧层总览

建议参数：

- `--drama-id`
- `--episode-summary-dir`
- `--timeline-path`
- `--output-dir`
- `--model`

### 8.5 `05_build_tail_snapshot.py`

职责：

- 聚合尾集和全剧上下文
- 输出尾集快照

建议参数：

- `--drama-id`
- `--final-episode-id`
- `--final-transcript-path`
- `--final-summary-path`
- `--timeline-path`
- `--series-overview-path`
- `--output-dir`
- `--model`

### 8.6 `06_assemble_story_context.py`

职责：

- 读取全部中间产物
- 组装最终上下文包
- 输出版本号

建议参数：

- `--drama-id`
- `--title`
- `--episode-summary-dir`
- `--timeline-path`
- `--series-overview-path`
- `--tail-snapshot-path`
- `--output-dir`

### 8.7 `run_story_context_pipeline.py`

职责：

- 统一编排执行顺序
- 更新 `manifest.json`
- 记录失败步骤和可重试点

建议参数：

- `--drama-id`
- `--input-root`
- `--output-root`
- `--final-episode-id`
- `--model-provider`
- `--model-name`

## 9. 目录与产物规范

建议产物目录：

```text
artifacts/story_context/{dramaId}/
  transcripts/
  episode_summaries/
  timeline/
  series/
  tail/
  package/
  manifest.json
```

建议约定：

- `episode_summaries/{episodeId}.json`
- `timeline/event_timeline.json`
- `series/series_overview.json`
- `tail/tail_state_snapshot.json`
- `package/story_context_package.json`

## 10. Manifest 设计

整条链路建议维护一个：

`manifest.json`

建议字段：

```json
{
  "drama_id": "drama_001",
  "pipeline_status": "timeline_ready",
  "completed_steps": ["transcript", "episode_summary"],
  "failed_steps": [],
  "episode_progress": {
    "ep_001_01": "done",
    "ep_001_02": "done"
  },
  "model_provider": "doubao",
  "model_name": "xxx",
  "started_at": "2026-05-31T20:00:00+08:00",
  "updated_at": "2026-05-31T20:30:00+08:00",
  "version": "2026-05-31T20:30:00+08:00"
}
```

推荐阶段状态：

- `pending`
- `transcript_ready`
- `episode_summary_ready`
- `timeline_ready`
- `series_overview_ready`
- `tail_snapshot_ready`
- `package_ready`
- `failed`

## 11. 与 BranchTask 的衔接方式

推荐顺序不是：

`用户输入 prompt -> 临时拼前情上下文 -> 直接生成分支`

而是：

`离线生成 story_context_package -> 用户输入 prompt -> 创建 BranchTask -> 读取上下文包 -> 生成分支`

`BranchTask` 执行时建议：

1. 根据 `episode_id` 找到所属 `drama_id`
2. 读取当前有效的 `story_context_package`
3. 将上下文版本写入 `story_context_version`
4. 将完整文件路径写入 `story_context_asset_path`
5. 将摘要内容写入 `story_context_json`
6. 将 `tail_state_snapshot` 单独写入 `tail_state_snapshot_json`
7. 基于这些上下文再组装 `prompt_package_json`

这样做的价值：

- 分支任务与原始视频解耦
- 上下文生成和分支生成可以分别排障
- 一部剧的上下文包可复用于多个用户分支任务

## 12. 第一版实施建议

建议按三步落地：

### 12.1 第一步

先做：

- `transcript`
- `episode_summary`
- `timeline`
- `tail_snapshot`
- `story_context_package`

先不追求：

- 极准的人物说话人识别
- 自动抽参考图
- 视频生成接入

### 12.2 第二步

补：

- `series_overview.character_bible`
- `canon_constraints`
- 人工修订入口

### 12.3 第三步

再把上下文包稳定挂到：

- `BranchTask`
- 后台任务详情
- 重试与回溯机制

## 13. 最小验收标准

第一版可验收标准建议为：

1. 能从指定剧集视频或字幕生成每集 `episode_transcript.json`
2. 能为每一集生成结构化 `episode_summary.json`
3. 能生成整部剧的 `event_timeline.json`
4. 能生成尾集 `tail_state_snapshot.json`
5. 能组装 `story_context_package.json`
6. 能通过版本号和文件路径被 `BranchTask` 命中
7. 单集摘要错误时，可只重跑该集而非整部剧

## 14. 风险与应对

### 14.1 ASR / 字幕质量不稳定

风险：

- 后续所有层都会被污染

应对：

- 优先使用外挂字幕
- transcript 层保留原文和置信度
- 允许人工修正 transcript 再继续后续步骤

### 14.2 分集摘要漂移

风险：

- 时间线和尾集快照理解错位

应对：

- 分集层输出必须结构化
- 关键字段加长度和格式约束
- 对异常集数支持局部重跑

### 14.3 全剧层过度抽象

风险：

- 分支生成仍然会丢细节

应对：

- 分支生成时不要只喂 `series_overview`
- 必须同时带上 `tail_state_snapshot` 和 `recent_event_timeline`

### 14.4 上下文包版本混乱

风险：

- 同一任务复现不了

应对：

- `BranchTask` 必须记录 `story_context_version`
- 产物文件必须落地，不能只存数据库摘要

## 15. 结论

剧情上下文包生成链路不应被视为“分支生成前随手补一段前情”，而应作为独立的剧集级基础设施存在。

推荐最终技术路线为：

`原始视频/字幕 -> transcript -> 分集层 -> 时间线层 -> 全剧层 -> 尾集快照层 -> story_context_package -> BranchTask 分支生成`

只有先把这条链路做稳，尾集分支生成才有真正稳定的剧情基础。
