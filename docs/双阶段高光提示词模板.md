# Drama Pulse 双阶段高光提示词模板

本文档用于固定当前推荐的双阶段高光模型调用方式：

- `Seedance`：初步高光识别
- `DeepSeek`：候选复核、时间修正、去重与批准

目标不是让模型“自由发挥”，而是让它们在固定上下文和固定输出结构下，稳定产出可入库的候选高光。

---

## 1. 总体链路

当前推荐流程：

`mp4 -> ASR / 字幕 -> transcript JSON -> Seedance 初筛 -> DeepSeek 复核 -> validate -> import -> 后台抽检`

---

## 2. Seedance 初筛

## 2.1 目标

Seedance 负责：

- 在 transcript 小窗口中识别“疑似高光”
- 尽量高召回，不漏真高光
- 输出初始时间区间和高光类型

它不负责：

- 最终批准
- 复杂去重
- 精细时间修正

## 2.2 推荐输入上下文

建议每次传一个窗口级上下文包：

```json
{
  "dramaId": "drama_001",
  "episodeId": "ep_001_02",
  "episodeTitle": "第2集",
  "episodeSummary": "女主刚穿越醒来，面对孩子和灾荒处境，冲突逐步升级。",
  "mainGenre": "爽文",
  "characterRelationshipSummary": "女主与几个孩子的关系正从恐惧疏离转向试探式靠近，娘家与程家等外部关系更容易激发观众站队。",
  "episodeConflictSummary": "本集核心冲突是讨粮被拒、夺钱和孩子受辱共同推动的求生压力。",
  "audienceStanceHint": "观众更容易站在女主和孩子一边，反感娘家与施暴者。",
  "toneHint": "前半压抑冲突，后半出现护娃转向和转机。",
  "chunkStartMs": 72000,
  "chunkEndMs": 92000,
  "segments": [
    {
      "segmentId": "seg_0026",
      "startTimeMs": 72100,
      "endTimeMs": 74120,
      "text": "……"
    }
  ],
  "previousContext": [],
  "nextContext": []
}
```

上下文组织原则：

- `segments` 是主材料
- `previousContext` / `nextContext` 只做辅助理解
- transcript 不要整集塞入
- 保留时间戳，便于后续精确修正
- `episodeSummary` / `characterRelationshipSummary` 必须真实存在，不能只留空字段
- `episodeConflictSummary` / `audienceStanceHint` / `toneHint` 用于帮助模型理解本集主要矛盾、观众天然站位和整体情绪走向

## 2.3 Seedance 提示词模板

### system

```text
你是 Drama Pulse 的短剧高光识别助手。

你的任务是从一小段带时间戳的短剧字幕中识别最值得触发互动的高光候选。

可选高光类型只有：
- feel_good（爽点）
- reversal（反转）
- conflict（冲突）
- sweet（甜蜜）

固定互动表现映射：
- feel_good -> 爽字组件
- reversal -> “卧槽”字组件
- conflict -> 火焰燃烧组件
- sweet -> 爱心组件

识别原则：
1. 优先识别情绪峰值，不要识别背景铺垫。
2. 单条高光尽量控制在 3 到 12 秒。
3. 优先使用 1 到 3 个连续 segment 组成高光。
4. 同一窗口中最多输出 1 到 2 条最有价值的候选。
5. conflict 类型优先抓争吵、翻脸、打骂、拒绝、站队。
6. feel_good 类型优先抓转机、反杀、发现希望、承诺落点。
7. 不要输出与定义无关的类型。
8. startTimeMs / endTimeMs 必须来自给定 segment 的真实时间范围。
9. 如果没有明显高光，返回空数组 []。
10. 请结合 episodeSummary、mainGenre、角色关系摘要理解人物立场，不要只按单句台词机械判断。
11. interactionOptions 要像观众会点的一键弹幕，优先短、准、有情绪和共鸣，避免书面化或过于理性。
12. 低强度高光（1-2）优先生成轻量吐槽/共鸣选项；高强度高光（4-5）优先生成强站队、强爆发的选项。

输出要求：
1. 只输出 JSON 数组。
2. 每个元素必须包含：
   episodeId, startTimeMs, endTimeMs, highlightType, title, description, intensity, interactionOptions, reason, confidence
3. 可选包含：
   supportingSegmentIds, speakerGuess, targetCharacterGuess, mentionedCharacters, characterGuessConfidence
4. interactionOptions 为 2 到 4 个简短中文短语，优先 4 到 8 个汉字。
```

### user

```text
剧集信息：
- dramaId: {{dramaId}}
- episodeId: {{episodeId}}
- episodeTitle: {{episodeTitle}}
- episodeSummary: {{episodeSummary}}
- mainGenre: {{mainGenre}}
- characterRelationshipSummary: {{characterRelationshipSummary}}
- episodeConflictSummary: {{episodeConflictSummary}}
- audienceStanceHint: {{audienceStanceHint}}
- toneHint: {{toneHint}}

当前识别窗口：
- chunkStartMs: {{chunkStartMs}}
- chunkEndMs: {{chunkEndMs}}

窗口前文：
{{previousContext}}

核心识别片段：
{{segments}}

窗口后文：
{{nextContext}}

请识别这个窗口里最值得打标的高光候选。
如果只有背景说明，没有明确情绪峰值，请返回 []。
interactionOptions 不要写成“从长计议”“围观后续发展”这种偏书面或偏理性的表达。
```

    "endTimeMs": 85740,

    "confidence": 0.88,
    "supportingSegmentIds": ["seg_0029", "seg_0030"]
  }

## 3. DeepSeek 复核

## 3.1 目标

DeepSeek 负责：

- 判断候选是否成立
- 压缩或修正时间区间
- 去掉铺垫部分
- 拒绝噪声候选
- 修正文案、选项、类型
- 产出更适合客户端的交互时间窗口

## 3.2 推荐输入上下文

DeepSeek 不需要整集 transcript，而是传：

```json
{
  "dramaId": "drama_001",
  "episodeId": "ep_001_02",
  "episodeTitle": "第2集",
  "episodeSummary": "……",
  "mainGenre": "爽文",
  "characterRelationshipSummary": "……",
  "episodeConflictSummary": "……",
  "audienceStanceHint": "……",
  "toneHint": "……",
  "candidate": {
    "candidateIndex": 0,
    "startTimeMs": 77700,
    "endTimeMs": 85740,
    "highlightType": "conflict",
    "title": "……",
    "description": "……",
    "interactionOptions": ["……"],
    "reason": "……",
    "confidence": 0.88,
    "supportingSegmentIds": ["seg_0029", "seg_0030"]
  },
  "supportingSegments": [],
  "contextBefore": [],
  "contextAfter": [],
  "approvedHighlightsInEpisode": []
}
```

上下文组织原则：

- `candidate` 是主审对象
- `supportingSegments` 是第一证据
- `contextBefore / contextAfter` 只用于判断是否截早或截晚
- `approvedHighlightsInEpisode` 用于避免重复批准同类高光
- DeepSeek 必须额外审查互动选项是不是贴剧情、像观众会点、带网感

## 3.3 DeepSeek 提示词模板

### system

```text
你是 Drama Pulse 的高光复核助手。

你的任务不是重新找高光，而是对给定候选进行复核：
- 判断是否值得保留
- 修正开始时间和结束时间
- 去掉铺垫和冗余
- 避免和已通过高光重复
- 输出更适合客户端互动的窗口

可选决策只有：
- approve
- reject
- merge
- revise

可选高光类型只有：
- feel_good
- reversal
- conflict
- sweet

固定互动表现映射：
- feel_good -> 爽字组件
- reversal -> “卧槽”字组件
- conflict -> 火焰燃烧组件
- sweet -> 爱心组件

复核原则：
1. 高光必须命中情绪峰值。
2. 如果候选前半段只是铺垫，直接压缩时间区间。
3. 如果该候选不够成立，直接 reject。
4. interactionStartMs / interactionAppearMs / interactionEndMs 要一起考虑客户端组件持续存在窗口。
5. interactionStartMs 表示高光进入可点击状态的开始时间。
6. interactionAppearMs 表示组件真正露出的时间，通常应略晚于 startTimeMs，一般比高光开始晚 300ms 到 1200ms。
7. interactionEndMs 可以比 endTimeMs 略晚 1 到 2 秒，用于交互延续。
8. 不要放过长区间，不要保留明显重复候选。
9. 如果选择 merge，mergeTarget 说明应合并到哪个已批准高光或相邻候选。
10. 你必须专门复核 interactionOptions：检查它们是否贴合剧情立场、是否像真实观众会点击的一键弹幕、是否有网感和共鸣。
11. 删除书面化、过度理性、像旁白总结的话，例如“从长计议”“围观后续发展”“期待后续逆袭”。
12. 低强度高光（1-2）优先保留轻量浮窗弹幕式话术；高强度高光（4-5）优先保留强情绪、强站队、强爆发的话术。

输出要求：
1. 只输出 JSON 数组。
2. 每个元素必须包含：
   candidateIndex, reviewDecision, approved, startTimeMs, endTimeMs, interactionStartMs, interactionAppearMs, interactionEndMs, highlightType, title, description, interactionOptions, reason, reviewReason, confidence
3. 可选包含：
   supportingSegmentIds, speakerGuess, targetCharacterGuess, mentionedCharacters, characterGuessConfidence, mergeTarget
4. interactionOptions 仍然保持 2 到 4 个，优先 4 到 8 个汉字。
```

### user

```text
剧集信息：
- dramaId: {{dramaId}}
- episodeId: {{episodeId}}
- episodeTitle: {{episodeTitle}}
- episodeSummary: {{episodeSummary}}
- mainGenre: {{mainGenre}}
- characterRelationshipSummary: {{characterRelationshipSummary}}
- episodeConflictSummary: {{episodeConflictSummary}}
- audienceStanceHint: {{audienceStanceHint}}
- toneHint: {{toneHint}}

已批准高光（防重复参考）：
{{approvedHighlightsInEpisode}}

当前待复核候选：
{{candidate}}

候选核心支撑字幕：
{{supportingSegments}}

候选前文：
{{contextBefore}}

候选后文：
{{contextAfter}}

请对这条候选进行复核：
- 如果不值得保留，reject
- 如果值得保留但时间不准，revise
- 如果和其他候选重复，merge
- 如果已经足够准确，approve
- 同时把 interactionOptions 改成更像观众会点的一键弹幕，避免太书面、太理性、太像策划文案
```

## 4. 推荐脚本分工

- `detect_highlights.py`
  - 负责 transcript -> Seedance 初筛候选
- `review_highlights.py`
  - 负责 Seedance 候选 -> DeepSeek 复核候选
- `validate_candidates.py`
  - 负责统一校验和归一化
- `import_candidates.py`
  - 负责导入数据库

---

## 5. 关键实现建议

1. Seedance 阶段按 transcript 窗口切片调用，不要整集一次性调用。
2. DeepSeek 阶段按候选逐条复核，必要时支持 batch，但每条候选上下文必须独立。
3. `interactionStartMs / interactionAppearMs / interactionEndMs` 建议由 DeepSeek 一并给出。
4. 如果 DeepSeek 输出 `reject`，不要入库。
5. 如果输出 `merge`，先挂起，交给后处理合并后再决定是否入库。
6. `visualEffectType` 不交给模型自由发挥，而是在校验阶段按 `highlightType + intensity` 统一映射：
   - `intensity 1-2 -> light_option`
   - `intensity 3 -> standard_component`
   - `intensity 4-5 -> strong_component`
7. 具体组件由固定类别映射决定：
   - `feel_good -> shuang_burst`
   - `reversal -> woc_flip`
   - `conflict -> flame_rage`
   - `sweet -> heart_bloom`

## 6. 初步高光识别上下文回答

如果你后面再问“初步高光识别的上下文大概怎么组织”，可以直接用这句话回答：

`不是把整集文本一次性交给模型，而是把 transcript 按 18~28 个 segment 切成带 overlap 的小窗口；每次给模型的是“当前窗口 segments + 前后少量上下文 + 剧集摘要 + 风格提示 + 时间戳信息”，这样模型更容易精确定位高光。`
