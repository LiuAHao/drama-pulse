## 2. 技术目标

### 2.1 核心目标

- 支撑短剧播放、高光互动、历史互动沉淀、尾集分支 AIGC 的完整闭环
- 保障本地部署环境下的稳定性和可演示性
- 支撑后续文档、开发、答辩中的技术表达

### 2.2 技术原则

- 主链路优先稳定，不将复杂能力堆到实时播放链路
- 所有 AI 重任务尽量前置离线化或异步化
- 客户端渲染链路轻量化，优先保证视频播放稳定
- 服务端职责清晰，模块可替换、可扩展
- 所有能力均以“本地演示稳定”为第一约束

## 3. 技术范围

当前版本技术范围覆盖：

- Android 客户端
- 本地服务端
- 本地 Web 管理后台
- 离线高光提取与标注链路
- 历史互动统计与回流链路
- 尾集固定分支与自定义分支任务链路

当前版本不追求：

- 公网部署
- 多人实时联机
- 实时视频理解
- 实时视频生成
- 跨端统一代码架构

## 4. 总体架构

系统建议采用四层结构：

1. `客户端层`
Android App，负责短剧列表、视频播放、高光触发、互动渲染、尾集分支入口与结果展示。

2. `服务层`
本地服务端，负责剧集管理、高光标签下发、互动事件写入、历史统计聚合、分支任务管理、AI 任务编排。

3. `管理层`
本地 Web 管理后台，负责高光修正、资源配置、互动查看、任务管理、演示数据初始化。

4. `AI 处理层`
包括高光提取离线链路和分支生成异步链路，负责调用 ASR 与大模型能力，并输出结构化结果。

## 6. 冻结技术选型

## 6.1 Android 客户端

冻结技术栈：

- `Kotlin`
- `Jetpack Compose`
- `Media3 / ExoPlayer`
- `ViewModel + StateFlow`
- `Retrofit + OkHttp`
- `Room`
- `Coil` 用于图片加载

选择理由：

- Kotlin + Compose 开发效率高，适合单人快速迭代
- Media3 在 Android 视频播放上更稳，适合定制时间轴监听
- StateFlow 更适合播放状态、高光状态、任务状态管理

## 6.2 本地服务端

冻结技术栈：

- `Node.js + TypeScript`
- `Fastify`
- `SQLite`
- `Prisma`
- `Zod` 用于接口输入输出校验

选择理由：

- Node.js 全栈开发效率高，适合单人比赛项目
- TypeScript 在接口、数据结构和 AI 输出解析上更容易控类型
- SQLite 足够支撑本地部署、低运维、易备份

## 6.3 本地 Web 管理后台

冻结技术栈：

- `Vite + React`
- `TypeScript`
- `React Router`
- `TanStack Query`
- `Tailwind CSS`

选择理由：

- 页面结构清晰，适合快速搭建管理后台
- 与 Node 服务端数据结构更好统一
- 更适合比赛答辩展示和快速联调

## 6.4 工具链版本基线

为了避免后续开发环境和依赖矩阵反复变动，当前建议冻结以下工具链基线：

- Android Gradle Plugin：`8.7.x`
- Gradle Wrapper：`8.10.x`
- Kotlin：`2.0.x`
- JDK：`17`
- Node.js：`20 LTS`
- pnpm：`9.x`
- Python：`3.11`
- SQLite：`3.x`

说明：

- Android 工程允许在小版本内升级，但不建议在项目中途跨大版本升级 AGP、Gradle 或 Kotlin。
- 服务端与后台统一采用 Node.js 20 LTS，避免本地开发和 CI 环境不一致。

## 6.5 AI 处理链路

建议能力来源：

- `ASR`：优先使用稳定的语音转文本能力
- `LLM`：高光识别、标签生成、尾集分支剧情扩写
- 可选的视频结果：后续扩展，不作为主依赖

模型策略：

- 高光识别：`DeepSeek` 离线双阶段批处理
- 分支生成：`豆包` 异步任务处理
- 所有模型输出统一为结构化 JSON

## 7. 核心模块设计

## 7.1 Android 客户端模块

`DramaList Module`

- 获取短剧列表
- 展示主打剧和备选剧
- 维护继续观看入口

`Player Module`

- 视频播放与播控
- 播放进度监听
- 视频结束检测

`Episode Selector Module`

- 播放页内切换剧集
- 更新当前视频与高光数据

`Highlight Trigger Module`

- 缓存高光点列表
- 基于播放进度命中高光点
- 管理单次播放会话内的触发去重
- 处理“高光点过近”时的弱化展示逻辑

`Interaction Render Module`

- 渲染互动按钮
- 渲染粒子、贴纸、弹幕与热度反馈
- 控制互动组件生命周期

`Branch Module`

- 检测最后一集结束
- 展示尾集分支入口
- 请求固定分支与创建自定义任务
- 展示结果、点赞、评论

`Network Module`

- 调用服务端接口
- 统一错误处理与重试

`Local Cache Module`

- 缓存短剧列表、剧集信息、高光列表、用户分支历史
- 在弱网或局部接口失败时作为降级手段

## 7.2 服务端模块

`Content Service`

- 管理短剧、剧集、资源路径
- 提供短剧和剧集查询

`Highlight Service`

- 管理高光标签的存储、查询、状态变更
- 提供按集下发的高光点列表

`Interaction Service`

- 记录用户互动行为
- 聚合同一高光点的统计数据
- 提供历史互动回流数据

`Branch Service`

- 返回固定分支配置
- 创建与查询自定义分支任务
- 持久化分支结果、点赞和评论

`AI Orchestrator`

- 封装模型调用
- 负责 Prompt 拼接、结构校验、解析重试、结果落库

`Admin Service`

- 支撑后台管理操作
- 提供高光修正、资源配置、任务管理、演示数据重置

## 8. 数据存储设计

## 8.1 存储划分

建议采用“结构化数据 + 本地文件资源”双存储设计。

结构化数据存入 `SQLite`：

- 短剧元信息
- 剧集元信息
- 高光标签
- 互动事件
- 高光统计
- 固定分支配置
- 自定义分支任务
- 点赞评论数据

文件资源存入本地目录：

- 视频文件
- 封面图
- 分支预生成图文结果资源
- AI 中间结果 JSON
- 分镜草稿导出结果

## 8.2 建议目录结构

```text
project-root/
├── android-app/
├── server/
├── admin-web/
├── scripts/
├── data/
│   ├── app.db
│   ├── seed/
│   └── exports/
├── assets/
│   ├── dramas/
│   │   ├── covers/
│   │   ├── episodes/
│   │   └── branches/
│   └── effects/
└── docs/
```

## 8.3 资源路径约定

为了避免数据库、接口和本地文件路径三套口径不一致，当前约定如下：

- 数据库中只存 `path`，不直接存最终可访问 URL
- 服务端根据 `path` 拼接为客户端可访问的 `url`
- 所有 `path` 都应为相对项目根目录的相对路径

具体约定：

- 剧集视频：`videos/<drama_title>/第N集.mp4`
- 剧封面：`assets/dramas/covers/<drama_id>.jpg`
- 固定分支结果：`assets/dramas/branches/<episode_id>/<option_id>/manifest.json`
- 固定分支图片资源：`assets/dramas/branches/<episode_id>/<option_id>/frames/*.png`
- 高光导出 JSON：`data/exports/highlights/<episode_id>.json`
- 分支结果导出 JSON：`data/exports/branches/<task_id>.json`

## 8.4 演示数据与真实数据划分

为了支持答辩演示和后续真实操作，服务端需明确区分两类数据：

- `seed data`：初始化剧集、高光、固定分支和基础互动统计
- `runtime data`：运行过程中真实产生的互动、任务、点赞和评论

管理后台中的“演示数据重置”只重置 `runtime data`，不覆盖固定内容资源和基础配置。

## 9. 数据模型与字段定义

数据模型延续产品方案中的核心对象，并补充技术实现建议。

## 9.1 Drama

主键与关系：

- 主键：`id`
- 一对多关联：`Drama 1 -> N Episode`

字段定义：

- `id`：字符串主键，推荐格式 `drama_xxx`
- `title`：剧名，唯一
- `description`：剧简介
- `cover_path`：封面相对路径
- `tags_json`：标签数组 JSON
- `main_genre`：主类型，例如 `爽文`、`情感`、`悬疑`
- `is_featured`：是否主推剧
- `display_order`：展示排序
- `status`：`active | hidden`
- `created_at`
- `updated_at`

## 9.2 Episode

主键与关系：

- 主键：`id`
- 外键：`drama_id -> Drama.id`
- 一对多关联：`Episode 1 -> N Highlight`
- 一对多关联：`Episode 1 -> N BranchOption`
- 一对多关联：`Episode 1 -> N BranchTask`

字段定义：

- `id`：字符串主键，推荐格式 `ep_xxx`
- `drama_id`
- `episode_no`：集数，整型，剧内唯一
- `title`
- `video_path`：视频相对路径
- `duration_ms`
- `summary`
- `is_final_episode`
- `has_branch`：第一版仅在最后一集为 `true`
- `status`：`active | hidden`
- `created_at`
- `updated_at`

## 9.3 Highlight

主键与关系：

- 主键：`id`
- 外键：`episode_id -> Episode.id`
- 一对多关联：`Highlight 1 -> N InteractionEvent`
- 一对一聚合：`Highlight 1 -> 1 HighlightStats`

字段定义：

- `id`：字符串主键，推荐格式 `hl_xxx`
- `episode_id`
- `start_time_ms`
- `end_time_ms`
- `interaction_start_ms`
- `interaction_appear_ms`
- `interaction_end_ms`
- `type`：见枚举定义
- `title`
- `description`
- `intensity`：`1-5`
- `interaction_options_json`：固定互动选项数组，第一版不支持自由文本互动
- `visual_effect_type`
- `source`：`manual | ai | ai_adjusted | heat_feedback`
- `confidence`：`0-1`
- `status`：`candidate | confirmed | disabled`
- `created_at`
- `updated_at`

约束：

- `start_time_ms < end_time_ms`
- `interaction_start_ms <= interaction_appear_ms < interaction_end_ms`
- 同一 `episode_id` 下高光按 `start_time_ms` 排序
- 客户端只消费 `status = confirmed` 的高光点

## 9.4 InteractionEvent

主键与关系：

- 主键：`id`
- 外键：`drama_id -> Drama.id`
- 外键：`episode_id -> Episode.id`
- 外键：`highlight_id -> Highlight.id`

字段定义：

- `id`：字符串主键，推荐格式 `ie_xxx`
- `user_id`：服务端根据 `device_id` 生成的稳定匿名用户 ID
- `device_id`：客户端设备唯一标识，必填
- `drama_id`
- `episode_id`
- `highlight_id`
- `interaction_type`：`light_option | shuang_burst | haha_bounce | woc_flip | flame_rage | heart_bloom`
- `option_text`
- `client_timestamp`
- `server_timestamp`

约束：

- 第一版客户端不允许自由输入互动文本
- 同一设备在同一高光点可重复点击，但主统计只按去重规则累计

## 9.5 HighlightStats

主键与关系：

- 主键兼外键：`highlight_id -> Highlight.id`

字段定义：

- `highlight_id`
- `total_count`
- `unique_device_count`
- `heat_level`：`0-4`
- `top_option`
- `option_stats_json`
- `recent_texts_json`
- `last_event_at`
- `updated_at`

说明：

- 该表是聚合快照表，可由 `InteractionEvent` 重算
- 客户端展示优先读该表，避免每次播放时做重聚合

## 9.6 BranchOption

主键与关系：

- 主键：`id`
- 外键：`episode_id -> Episode.id`

字段定义：

- `id`：字符串主键，推荐格式 `bo_xxx`
- `episode_id`
- `title`
- `description`
- `result_type`：第一版固定为 `image_story`
- `result_content_path`：分支结果 manifest 相对路径
- `cover_path`
- `sort_index`
- `status`：`active | hidden`
- `created_at`
- `updated_at`

约束：

- 第一版只有最后一集存在固定分支
- 每个可分支剧集固定存在 `2` 个启用的 `BranchOption`
- `BranchOption` 仍然是尾集的默认可直接消费入口，不依赖用户自定义输入
- 本次方案调整不改变“2 个固定分支 + 1 个自定义分支任务入口”的基础结构，只改变结果资源形态
- 固定分支建议走与自定义分支一致的标准生成流水线，但在内容侧提前离线生成并沉淀 manifest 与配图资源
- 固定分支的方向不建议人工写死，建议复用原有分支生成链路：基于 `story_context_package + tail_state_snapshot` 先生成多条固定分支候选，再选出 `2` 条写入 `BranchOption`

## 9.7 BranchTask

主键与关系：

- 主键：`id`
- 外键：`episode_id -> Episode.id`

字段定义：

- `id`：字符串主键，推荐格式 `bt_xxx`
- `user_id`
- `device_id`
- `episode_id`
- `user_prompt`
- `status`：`pending | running | success | failed | timeout | blocked`
- `result_title`
- `result_hook`
- `result_story`
- `storyboard_json`
- `result_tags_json`
- `result_interaction_options_json`
- `result_source`：建议记录实际产出来源，例如 `deepseek | doubao | mixed | manual`
- `fail_reason`
- `created_at`
- `started_at`
- `finished_at`

说明：

- 第一版自定义分支结果以 `剧情文本 + 分镜结构 + 分镜配图` 为主，不依赖实时视频生成
- `BranchTask` 只允许在最后一集上创建
- 当前仓库实现已经覆盖 `user_prompt / status / result_title / result_hook / result_story / storyboard_json / result_tags_json / result_interaction_options_json / retry_count` 这一层骨架，可支撑“异步文本分支生成 + 结果回看 + 后台重试”
- `BranchTask` 只对应用户主动选择“自定义分支”后的任务，不替代固定分支 `BranchOption`
- 固定分支建议复用与 `BranchTask` 一致的标准生产流水线，只是提前离线生成；固定分支结果建议为 `8 到 12` 张分镜，自定义分支建议为 `5 到 7` 张分镜

### BranchTask 二阶段扩展建议

为了把“文本分支”稳定扩展到“参考素材收集 + 分镜细化 + 分镜配图 + 图文结果展示”，建议在不改动现有客户端基础返回结构的前提下，新增以下字段：

- `branch_type`：`revenge | truth_reveal | misunderstanding_fix | hidden_line | daily_extra | custom`
- `pipeline_stage`：任务内部阶段，见下文状态机
- `story_context_json`：全剧剧情资料包摘要，供本次任务留痕
- `story_context_version`：本次任务命中的剧情上下文包版本号
- `story_context_asset_path`：剧情上下文包完整产物在本地文件系统中的路径
- `tail_state_snapshot_json`：尾集结束状态包，回答“这一刻谁知道什么、谁和谁是什么关系”
- `character_bible_json`：本次任务使用的角色资料包
- `reference_assets_json`：角色参考图、场景参考图、来源 URL、本地缓存路径、参考用途说明
- `prompt_package_json`：模型输入包，包括用户原始提示词、归一化提示词、约束项、禁区、风格要求
- `story_expansion_json`：结构化剧情扩写结果，建议包含 `title / hook / summary / conflict / twist / ending / emotion_tags / cast`
- `shot_prompt_json`：分镜级 prompt 包，作为后续生图输入
- `storyboard_images_json`：分镜配图结果列表，包含镜头图 prompt、参考图依赖、生成状态、资源路径、文案说明
- `storyboard_manifest_json`：面向客户端的图文分镜结果清单，包含图片、标题、旁白、对白、顺序信息
- `narration_payload_json`：旁白与可选 TTS 预留字段，建议包含 `narrationText / narrationVoice / narrationAudioStatus / narrationAudioPath`
- `image_task_status`：`not_started | generating | success | partial_failed | failed | skipped`
- `image_task_payload_json`：提交给文生图 / 图生图服务的任务载荷和返回信息
- `review_notes`：人工审核、失败复盘、模型问题备注

约束建议：

- 第一阶段只强依赖 `branch_type / pipeline_stage / prompt_package_json / story_expansion_json / shot_prompt_json`
- 第二阶段引入 `reference_assets_json / storyboard_images_json / storyboard_manifest_json / image_task_status`
- `narration_payload_json` 可与 `storyboard_manifest_json` 同阶段引入，默认只写文本旁白，不强依赖音频
- 视频生成相关字段降级为未来增强项，不作为当前主链路默认字段
- 所有新增 JSON 字段都应存合法 JSON 字符串，便于当前 SQLite + Prisma 结构平滑演进
- 客户端第一版仍以 `result_title / result_hook / result_story / storyboard_json` 为主消费字段，避免 Android 端一次性跟着大改
- 人物参考图优先复用 `assets/reference/branch-characters/<drama_id>/` 下的三视图；若缺少场景参考图，则允许任务执行阶段在 `assets/reference/branch-scenes/<drama_id>/` 或任务缓存目录下沉淀场景参考资产，后续复用到固定分支与自定义分支

## 10. API 设计与契约

统一响应包结构：

```json
{
  "code": 0,
  "message": "ok",
  "data": {}
}
```

错误响应：

```json
{
  "code": 40001,
  "message": "invalid request",
  "data": null
}
```

规则：

- `code = 0` 表示成功
- 所有时间统一使用毫秒时间戳
- 所有客户端接口返回的资源字段统一为 `url`
- 所有后台接口保留原始 `path` 字段，便于运维和修正

## 10.1 内容接口

- `GET /dramas`
- `GET /dramas/:dramaId/episodes`
- `GET /episodes/:episodeId`

约定：

- `GET /dramas` 返回主打剧、备选剧和继续观看摘要
- `GET /dramas/:dramaId/episodes` 按 `episode_no` 正序返回
- `GET /episodes/:episodeId` 返回剧集基础信息、是否尾集、是否有分支

## 10.2 高光接口

- `GET /episodes/:episodeId/highlights`
- `GET /highlights/:highlightId/stats`

接口返回应包含：

- 高光基础信息
- 互动类别与强度信息
- 历史热度信息
- 高光强度信息

约定：

- 只返回 `status = confirmed` 的高光
- 按 `start_time_ms` 正序返回
- 返回结果中应包含 `interaction_start_ms / interaction_appear_ms / interaction_end_ms`
- 同时返回 `stats` 快照，避免客户端二次请求

## 10.3 互动接口

- `POST /interactions`

请求示例：

```json
{
  "deviceId": "device-001",
  "episodeId": "ep-001",
  "highlightId": "hl-001",
  "interactionType": "light_option",
  "optionText": "爽了",
  "clientTimestamp": 1710000000
}
```

约定：

- 服务端根据 `deviceId` 生成或查找 `userId`
- 接口成功后返回当前高光最新聚合统计

## 10.4 分支接口

- `GET /episodes/:episodeId/branch-options`
- `POST /branch-tasks`
- `GET /branch-tasks/:taskId`
- `GET /users/:userId/branch-tasks`
- `POST /branch-tasks/:taskId/likes`
- `POST /branch-tasks/:taskId/comments`
- `GET /branch-tasks/:taskId/comments`

约定：

- `GET /episodes/:episodeId/branch-options` 仅最后一集可返回数据
- `POST /branch-tasks` 仅最后一集可创建任务
- `BranchTask` 结果页支持点赞和评论
- 评论查询按 `created_at` 倒序分页

分支任务扩展约定：

- `POST /branch-tasks` 第一阶段仍只要求 `deviceId + episodeId + userPrompt`
- 服务端创建任务时同步补齐 `branch_type` 初判和 `prompt_package_json` 初稿，避免后续任务拿到的只是裸 prompt
- `GET /branch-tasks/:taskId` 面向客户端继续返回轻量结构；新增字段优先通过可选字段渐进下发，不破坏 Android 当前 DTO
- 后台详情接口可返回完整上下文包、阶段耗时、素材清单、失败阶段，以及未来可选的视频增强任务信息
- 若后续需要主动刷新而不是纯轮询，可额外补 `POST /branch-tasks/:taskId/refresh`，但第一版不是必需

权限约定：

## 11. 核心技术流程

## 11.1 高光离线处理流程

```text
视频资源准备
  -> 音频抽取
  -> ASR 转写
  -> 文本结构化
  -> LLM 识别高光点
  -> 输出结构化高光 JSON

  -> 写入数据库
```

## 11.2 播放态互动流程

```text
客户端加载剧集
  -> 获取高光列表
  -> 开始播放
  -> 监听播放进度
  -> 命中高光点
  -> 渲染主互动组件
  -> 弱展示历史热度
  -> 用户点击
  -> 播放动效
  -> 上报互动事件
  -> 服务端更新统计
  -> 客户端刷新局部热度展示
```

## 11.3 尾集固定分支流程

```text
播放最后一集结束
  -> 弹出尾集分支弹窗
  -> 请求固定分支选项
  -> 展示 2 个固定分支和 1 个自定义选项
  -> 用户选择固定分支
  -> 返回预生成图文分镜 manifest 与对应图片资源
  -> 用户点赞 / 评论
```

## 11.4 尾集自定义分支流程

```text
播放最后一集结束
  -> 弹出尾集分支弹窗

  -> 客户端轮询任务状态
```

## 12. AI 处理链路设计

## 12.1 高光识别链路

输入：

- 剧集转写文本
- 时间戳信息
- 人物信息
- 文本上下文

输出：

- 高光点区间
- 高光类型
- 强度等级
- 推荐标题
- 推荐互动按钮
- 推荐互动表现
- 置信度

要求：

- 必须输出合法 JSON
- 输出中必须包含可映射到前端类别表现的字段
- 输出结果需可由后台人工修正

## 12.2 分支生成链路

输入：

- 全剧剧情资料包
- 尾集结束状态包
- 固定方向或用户自定义提示词
- 角色资料包
- 场景资料包
- 风格和安全约束

输出：

- 标题
- hook
- 剧情正文
- 分镜结构
- 分镜图片列表
- 图片生成 Prompt 包
- 图文展示 Manifest
- 标签
- 推荐互动按钮

要求：

- 先完成剧情扩写和分镜脚本，再进入分镜配图
- 第一版主产物是 `剧情文本 + 分镜脚本 + 分镜展示图 + 图文展示 Manifest`
- 分镜配图属于当前主链路硬依赖
- 视频生成降级为未来增强项，不作为当前主链路默认目标
- 结果必须可单独展示
- 输出结构可回写到数据库

推荐内部链路：

`剧情上下文包 -> 尾集结束状态包 -> 用户提示词解析 -> 分支类型归类 -> 剧情扩写 -> 分镜脚本 -> 分镜配图 -> 组装图文展示 Manifest`

补充约定：

- 用户提示词是自定义分支生成的必选输入，不能被省略
- 分支任务要保存本次提示词、解析出的分支类型和命中的剧情上下文包版本
- 每个分镜都要记录对应的参考图依赖和生图 prompt，便于用户或 Codex 直接复用生成
- 如果分镜配图失败，仍应保留剧情文本和分镜结果作为降级展示产物

### 12.2.1 分支生成任务的标准输入包

为避免任务执行时临时再去拼装上下文，建议在任务开始后尽早形成 `prompt_package_json`，至少包含：

- `user_prompt_raw`：用户原始输入
- `user_prompt_normalized`：去噪后的提示词
- `branch_type`：归类后的分支方向
- `world_rules`：世界观和人物约束
- `negative_constraints`：禁止项，例如角色性格跑偏、关系逆转、超出现有题材
- `output_contract`：要求生成哪些结构化字段
- `asset_requirements`：需要哪些角色图、场景图、封面图

这个包的目标不是给客户端看，而是保证模型调用、重试、人工复盘都基于同一份稳定输入。

### 12.2.2 剧情上下文包生成流水线

为了让尾集分支生成不依赖“临时拼一段前情文字”，建议在 `BranchTask` 之前增加一条独立的离线预处理工作流，把整部剧处理成可复用的 `story_context_package`。

这条链路的设计目标是：

- 用统一方式回答“之前发生了什么剧情”
- 让模型准确理解跨集因果和人物关系变化
- 在尾集处形成一份可直接续写的状态快照
- 让一部剧的上下文可以被多个 `BranchTask` 复用

推荐处理顺序：

```text
原始视频 / 字幕
  -> 单集台词文本化
  -> 分集层摘要
  -> 时间线层
  -> 全剧层
  -> 尾集快照层
  -> 组装 story_context_package
```

#### 12.2.2.1 五层产物设计

建议固定为以下五层，不建议跳层直接生成：

1. `transcript layer`

- 每一集的台词文本化结果
- 来源可以是 `ASR`、外挂字幕、OCR 补全或三者混合

2. `episode layer`

- 每一集的结构化剧情摘要
- 负责回答“这一集发生了什么”

3. `timeline layer`

- 跨集关键事件时间线
- 负责回答“这些事情按什么顺序发生、彼此因果是什么”

4. `series layer`

- 全剧主线、角色关系、题材风格、世界规则
- 负责回答“这部剧整体讲的是什么、有哪些不能改的设定”

5. `tail snapshot layer`

- 尾集结束瞬间的状态快照
- 负责回答“分支从哪里开始接、哪些事实已经锁死”

#### 12.2.2.2 单集台词文本化

输入建议：

- `episode video`
- 可选外挂字幕文件
- 可选角色名单

输出文件建议：

`episode_transcript.json`

建议字段：

- `episode_id`
- `episode_no`
- `source_type`：`subtitle | asr | mixed`
- `language`
- `segments`

`segments` 内建议包含：

- `segment_id`
- `start_ms`
- `end_ms`
- `speaker_guess`
- `text`
- `confidence`

这一步的原则：

- 只做文本抽取、清洗和基础结构化
- 不在这一层做剧情理解
- 如果已有稳定字幕，优先以字幕为主，再用 ASR 补缺失片段

#### 12.2.2.3 分集层摘要

输入：

- `episode_transcript.json`

输出文件建议：

`episode_summary.json`

建议字段：

- `episode_id`
- `episode_no`
- `summary`
- `key_events`
- `character_changes`
- `new_reveals`
- `open_questions`
- `episode_ending_state`

设计要求：

- 必须是结构化 JSON，而不是单段自然语言总结
- 每一集独立处理，便于局部重跑和人工修订

作用：

- 给时间线层提供稳定事件源
- 给全剧层提供跨集摘要输入

#### 12.2.2.4 时间线层

输入建议：

- 全部 `episode_summary.json`
- 必要时回看对应 `episode_transcript.json`

输出文件建议：

`event_timeline.json`

建议字段：

- `event_id`
- `episode_id`
- `episode_no`
- `order_in_episode`
- `actors`
- `event`
- `cause`
- `impact`
- `resolved`

设计要求：

- 必须可排序，不能只有散点描述
- `cause` 和 `impact` 要尽量显式，不然后续分支生成容易丢因果
- 第一版可以只抽主线关键事件，不要求覆盖所有细枝末节

#### 12.2.2.5 全剧层

输入：

- 全部 `episode_summary.json`
- `event_timeline.json`

输出文件建议：

`series_overview.json`

建议字段：

- `series_premise`
- `genre`
- `main_conflict`
- `tone`
- `character_bible`
- `relationship_graph`
- `canon_constraints`

其中 `character_bible` 建议至少包含：

- `character_name`
- `identity`
- `personality`
- `motivation`
- `relationship_summary`
- `do_not_change`

其中 `canon_constraints` 建议至少包含：

- 不能推翻的既有事实
- 不能跑偏的人设约束
- 不能越界的题材规则

#### 12.2.2.6 尾集快照层

输入：

- 尾集 `episode_transcript.json`
- 尾集 `episode_summary.json`
- `event_timeline.json`
- `series_overview.json`

输出文件建议：

`tail_state_snapshot.json`

建议字段：

- `current_conflict`
- `alive_status`
- `who_knows_what`
- `relationship_states`
- `unresolved_questions`
- `branch_entry_points`
- `hard_constraints`

这一层是后续分支生成的最高优先级输入之一。

推荐优先级：

- `tail_state_snapshot`
- `character_bible`
- `recent_event_timeline`
- `series_overview`

也就是说，分支生成不应直接把全剧原始 transcript 作为主输入。

#### 12.2.2.7 总包组装

最终建议组装成：

`story_context_package.json`

建议结构：

```json
{
  "drama_id": "drama_001",
  "title": "示例剧名",
  "version": "2026-05-31T18:00:00+08:00",
  "series_overview": {},
  "character_bible": [],
  "episode_summaries": [],
  "event_timeline": [],
  "tail_state_snapshot": {},
  "canon_constraints": []
}
```

说明：

- `version` 必须保留，便于 `BranchTask` 记录命中的上下文版本
- `story_context_json` 建议存摘要
- 完整包建议落本地文件，通过 `story_context_asset_path` 指向

#### 12.2.2.8 脚本与目录设计

建议脚本目录：

```text
scripts/story_context/
  01_extract_transcript.py
  02_generate_episode_summary.py
  03_build_event_timeline.py
  04_build_series_overview.py
  05_build_tail_snapshot.py
  06_assemble_story_context.py
  run_story_context_pipeline.py
```

建议产物目录：

```text
artifacts/story_context/{dramaId}/
  transcripts/
  episode_summaries/
  timeline/
  series/
  tail/
  package/
  manifest.json
```

各脚本职责：

- `01_extract_transcript.py`
  - 输入视频或字幕
  - 输出单集 transcript
- `02_generate_episode_summary.py`
  - 输入单集 transcript
  - 输出单集结构化摘要
- `03_build_event_timeline.py`
  - 聚合全部分集摘要
  - 输出关键事件时间线
- `04_build_series_overview.py`
  - 聚合分集摘要和时间线
  - 输出全剧层总览
- `05_build_tail_snapshot.py`
  - 聚合尾集和全剧上下文
  - 输出尾集快照
- `06_assemble_story_context.py`
  - 组装总包并写出版本号
- `run_story_context_pipeline.py`
  - 统一编排执行顺序、失败重试和 manifest 更新

#### 12.2.2.9 manifest 与阶段状态

整条剧情上下文包链路建议维护一个 `manifest.json`，至少记录：

- `drama_id`
- `pipeline_status`
- `completed_steps`
- `failed_steps`
- `episode_progress`
- `model_provider`
- `model_name`
- `started_at`
- `updated_at`
- `version`

建议阶段状态：

- `pending`
- `transcript_ready`
- `episode_summary_ready`
- `timeline_ready`
- `series_overview_ready`
- `tail_snapshot_ready`
- `package_ready`
- `failed`

设计目的：

- 支持剧集级任务重跑
- 支持后台查看当前处理进度
- 支持人工修订后重新组包

#### 12.2.2.10 与 BranchTask 的衔接方式

推荐执行顺序不是：

`用户输入 prompt -> 临时拼全剧上下文 -> 直接生成分支`

而是：

`原剧离线处理 -> story_context_package -> 用户输入 prompt -> 创建 BranchTask -> 读取上下文包 -> 生成分支`

`BranchTask` 执行时建议这样接入：

1. 根据 `episode_id` 找到所属 `drama_id`
2. 读取当前有效的 `story_context_package`
3. 将 `version` 写入 `story_context_version`
4. 将完整文件路径写入 `story_context_asset_path`
5. 将本次实际使用的摘要内容写入 `story_context_json`
6. 将 `tail_state_snapshot` 单独写入 `tail_state_snapshot_json`
7. 基于这些上下文再组装 `prompt_package_json`

这样做的价值：

- 分支任务与原始视频解耦
- 上下文生成和分支生成可以分别排障
- 一部剧的上下文包可以复用于多个用户分支任务

#### 12.2.2.11 第一版实施建议

为了控制比赛阶段复杂度，建议按三步走：

第一步：

- 先做 `transcript -> episode_summary -> timeline -> tail_snapshot -> package`
- 暂不要求角色识别极致准确

第二步：

- 引入 `series_overview.character_bible`
- 补充人工修订能力

第三步：

- 再把上下文包稳定挂到 `BranchTask` 的执行链路中

第一版最小闭环只要稳住这五份产物即可：

- `episode_transcript.json`
- `episode_summary.json`
- `event_timeline.json`
- `tail_state_snapshot.json`
- `story_context_package.json`

### 12.2.3 分支任务内部阶段状态机

对外仍保留 `pending / running / success / failed / timeout / blocked` 六态，便于 Android 继续复用现有枚举。

对内建议新增 `pipeline_stage`，用来表达任务当前卡在哪一段：

```text
queued
  -> context_prepared
  -> prompt_interpreted
  -> story_generated
  -> assets_prepared
  -> storyboard_generated
  -> storyboard_images_generating
  -> storyboard_images_generated
  -> manifest_generated
  -> review_pending
  -> completed
```

失败落点规则：

- 任一阶段模型输出不合法，进入 `failed`，并记录 `fail_reason + pipeline_stage`
- 超时统一映射到外层 `timeout`，内部阶段保留最后执行位置
- 如果命中过审或内容安全规则，进入 `blocked`
- 如果文本和分镜成功，但配图失败，可保持 `status = success`，同时 `image_task_status = partial_failed`
- 如果配图成功但 manifest 组装失败，进入 `failed`，并保留已生成图片结果供排障

### 12.2.4 结构化产物落库策略

建议将分支任务结果拆成四层：

- `客户端结果层`
  - `result_title`
  - `result_hook`
  - `result_story`
  - `storyboard_json`
- `剧情上下文层`
  - `story_context_version`
  - `story_context_asset_path`
  - `story_context_json`
  - `tail_state_snapshot_json`
- `任务执行层`
  - `pipeline_stage`
  - `prompt_package_json`
  - `story_expansion_json`
  - `shot_prompt_json`
  - `fail_reason`
- `图文资产层`
  - `reference_assets_json`
  - `storyboard_images_json`
  - `storyboard_manifest_json`
  - `image_task_status`
  - `image_task_payload_json`
- `可选增强层`
  - 后续如果重新引入视频增强，再额外补 `video_*` 字段，不进入当前主里程碑

这样做的目的是：

- Android 端继续消费稳定字段
- 上下文包与分支结果能够分别追溯版本
- 后台能看到完整生成证据链
- 分镜结果即使没有视频，也能形成稳定的可展示骨架
- 后续如果重新启用视频增强，也不会破坏当前图文主链路

### 12.2.4.1 分镜生图合同

由于当前分支主路线调整为“图文结合的分镜小说”，分镜结果不能再停留在“只有一句描述 + 一个 duration”的粗粒度结构。

从这一版开始，建议将 `shot_prompt_json` 升级为 `分镜生图包`，用于直接提交文生图 / 图生图模型，并同时驱动客户端图文展示。

推荐约束：

- 固定分支建议生成 `8 到 12` 张分镜图
- 自定义分支建议生成 `5 到 7` 张分镜图
- 每张图必须对应一段明确剧情作用
- 每张图必须能挂一段图下长旁白或对白，不允许只有视觉没有文案
- 每张图都要显式声明依赖的角色参考图和场景参考图

`storyboard_json` 仍保持轻量兼容，只保留：

- `scene`
- `description`
- `duration`

`shot_prompt_json` 则升级为更完整的对象，建议至少包含：

- `contractVersion`
- `storyTitle`
- `storyHook`
- `readingMode`
- `visualStyle`
- `globalCharacterConsistencyNotes`
- `globalSceneConsistencyNotes`
- `shots`

其中 `shots` 内每个分镜建议至少包含：

- `scene`
- `sceneTitle`
- `storyBeat`
- `plotPurpose`
- `narrationText`
- `narrationPlacement`
- `dialogueText`
- `subtitleText`
- `visualFocus`
- `emotion`
- `camera`
- `location`
- `characters`
- `assetReferences`
- `referenceTaskImages`
- `imagePrompt`
- `negativePrompt`
- `layoutHint`

这样做的价值是：

- 分镜图、旁白文案和对白说明天然绑定，不会“图和文各说各的”
- 角色参考图、场景参考图和任务参考图都能被明确追踪
- 用户或 Codex 可以直接复用单张分镜的 prompt 去生成展示图
- 前端可以直接按分镜顺序渲染为上下滑漫画式结果页

### 12.2.4.2 分镜任务参考图约定

为了让用户或 Codex 能直接生成某一张分镜展示图，建议每个分镜任务都显式维护 `referenceTaskImages`。

推荐结构：

- `characterRefs`
  当前分镜必须依赖的人物参考图
- `sceneRefs`
  当前分镜必须依赖的场景参考图
- `styleRefs`
  当前分镜的风格参考图，可为空
- `carryNotes`
  这一张图必须继承上一张图的哪些连续性特征

每个引用对象建议至少包含：

- `assetId`
- `assetType`
- `assetPath`
- `usage`
- `priority`

这样做的目的：

- 让同一分镜的生图输入具备可复现性
- 让后台能清楚看到每张图到底依赖了哪些素材
- 让用户或 Codex 可以单独拎出一张分镜，直接复用同一套参考图与 prompt 重新生成

当前已确认可复用的首批角色三视图目录为：

- `assets/reference/branch-characters/drama_001/`

命名建议继续保持：

- `<character_slug>-three-view-v1.png`

这样后续 `reference_assets_json` 可以直接稳定引用本地资产路径。

### 12.2.5 当前代码骨架与建议扩展位

当前仓库已经具备以下实现基础：

- `server/src/modules/branch/routes.ts`
  - 创建任务、查询任务、点赞评论
- `server/src/services/taskQueue/index.ts`
  - 进程内轮询队列
  - 单任务串行执行
  - 超时转 `timeout`
- `server/src/services/clientPayload/index.ts`
  - 客户端任务字段整形
- `server/src/modules/admin/routes.ts`
  - 后台任务列表、详情、重试

建议下一步扩展方式：

- 保留 `taskQueue/index.ts` 作为调度入口，不直接在这里堆叠所有业务逻辑
- 新增 `server/src/services/branchTask/` 下的任务子服务，例如：
  - `contextBuilder`
  - `promptInterpreter`
  - `storyGenerator`
  - `assetCollector`
  - `storyboardGenerator`
  - `storyboardImageGenerator`
  - `storyboardManifestBuilder`
- `processTask()` 只负责任务状态流转和阶段编排，每一步实际逻辑交由独立服务执行
- 每个子服务返回结构化结果，统一由 `processTask()` 回写数据库

## 12.3 任务执行策略

建议采用异步任务设计：

- 用户请求先入库
- 服务端后台消费任务
- 返回 `pending/running/success/failed/timeout`
- 客户端轮询或定时刷新结果

补充约定：

- 第一版采用服务端进程内单任务队列，默认并发度 `1`
- 任务执行超时后标记为 `timeout`
- 管理后台可对 `failed/timeout` 任务执行重试

### 12.3.1 当前实现现状

当前代码已经落地的策略是：

- `POST /branch-tasks` 入库一条 `status = pending` 的任务
- `server/src/services/taskQueue/index.ts` 每 `5s` 轮询一条最早的 `pending` 任务
- 任务开始时写入 `status = running` 和 `started_at`
- 默认生成器产出 `resultTitle / resultHook / resultStory / storyboardJson / resultTagsJson / resultInteractionOptionsJson`
- 完成后写入 `status = success`
- 超时或异常写入 `status = timeout | failed` 和 `fail_reason`
- 后台通过 `/admin/branch-tasks/:taskId/retry` 重试，最多 `3` 次

这套实现已经满足比赛第一阶段“异步文本分支生成”的最低闭环，不建议推翻重来；后续重点是在此基础上补 `分镜配图` 这层可视化产物。

### 12.3.2 下一阶段建议的任务编排方式

为了支持更复杂的分支生成，建议把单个 `taskGenerator(task)` 升级为分阶段执行：

```text
prepareContext(task)
  -> interpretPrompt(task, context)
  -> generateStory(task, context, prompt)
  -> collectAssets(task, story)
  -> generateStoryboard(task, story, assets)
  -> generateStoryboardImages(task, storyboard, assets)
  -> buildStoryboardManifest(task, storyboard, images)
  -> finalizeTask(task)
```

执行原则：

- 每个阶段结束都落库一次，更新 `pipeline_stage`
- 阶段间只传结构化对象，不直接传大段自然语言拼接串
- `generateStoryboardImages` 是比赛第一版主链路的一部分，不应视作可有可无
- `buildStoryboardManifest` 负责把图片、标题、旁白、对白整理成前端直接消费的图文结构
- 所有模型返回都先做 JSON 校验，再更新主结果字段

### 12.3.3 重试与人工介入策略

重试不应简单理解成“整任务重跑”，建议分成两类：

- `系统级重试`
  - 模型超时
  - 网络失败
  - 图片生成服务临时不可用
  - 可选视频增强平台临时不可用
- `内容级重试`
  - 剧情跑偏
  - 人设不一致
  - 分镜不可用
  - 分镜图风格不一致

第一版后台可先继续保留“整任务重试”。

第二版建议后台详情页增加：

- 当前 `pipeline_stage`
- 每阶段耗时
- 本次使用的 `prompt_package_json`
- 本次命中的 `reference_assets_json`
- 文本结果预览
- 分镜结果预览
- 分镜配图预览
- 图片生成状态与返回
- 可选视频增强任务状态与返回
- 人工备注与重新执行入口

### 12.3.4 分阶段验收标准

建议按两层验收：

- `分镜可视化分支 MVP`
  - 能创建任务
  - 能生成标题、hook、剧情、分镜
  - 能生成分镜配图
  - 失败可重试
  - 后台能看详情
- `视频分支增强版`
  - 能输出角色/场景素材清单
  - 能提交视频生成任务
  - 视频失败不影响文本、分镜、配图展示
  - 后台能定位失败阶段并复跑

```text
脚本输出高光 JSON
  -> 导入候选高光 status=candidate
  -> 后台人工校准
  -> 确认后的高光 status=confirmed

```

## 13. 本地部署方案

## 13.1 本地部署形态

建议采用以下部署方式：

- Android 真机运行客户端
- 笔记本或个人电脑运行本地 API 服务端
- 同一台电脑运行本地 Web 管理后台
- 视频资源与数据库均存于本地目录

## 13.2 启动方式建议

建议提供最少三条启动命令：

- 启动服务端
- 启动管理后台
- 初始化数据库与演示数据

例如：

```bash
pnpm install
pnpm db:seed
pnpm dev:server
pnpm dev:admin
```

## 13.3 局域网访问建议

为方便 Android 真机联调，建议：

- 服务端监听 `0.0.0.0`
- 管理后台也可监听局域网地址
- 使用固定局域网 IP 或 mDNS
- 在 Android 配置中可切换本地 API Base URL

## 13.4 CORS 与静态资源策略

- 服务端需开放 Android 客户端和本地 Web 后台访问
- 视频、封面、固定分支图文资源通过静态资源路由暴露
- 客户端统一通过 HTTP 访问服务端提供的资源地址，不直接读取电脑文件路径

## 14. 性能与稳定性设计

## 14.1 客户端

- 视频播放优先级高于特效
- 同屏只允许一个主互动组件
- 高光过近时第二个点弱化展示
- 弹幕、贴纸和粒子效果需要设置总量上限
- 非关键统计更新不阻塞 UI

## 14.2 服务端

- 高光列表查询要走索引
- 互动写入与统计聚合应分离
- 分支任务执行不得阻塞主 API 线程
- 管理后台查询接口允许分页和筛选

## 14.3 AI 链路

- 模型输出解析失败时自动重试一次
- 高光识别采用离线批处理，避免播放时调用模型
- 自定义分支采用异步任务，避免请求超时

## 15. 异常与降级方案

## 15.1 客户端降级

- 高光接口失败：继续播放视频，隐藏互动层或读取缓存
- 互动上报失败：本地先展示效果，后台异步补写
- 分支任务查询失败：保留“稍后刷新”入口

## 15.2 服务端降级

- 模型调用失败：分支任务标记失败并保留固定分支能力
- 高光提取失败：允许手动录入高光
- 数据库异常：优先保障只读播放链路

## 16. 安全与配置管理

建议：

- 所有模型密钥写入环境变量
- 仓库提交 `.env.example`
- `.env` 加入 `.gitignore`
- 视频资源不直接入仓库
- 管理后台接口至少具备本地访问控制或简单鉴权

补充：

- `ADMIN_TOKEN` 作为后台最小鉴权凭据
- `device_id` 必填，用于匿名用户稳定识别
- 评论和自定义 Prompt 需经过敏感词过滤
- AI 输出需经过结构校验和基础安全检查后落库
