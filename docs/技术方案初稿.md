# Drama Pulse 技术方案初稿

## 1. 文档目的

本文档基于 [产品方案初稿](/Users/a0000/Desktop/项目文件/drama-pulse/docs/产品方案初稿.md)，从技术落地视角定义 Drama Pulse 的整体架构、模块边界、核心流程、数据模型、接口能力、AI 处理链路与本地部署方案。

本文档的目标不是直接替代详细设计文档，而是作为后续以下工作的技术基础：

- Android 客户端开发拆解
- 本地服务端开发拆解

- 管理后台设计
- 数据库与接口设计
- 开发排期与联调计划

## 2. 技术目标

### 2.1 核心目标

- 支撑短剧播放、高光互动、历史互动沉淀、尾集分支 AIGC 的完整闭环
- 保障本地部署环境下的稳定性和可演示性
- 支撑后续文档、开发、答辩中的技术表达

### 2.2 技术原则

- 主链路优先稳定，不将复杂能力堆到实时播放链路
- 所有 AI 重任务尽量前置离线化或异步化
- 客户端渲染链路轻量化，优先保证视频播放稳定
- 服务端职责清晰，模块可替换、可扩展
- 所有能力均以“本地演示稳定”为第一约束

## 3. 技术范围

当前版本技术范围覆盖：

- Android 客户端
- 本地服务端
- 本地 Web 管理后台
- 离线高光提取与标注链路
- 历史互动统计与回流链路
- 尾集固定分支与自定义分支任务链路

当前版本不追求：

- 公网部署
- 多人实时联机
- 实时视频理解
- 实时视频生成
- 跨端统一代码架构

## 4. 总体架构

系统建议采用四层结构：

1. `客户端层`
Android App，负责短剧列表、视频播放、高光触发、互动渲染、尾集分支入口与结果展示。

2. `服务层`
本地服务端，负责剧集管理、高光标签下发、互动事件写入、历史统计聚合、分支任务管理、AI 任务编排。

3. `管理层`
本地 Web 管理后台，负责高光修正、资源配置、互动查看、任务管理、演示数据初始化。

4. `AI 处理层`
包括高光提取离线链路和分支生成异步链路，负责调用 ASR 与大模型能力，并输出结构化结果。

## 6. 技术选型建议

## 6.1 Android 客户端

建议技术栈：

- `Kotlin`
- `Jetpack Compose`
- `Media3 / ExoPlayer`
- `ViewModel + StateFlow`
- `Retrofit + OkHttp`
- `Room` 或轻量本地缓存
- `Coil` 用于图片加载

选择理由：

- Kotlin + Compose 开发效率高，适合单人快速迭代
- Media3 在 Android 视频播放上更稳，适合定制时间轴监听
- StateFlow 更适合播放状态、高光状态、任务状态管理

## 6.2 本地服务端

建议技术栈：

- `Node.js + TypeScript`
- `Express` 或 `Fastify`
- `SQLite`
- `Prisma` 或 `Drizzle ORM`
- `Zod` 用于接口输入输出校验

选择理由：

- Node.js 全栈开发效率高，适合单人比赛项目
- TypeScript 在接口、数据结构和 AI 输出解析上更容易控类型
- SQLite 足够支撑本地部署、低运维、易备份

如果你更偏向 Python，也可备选：

- `Python + FastAPI + SQLModel/SQLAlchemy + SQLite`

但从 Android + Web 管理后台 + 本地 API 一体协作效率看，推荐优先使用 Node.js。

## 6.3 本地 Web 管理后台

建议技术栈：

- `Next.js` 或 `Vite + React`
- `TypeScript`
- `TanStack Query`
- `Tailwind CSS`

选择理由：

- 页面结构清晰，适合快速搭建管理后台
- 与 Node 服务端数据结构更好统一
- 更适合比赛答辩展示和快速联调

## 6.4 AI 处理链路

建议能力来源：

- `ASR`：优先使用稳定的语音转文本能力
- `LLM`：高光识别、标签生成、尾集分支剧情扩写
- 可选的视频结果：后续扩展，不作为主依赖

模型策略：

- 高光识别：离线批处理
- 分支生成：异步任务处理
- 所有模型输出统一为结构化 JSON

## 7. 核心模块设计

## 7.1 Android 客户端模块

`DramaList Module`

- 获取短剧列表
- 展示主打剧和备选剧
- 维护继续观看入口

`Player Module`

- 视频播放与播控
- 播放进度监听
- 视频结束检测

`Episode Selector Module`

- 播放页内切换剧集
- 更新当前视频与高光数据

`Highlight Trigger Module`

- 缓存高光点列表
- 基于播放进度命中高光点
- 管理单次播放会话内的触发去重
- 处理“高光点过近”时的弱化展示逻辑

`Interaction Render Module`

- 渲染互动按钮
- 渲染粒子、贴纸、弹幕与热度反馈
- 控制互动组件生命周期

`Branch Module`

- 检测最后一集结束
- 展示尾集分支入口
- 请求固定分支与创建自定义任务
- 展示结果、点赞、评论

`Network Module`

- 调用服务端接口
- 统一错误处理与重试

`Local Cache Module`

- 缓存短剧列表、剧集信息、高光列表、用户分支历史
- 在弱网或局部接口失败时作为降级手段

## 7.2 服务端模块

`Content Service`

- 管理短剧、剧集、资源路径
- 提供短剧和剧集查询

`Highlight Service`

- 管理高光标签的存储、查询、状态变更
- 提供按集下发的高光点列表

`Interaction Service`

- 记录用户互动行为
- 聚合同一高光点的统计数据
- 提供历史互动回流数据

`Branch Service`

- 返回固定分支配置
- 创建与查询自定义分支任务
- 持久化分支结果、点赞和评论

`AI Orchestrator`

- 封装模型调用
- 负责 Prompt 拼接、结构校验、解析重试、结果落库

`Admin Service`

- 支撑后台管理操作
- 提供高光修正、资源配置、任务管理、演示数据重置

## 11. 核心技术流程

## 11.1 高光离线处理流程

```text
视频资源准备
  -> 音频抽取
  -> ASR 转写
  -> 文本结构化
  -> LLM 识别高光点
  -> 输出结构化高光 JSON
  -> 人工复核与修正
  -> 写入数据库
  -> 客户端播放时按集下发
```

## 11.2 播放态互动流程

```text
客户端加载剧集
  -> 获取高光列表
  -> 开始播放
  -> 监听播放进度
  -> 命中高光点
  -> 渲染主互动组件
  -> 弱展示历史热度
  -> 用户点击
  -> 播放动效
  -> 上报互动事件
  -> 服务端更新统计
  -> 客户端刷新局部热度展示
```

## 11.3 尾集固定分支流程

```text
播放最后一集结束
  -> 展示尾集分支入口
  -> 请求固定分支选项
  -> 用户选择分支
  -> 返回预生成短视频或结果内容
  -> 用户点赞 / 评论
```

## 11.4 尾集自定义分支流程

```text
播放最后一集结束
  -> 用户输入 Prompt
  -> 创建 BranchTask
  -> 服务端异步执行剧情扩写
  -> 输出结构化结果
  -> 写入任务结果
  -> 客户端轮询任务状态
  -> 生成完成后进入结果页
  -> 用户点赞 / 评论
```

## 12. AI 处理链路设计

## 12.1 高光识别链路

输入：

- 剧集转写文本
- 时间戳信息
- 人物信息
- 文本上下文

输出：

- 高光点区间
- 高光类型
- 强度等级
- 推荐标题
- 推荐互动按钮
- 推荐模板
- 置信度

要求：

- 必须输出合法 JSON
- 输出中必须包含可映射到前端模板的字段
- 输出结果需可由后台人工修正

## 12.2 分支生成链路

输入：

- 原剧摘要
- 最后一集剧情摘要
- 固定方向或用户自定义 Prompt
- 风格和安全约束

输出：

- 标题
- hook
- 剧情正文
- 分镜结构
- 标签
- 推荐互动按钮

要求：

- 不直接依赖实时视频生成
- 结果必须可单独展示
- 输出结构可回写到数据库

## 12.3 任务执行策略

建议采用异步任务设计：

- 用户请求先入库
- 服务端后台消费任务
- 返回 `pending/running/success/failed/timeout`
- 客户端轮询或定时刷新结果

## 13. 本地部署方案

建议采用以下部署方式：

- Android 真机运行客户端
- 笔记本或个人电脑运行本地 API 服务端

- 视频资源与数据库均存于本地目录

建议提供最少三条启动命令：

- 启动服务端

- 初始化数据库与演示数据

```bash

pnpm dev:server

```

## 13.3 局域网访问建议

为方便 Android 真机联调，建议：

- 服务端监听 `0.0.0.0`
- 管理后台也可监听局域网地址
- 使用固定局域网 IP 或 mDNS
- 在 Android 配置中可切换本地 API Base URL

## 14. 性能与稳定性设计

## 14.1 客户端

- 视频播放优先级高于特效
- 同屏只允许一个主互动组件
- 高光过近时第二个点弱化展示
- 弹幕、贴纸和粒子效果需要设置总量上限
- 非关键统计更新不阻塞 UI

## 14.2 服务端

- 高光列表查询要走索引
- 互动写入与统计聚合应分离
- 分支任务执行不得阻塞主 API 线程
- 管理后台查询接口允许分页和筛选

## 14.3 AI 链路

- 模型输出解析失败时自动重试一次
- 高光识别采用离线批处理，避免播放时调用模型
- 自定义分支采用异步任务，避免请求超时

## 15. 异常与降级方案

## 15.1 客户端降级

- 高光接口失败：继续播放视频，隐藏互动层或读取缓存
- 互动上报失败：本地先展示效果，后台异步补写
- 分支任务查询失败：保留“稍后刷新”入口

## 15.2 服务端降级

- 模型调用失败：分支任务标记失败并保留固定分支能力
- 高光提取失败：允许手动录入高光
- 数据库异常：优先保障只读播放链路

## 16. 安全与配置管理

建议：

- 所有模型密钥写入环境变量
- 仓库提交 `.env.example`
- `.env` 加入 `.gitignore`
- 视频资源不直接入仓库
- 管理后台接口至少具备本地访问控制或简单鉴权

## 17. 验收导向的技术完成标准

从技术视角，第一版完成标准建议定义为：

- Android 客户端可稳定播放指定短剧
- 播放页能按时间轴触发高光互动
- 用户互动可写入本地服务端并形成统计
- 历史互动可回流到播放页
- 最后一集结束后可展示 2 个固定分支
- 用户可提交自定义分支任务并看到生成结果
- 分支结果页支持点赞与评论
- 本地 Web 管理后台可完成高光管理、任务查看、数据初始化
- 高光识别链路可产出结构化标签并导入服务端

## 18. 开发建议顺序

建议按以下顺序推进：

1. 服务端基础骨架与数据库
2. Android 播放闭环
3. 高光下发与触发
4. 互动渲染与上报
5. 历史互动统计回流
6. 固定分支结果链路
7. 自定义分支任务链路
8. 本地 Web 管理后台
9. 高光离线处理脚本与导入工具
10. 演示数据与资源配置
