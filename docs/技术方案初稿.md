# Drama Pulse 技术方案初稿

## 1. 文档目的

本文档基于 [产品方案初稿](/Users/a0000/Desktop/项目文件/drama-pulse/docs/产品方案初稿.md)，从技术落地视角定义 Drama Pulse 的整体架构、模块边界、核心流程、数据模型、接口能力、AI 处理链路与本地部署方案。

本文档的目标不是直接替代详细设计文档，而是作为后续以下工作的技术基础：

- 数据库与接口设计
- 开发排期与联调计划

## 2. 技术目标

### 2.1 核心目标

- 支撑短剧播放、高光互动、历史互动沉淀、尾集分支 AIGC 的完整闭环
- 保障本地部署环境下的稳定性和可演示性
- 支撑后续文档、开发、答辩中的技术表达

### 2.2 技术原则

- 主链路优先稳定，不将复杂能力堆到实时播放链路
- 所有 AI 重任务尽量前置离线化或异步化
- 客户端渲染链路轻量化，优先保证视频播放稳定
- 服务端职责清晰，模块可替换、可扩展
- 所有能力均以“本地演示稳定”为第一约束

## 3. 技术范围

当前版本技术范围覆盖：

- Android 客户端
- 本地服务端
- 本地 Web 管理后台
- 离线高光提取与标注链路
- 历史互动统计与回流链路
- 尾集固定分支与自定义分支任务链路

当前版本不追求：

- 公网部署
- 多人实时联机
- 实时视频理解
- 实时视频生成
- 跨端统一代码架构

## 4. 总体架构

系统建议采用四层结构：

1. `客户端层`
Android App，负责短剧列表、视频播放、高光触发、互动渲染、尾集分支入口与结果展示。

2. `服务层`
本地服务端，负责剧集管理、高光标签下发、互动事件写入、历史统计聚合、分支任务管理、AI 任务编排。

3. `管理层`
本地 Web 管理后台，负责高光修正、资源配置、互动查看、任务管理、演示数据初始化。

4. `AI 处理层`
包括高光提取离线链路和分支生成异步链路，负责调用 ASR 与大模型能力，并输出结构化结果。

## 6. 冻结技术选型

## 6.1 Android 客户端

冻结技术栈：

- `Kotlin`
- `Jetpack Compose`
- `Media3 / ExoPlayer`
- `ViewModel + StateFlow`
- `Retrofit + OkHttp`
- `Room`
- `Coil` 用于图片加载

选择理由：

- Kotlin + Compose 开发效率高，适合单人快速迭代
- Media3 在 Android 视频播放上更稳，适合定制时间轴监听
- StateFlow 更适合播放状态、高光状态、任务状态管理

## 6.2 本地服务端

冻结技术栈：

- `Node.js + TypeScript`
- `Fastify`
- `SQLite`
- `Prisma`
- `Zod` 用于接口输入输出校验

选择理由：

- Node.js 全栈开发效率高，适合单人比赛项目
- TypeScript 在接口、数据结构和 AI 输出解析上更容易控类型
- SQLite 足够支撑本地部署、低运维、易备份

## 6.3 本地 Web 管理后台

冻结技术栈：

- `Vite + React`
- `TypeScript`
- `React Router`
- `TanStack Query`
- `Tailwind CSS`

选择理由：

- 页面结构清晰，适合快速搭建管理后台
- 与 Node 服务端数据结构更好统一
- 更适合比赛答辩展示和快速联调

## 6.4 工具链版本基线

为了避免后续开发环境和依赖矩阵反复变动，当前建议冻结以下工具链基线：

- Android Gradle Plugin：`8.7.x`
- Gradle Wrapper：`8.10.x`
- Kotlin：`2.0.x`
- JDK：`17`
- Node.js：`20 LTS`
- pnpm：`9.x`
- Python：`3.11`
- SQLite：`3.x`

说明：

- Android 工程允许在小版本内升级，但不建议在项目中途跨大版本升级 AGP、Gradle 或 Kotlin。
- 服务端与后台统一采用 Node.js 20 LTS，避免本地开发和 CI 环境不一致。

## 6.5 AI 处理链路

建议能力来源：

- `ASR`：优先使用稳定的语音转文本能力
- `LLM`：高光识别、标签生成、尾集分支剧情扩写
- 可选的视频结果：后续扩展，不作为主依赖

模型策略：

- 高光识别：离线批处理
- 分支生成：异步任务处理
- 所有模型输出统一为结构化 JSON

## 7. 核心模块设计

## 7.1 Android 客户端模块

`DramaList Module`

- 获取短剧列表
- 展示主打剧和备选剧
- 维护继续观看入口

`Player Module`

- 视频播放与播控
- 播放进度监听
- 视频结束检测

`Episode Selector Module`

- 播放页内切换剧集
- 更新当前视频与高光数据

`Highlight Trigger Module`

- 缓存高光点列表
- 基于播放进度命中高光点
- 管理单次播放会话内的触发去重
- 处理“高光点过近”时的弱化展示逻辑

`Interaction Render Module`

- 渲染互动按钮
- 渲染粒子、贴纸、弹幕与热度反馈
- 控制互动组件生命周期

`Branch Module`

- 检测最后一集结束
- 展示尾集分支入口
- 请求固定分支与创建自定义任务
- 展示结果、点赞、评论

`Network Module`

- 调用服务端接口
- 统一错误处理与重试

`Local Cache Module`

- 缓存短剧列表、剧集信息、高光列表、用户分支历史
- 在弱网或局部接口失败时作为降级手段

## 7.2 服务端模块

`Content Service`

- 管理短剧、剧集、资源路径
- 提供短剧和剧集查询

`Highlight Service`

- 管理高光标签的存储、查询、状态变更
- 提供按集下发的高光点列表

`Interaction Service`

- 记录用户互动行为
- 聚合同一高光点的统计数据
- 提供历史互动回流数据

`Branch Service`

- 返回固定分支配置
- 创建与查询自定义分支任务
- 持久化分支结果、点赞和评论

`AI Orchestrator`

- 封装模型调用
- 负责 Prompt 拼接、结构校验、解析重试、结果落库

`Admin Service`

- 支撑后台管理操作
- 提供高光修正、资源配置、任务管理、演示数据重置

## 8. 数据存储设计

## 8.1 存储划分

建议采用“结构化数据 + 本地文件资源”双存储设计。

结构化数据存入 `SQLite`：

- 短剧元信息
- 剧集元信息
- 高光标签
- 互动事件
- 高光统计
- 固定分支配置
- 自定义分支任务
- 点赞评论数据

文件资源存入本地目录：

- 视频文件
- 封面图
- 分支预生成短视频
- AI 中间结果 JSON
- 分镜草稿导出结果

## 8.3 资源路径约定

为了避免数据库、接口和本地文件路径三套口径不一致，当前约定如下：

- 数据库中只存 `path`，不直接存最终可访问 URL
- 服务端根据 `path` 拼接为客户端可访问的 `url`
- 所有 `path` 都应为相对项目根目录的相对路径

具体约定：

- 剧集视频：`videos/<drama_title>/第N集.mp4`
- 剧封面：`assets/dramas/covers/<drama_id>.jpg`
- 固定分支结果：`assets/dramas/branches/<episode_id>/<option_id>.mp4`
- 高光导出 JSON：`data/exports/highlights/<episode_id>.json`
- 分支结果导出 JSON：`data/exports/branches/<task_id>.json`

## 8.4 演示数据与真实数据划分

为了支持答辩演示和后续真实操作，服务端需明确区分两类数据：

- `seed data`：初始化剧集、高光、固定分支和基础互动统计
- `runtime data`：运行过程中真实产生的互动、任务、点赞和评论

管理后台中的“演示数据重置”只重置 `runtime data`，不覆盖固定内容资源和基础配置。

## 9. 数据模型与字段定义

数据模型延续产品方案中的核心对象，并补充技术实现建议。

## 9.1 Drama

主键与关系：

- 主键：`id`
- 一对多关联：`Drama 1 -> N Episode`

字段定义：

- `id`：字符串主键，推荐格式 `drama_xxx`
- `title`：剧名，唯一
- `description`：剧简介
- `cover_path`：封面相对路径
- `tags_json`：标签数组 JSON
- `main_genre`：主类型，例如 `爽文`、`情感`、`悬疑`
- `is_featured`：是否主推剧
- `display_order`：展示排序
- `status`：`active | hidden`
- `created_at`
- `updated_at`

## 9.2 Episode

主键与关系：

- 主键：`id`
- 外键：`drama_id -> Drama.id`
- 一对多关联：`Episode 1 -> N Highlight`
- 一对多关联：`Episode 1 -> N BranchOption`
- 一对多关联：`Episode 1 -> N BranchTask`

字段定义：

- `id`：字符串主键，推荐格式 `ep_xxx`
- `drama_id`
- `episode_no`：集数，整型，剧内唯一
- `title`
- `video_path`：视频相对路径
- `duration_ms`
- `summary`
- `is_final_episode`
- `has_branch`：第一版仅在最后一集为 `true`
- `status`：`active | hidden`
- `created_at`
- `updated_at`

## 9.3 Highlight

主键与关系：

- 主键：`id`
- 外键：`episode_id -> Episode.id`
- 一对多关联：`Highlight 1 -> N InteractionEvent`
- 一对一聚合：`Highlight 1 -> 1 HighlightStats`

字段定义：

- `id`：字符串主键，推荐格式 `hl_xxx`
- `episode_id`
- `start_time_ms`
- `end_time_ms`
- `type`：见枚举定义
- `title`
- `description`
- `intensity`：`1-5`
- `template_id`：见模板枚举
- `interaction_options_json`：固定模板化按钮数组，第一版不支持自由文本互动
- `visual_effect_type`
- `source`：`manual | ai | ai_adjusted | heat_feedback`
- `confidence`：`0-1`
- `status`：`candidate | confirmed | disabled`
- `created_at`
- `updated_at`

约束：

- `start_time_ms < end_time_ms`
- 同一 `episode_id` 下高光按 `start_time_ms` 排序
- 客户端只消费 `status = confirmed` 的高光点

## 9.4 InteractionEvent

主键与关系：

- 主键：`id`
- 外键：`drama_id -> Drama.id`
- 外键：`episode_id -> Episode.id`
- 外键：`highlight_id -> Highlight.id`

字段定义：

- `id`：字符串主键，推荐格式 `ie_xxx`
- `user_id`：服务端根据 `device_id` 生成的稳定匿名用户 ID
- `device_id`：客户端设备唯一标识，必填
- `drama_id`
- `episode_id`
- `highlight_id`
- `interaction_type`：`emotion_button | vote_side | boost_action | suspense_lock`
- `option_text`
- `client_timestamp`
- `server_timestamp`

约束：

- 第一版客户端不允许自由输入互动文本
- 同一设备在同一高光点可重复点击，但主统计只按去重规则累计

## 9.5 HighlightStats

主键与关系：

- 主键兼外键：`highlight_id -> Highlight.id`

字段定义：

- `highlight_id`
- `total_count`
- `unique_device_count`
- `heat_level`：`0-4`
- `top_option`
- `option_stats_json`
- `recent_texts_json`
- `last_event_at`
- `updated_at`

说明：

- 该表是聚合快照表，可由 `InteractionEvent` 重算

## 10. API 设计与契约

统一响应包结构：

```json
{
  "code": 0,
  "message": "ok",
  "data": {}
}
```

错误响应：

```json
{
  "code": 40001,
  "message": "invalid request",
  "data": null
}
```

规则：

- `code = 0` 表示成功
- 所有时间统一使用毫秒时间戳
- 所有客户端接口返回的资源字段统一为 `url`
- 所有后台接口保留原始 `path` 字段，便于运维和修正

## 10.1 内容接口

- `GET /dramas`
- `GET /dramas/:dramaId/episodes`
- `GET /episodes/:episodeId`

约定：

- `GET /dramas` 返回主打剧、备选剧和继续观看摘要
- `GET /dramas/:dramaId/episodes` 按 `episode_no` 正序返回
- `GET /episodes/:episodeId` 返回剧集基础信息、是否尾集、是否有分支

## 10.2 高光接口

- `GET /episodes/:episodeId/highlights`
- `GET /highlights/:highlightId/stats`

接口返回应包含：

- 高光基础信息
- 互动模板信息
- 历史热度信息
- 高光强度信息

约定：

- 只返回 `status = confirmed` 的高光
- 按 `start_time_ms` 正序返回
- 同时返回 `stats` 快照，避免客户端二次请求

## 10.3 互动接口

- `POST /interactions`

请求示例：

```json
{
  "deviceId": "device-001",
  "episodeId": "ep-001",
  "highlightId": "hl-001",
  "interactionType": "emotion_button",
  "optionText": "爽了",
  "clientTimestamp": 1710000000
}
```

约定：

- 服务端根据 `deviceId` 生成或查找 `userId`
- 接口成功后返回当前高光最新聚合统计

## 10.4 分支接口

- `GET /episodes/:episodeId/branch-options`
- `POST /branch-tasks`
- `GET /branch-tasks/:taskId`
- `GET /users/:userId/branch-tasks`
- `POST /branch-tasks/:taskId/likes`
- `POST /branch-tasks/:taskId/comments`
- `GET /branch-tasks/:taskId/comments`

约定：

- `GET /episodes/:episodeId/branch-options` 仅最后一集可返回数据
- `POST /branch-tasks` 仅最后一集可创建任务
- `BranchTask` 结果页支持点赞和评论
- 评论查询按 `created_at` 倒序分页

## 10.5 管理后台接口

- `GET /admin/dramas`
- `GET /admin/episodes`
- `GET /admin/highlights`
- `PATCH /admin/highlights/:highlightId`
- `POST /admin/highlights/:highlightId/enable`
- `POST /admin/highlights/:highlightId/disable`
- `GET /admin/interactions`
- `GET /admin/branch-tasks`
- `POST /admin/branch-tasks/:taskId/retry`
- `POST /admin/demo/reset`
- `POST /admin/assets/config`

权限约定：

- 后台接口要求 `Authorization: Bearer <ADMIN_TOKEN>`
- 本地开发阶段同时限制来源为 `localhost` 或局域网

## 10.6 接口与资源映射规则

- 数据库存 `cover_path`，客户端接口返回 `cover_url`
- 数据库存 `video_path`，客户端接口返回 `video_url`
- 数据库存 `result_content_path`，客户端接口返回 `result_content_url`
- 管理后台接口同时返回 `path` 与 `url`

## 11. 核心技术流程

## 11.1 高光离线处理流程

```text
视频资源准备
  -> 音频抽取
  -> ASR 转写
  -> 文本结构化
  -> LLM 识别高光点
  -> 输出结构化高光 JSON
  -> 人工复核与修正
  -> 写入数据库
  -> 客户端播放时按集下发
```

## 11.2 播放态互动流程

```text
客户端加载剧集
  -> 获取高光列表
  -> 开始播放
  -> 监听播放进度
  -> 命中高光点
  -> 渲染主互动组件
  -> 弱展示历史热度
  -> 用户点击
  -> 播放动效
  -> 上报互动事件
  -> 服务端更新统计
  -> 客户端刷新局部热度展示
```

## 11.3 尾集固定分支流程

```text
播放最后一集结束
  -> 展示尾集分支入口
  -> 请求固定分支选项
  -> 用户选择分支
  -> 返回预生成短视频或结果内容
  -> 用户点赞 / 评论
```

## 11.4 尾集自定义分支流程

```text
播放最后一集结束
  -> 用户输入 Prompt
  -> 创建 BranchTask
  -> 服务端异步执行剧情扩写
  -> 输出结构化结果
  -> 写入任务结果
  -> 客户端轮询任务状态
  -> 生成完成后进入结果页
  -> 用户点赞 / 评论
```

## 12. AI 处理链路设计

## 12.1 高光识别链路

输入：

- 剧集转写文本
- 时间戳信息
- 人物信息
- 文本上下文

输出：

- 高光点区间
- 高光类型
- 强度等级
- 推荐标题
- 推荐互动按钮
- 推荐模板
- 置信度

要求：

- 必须输出合法 JSON
- 输出中必须包含可映射到前端模板的字段
- 输出结果需可由后台人工修正

## 12.2 分支生成链路

输入：

- 原剧摘要
- 最后一集剧情摘要
- 固定方向或用户自定义 Prompt
- 风格和安全约束

输出：

- 标题
- hook
- 剧情正文
- 分镜结构
- 标签
- 推荐互动按钮

要求：

- 不直接依赖实时视频生成
- 结果必须可单独展示
- 输出结构可回写到数据库

## 12.3 任务执行策略

建议采用异步任务设计：

- 用户请求先入库
- 服务端后台消费任务
- 返回 `pending/running/success/failed/timeout`
- 客户端轮询或定时刷新结果

补充约定：

- 第一版采用服务端进程内单任务队列，默认并发度 `1`
- 任务执行超时后标记为 `timeout`
- 管理后台可对 `failed/timeout` 任务执行重试

## 12.4 高光导入与人工校准流程

```text
脚本输出高光 JSON
  -> 导入候选高光 status=candidate
  -> 后台人工校准
  -> 确认后的高光 status=confirmed
  -> 客户端播放时消费 confirmed 数据
```

## 13. 本地部署方案

## 13.1 本地部署形态

建议采用以下部署方式：

- Android 真机运行客户端
- 笔记本或个人电脑运行本地 API 服务端
- 同一台电脑运行本地 Web 管理后台
- 视频资源与数据库均存于本地目录

## 13.2 启动方式建议

建议提供最少三条启动命令：

- 启动服务端
- 启动管理后台
- 初始化数据库与演示数据

例如：

```bash
pnpm install
pnpm db:seed
pnpm dev:server
pnpm dev:admin
```

## 13.3 局域网访问建议

为方便 Android 真机联调，建议：

- 服务端监听 `0.0.0.0`
- 管理后台也可监听局域网地址
- 使用固定局域网 IP 或 mDNS
- 在 Android 配置中可切换本地 API Base URL

## 13.4 CORS 与静态资源策略

- 服务端需开放 Android 客户端和本地 Web 后台访问
- 视频、封面、固定分支视频通过静态资源路由暴露
- 客户端统一通过 HTTP 访问服务端提供的资源地址，不直接读取电脑文件路径

## 14. 性能与稳定性设计

## 14.1 客户端

- 视频播放优先级高于特效
- 同屏只允许一个主互动组件
- 高光过近时第二个点弱化展示
- 弹幕、贴纸和粒子效果需要设置总量上限
- 非关键统计更新不阻塞 UI

## 14.2 服务端

- 高光列表查询要走索引
- 互动写入与统计聚合应分离
- 分支任务执行不得阻塞主 API 线程
- 管理后台查询接口允许分页和筛选

## 14.3 AI 链路

- 模型输出解析失败时自动重试一次
- 高光识别采用离线批处理，避免播放时调用模型
- 自定义分支采用异步任务，避免请求超时

## 15. 异常与降级方案

## 15.1 客户端降级

- 高光接口失败：继续播放视频，隐藏互动层或读取缓存
- 互动上报失败：本地先展示效果，后台异步补写
- 分支任务查询失败：保留“稍后刷新”入口

## 15.2 服务端降级

- 模型调用失败：分支任务标记失败并保留固定分支能力
- 高光提取失败：允许手动录入高光
- 数据库异常：优先保障只读播放链路

## 16. 安全与配置管理

建议：

- 所有模型密钥写入环境变量
- 仓库提交 `.env.example`
- `.env` 加入 `.gitignore`
- 视频资源不直接入仓库
- 管理后台接口至少具备本地访问控制或简单鉴权

补充：

- `ADMIN_TOKEN` 作为后台最小鉴权凭据
- `device_id` 必填，用于匿名用户稳定识别
- 评论和自定义 Prompt 需经过敏感词过滤
- AI 输出需经过结构校验和基础安全检查后落库
