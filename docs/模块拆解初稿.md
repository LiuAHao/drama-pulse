# Drama Pulse 模块拆解初稿

## 1. 文档目的

本文档用于将 Drama Pulse 项目拆解为可独立推进的功能模块，方便后续按模块逐步开发、联调和验收。

本拆解遵循以下原则：

- 每个模块边界尽量清晰
- 每个模块尽量具备明确输入输出
- 每个模块都能定义“完成标准”
- 模块之间保持可组合、可串联，而不是互相缠绕

## 2. 模块总览

当前建议将项目拆为 8 个模块：

1. 内容资源模块
2. 数据与配置模块
3. 本地服务端模块
4. Android 播放与基础体验模块
5. 高光互动模块
6. 历史互动沉淀模块
7. 尾集分支模块
8. AI 与管理后台模块

其中：

- `1-3` 是基础支撑层
- `4-7` 是用户可见主链路
- `8` 是智能与运营支撑层

## 3. 模块拆解

## 3.1 内容资源模块

### 模块目标

负责项目中所有短剧视频、封面、分支结果素材的整理与统一管理，为客户端、服务端和后台提供一致的内容基础。

### 主要功能点

- 整理主打剧和备选剧的视频资源
- 统一剧名和集数命名规则
- 准备剧封面、简介、标签等基础素材
- 准备固定分支结果素材
- 建立本地资源目录规范

### 输入

- 原始短剧视频资源
- 封面图与剧集信息
- 固定分支素材

### 输出

- 统一命名的视频目录
- 封面资源目录
- 分支结果资源目录
- 可供服务端读取的资源路径

### 依赖关系

- 是所有后续模块的基础
- 直接服务于服务端模块和管理后台模块

### 完成标准

- 主打剧和备选剧都能在本地目录中按统一规则找到
- 至少一部剧具备完整集数资源
- 固定分支结果素材已准备好并可被读取

## 3.2 数据与配置模块

### 模块目标

定义系统中的核心数据对象、数据库结构和配置规则，保证前后端、脚本和后台对同一数据有统一理解。

### 主要功能点

- 定义短剧数据结构
- 定义剧集数据结构
- 定义高光标签数据结构
- 定义互动事件数据结构
- 定义分支任务、点赞、评论数据结构
- 定义继续观看进度数据结构
- 定义数据库 `path` 与接口 `url` 的映射规则
- 定义状态枚举、类别映射规则和热度枚举
- 定义本地配置文件结构
- 定义环境变量示例

### 输入

- 产品方案中的核心对象定义
- 技术方案中的数据模型建议

### 输出

- 数据库 schema
- 接口字段定义
- 配置示例
- 字段关联关系说明
- 枚举与约束说明

### 依赖关系

- 是服务端模块的直接前置
- Android 客户端和管理后台也依赖其字段定义

### 完成标准

- 核心数据对象全部定义清楚
- 字段命名统一
- 能支撑服务端建表和接口开发

## 3.3 本地服务端模块

### 模块目标

作为整个项目的中枢，负责承接内容、互动、分支和后台相关的数据读写，向 Android 客户端和本地 Web 后台提供 API。

### 主要功能点

- 提供短剧和剧集查询接口
- 提供高光点下发接口
- 提供互动上报接口
- 提供高光统计接口
- 提供固定分支接口
- 提供自定义分支任务接口
- 提供点赞评论接口
- 提供后台管理接口
- 提供继续观看进度接口
- 提供静态资源 `path -> url` 映射能力

### 输入

- 资源路径与剧集信息
- 高光标签数据
- 用户互动事件
- 分支任务请求

### 输出

- 面向客户端的业务 API
- 面向后台的管理 API
- 本地结构化数据存储结果

### 依赖关系

- 依赖内容资源模块和数据与配置模块
- 被 Android 客户端和管理后台依赖

### 完成标准

- 客户端能通过服务端获取剧集、高光和分支信息
- 互动、点赞、评论、任务都能写入本地存储
- 后台能通过 API 完成基本管理动作

## 3.4 Android 播放与基础体验模块

### 模块目标

实现用户观看短剧的基础能力，保证短剧列表、播放页、选集和视频播放稳定可用。

### 主要功能点

- 短剧列表页
- 播放页
- 视频播放器接入
- 播放、暂停、进度条
- 播放页内选集
- 最后一集播放完成检测
- 本地缓存基础能力

### 输入

- 服务端返回的短剧列表和剧集信息
- 本地或服务端视频资源路径

### 输出

- 可稳定观看的 Android 主流程
- 可供高光互动模块监听的播放进度

### 依赖关系

- 依赖内容资源模块和本地服务端模块
- 是高光互动模块与尾集分支模块的承载基础

### 完成标准

- 用户能进入主打剧并播放任意一集
- 能在播放页内切换集数
- 能检测到最后一集播放结束事件

## 3.5 高光互动模块

### 模块目标

实现项目最核心的即时互动能力，让用户在剧情高光点出现时能快速表达情绪，并获得即时反馈。

### 主要功能点

- 高光点按时间轴露出与交互窗口触发
- 一主一辅互动机制
- 高光组件展示
- 高光过近时弱化展示
- 类别驱动的互动表现与选项态/组件态切换
- 点击后动效反馈
- 互动事件上报

### 输入

- 当前剧集的高光点列表
- 播放器实时进度
- 当前高光对应的类别与强度配置

### 输出

- 互动组件展示
- 动效反馈
- 互动上报事件

### 依赖关系

- 依赖 Android 播放与基础体验模块
- 依赖服务端下发高光点数据
- 为历史互动沉淀模块提供基础事件来源

### 完成标准

- 指定高光点能在 `interactionAppearMs` 露出，并在交互窗口内可点击
- 点击互动后有清晰视觉反馈
- 服务端能收到互动上报

## 3.6 历史互动沉淀模块

### 模块目标

将过往用户互动数据沉淀为历史热度、弹幕和群体反馈，增强陪伴感和高光情绪感染力。

### 主要功能点

- 高光互动统计聚合
- 历史热度等级计算
- 历史互动弹幕生成
- 高光点历史互动回流展示
- 去重和基础反垃圾策略

### 输入

- 历史互动事件
- 高光点信息
- 演示数据或真实用户数据

### 输出

- 历史弹幕
- 热度等级
- 高光统计结果

### 依赖关系

- 依赖本地服务端模块
- 依赖高光互动模块产生的事件
- 在 Android 播放页中展示

### 完成标准

- 用户播放到高光点时可看到历史互动痕迹
- 服务端能够聚合统计数据
- 客户端能按热度等级做差异化展示

## 3.7 尾集分支模块

### 模块目标

实现最后一集结束后的剧情延展能力，包括固定分支和自定义分支两类体验。

### 主要功能点

- 最后一集结束后展示分支入口
- 展示 2 个固定分支方向
- 播放固定分支结果
- 输入自定义 Prompt
- 创建异步分支任务
- 展示生成中状态与结果页
- 分支结果点赞与评论

### 输入

- 最后一集播放结束事件
- 固定分支配置
- 用户输入 Prompt

### 输出

- 固定分支结果页
- 自定义分支任务与结果
- 点赞评论数据

### 依赖关系

- 依赖 Android 播放模块检测最后一集结束
- 依赖服务端模块提供分支接口
- 依赖 AI 与管理后台模块支持自定义结果生成

### 完成标准

- 最后一集结束后会展示分支入口
- 2 个固定分支能正常展示结果
- 自定义分支任务能创建并返回结果
- 分支结果页能点赞和评论

## 3.8 AI 与管理后台模块

### 模块目标

负责高光识别、分支生成、后台配置和演示数据管理，是系统的智能与运营支撑层。

该模块在开发上需要拆成两个子方向：

- `后台子模块`：优先落地，服务资源配置、标签修正、任务查看和演示重置
- `离线 AI 子模块`：后置落地，服务高光识别、结构化导入和自定义分支生成

### 主要功能点

- ASR 转文本
- 文本结构化
- 高光识别与标签生成
- 高光标签导入与修正
- 分支剧情扩写
- 分支任务编排
- 本地 Web 管理后台
- 资源配置、标签管理、任务管理、演示数据初始化
- 管理后台鉴权与本地访问控制

### 输入

- 视频音频内容
- 剧情文本
- 用户自定义 Prompt
- 后台管理操作

### 输出

- 结构化高光标签
- 结构化分支结果
- 可视化后台管理能力

### 依赖关系

- 依赖内容资源模块和服务端模块
- 服务于高光互动模块和尾集分支模块

### 完成标准

- 至少一部剧能通过离线链路产出高光标签
- 后台能查看和修正高光标签
- 后台能查看和管理分支任务
- 能初始化演示数据并支持演示复位
- 能配置资源路径并验证客户端可访问地址

## 4. 模块分组建议

为了便于理解和开发推进，可以将 8 个模块再归成 3 组：

### A. 基础支撑层

- 内容资源模块
- 数据与配置模块
- 本地服务端模块

### B. 用户可见主链路

- Android 播放与基础体验模块
- 高光互动模块
- 历史互动沉淀模块
- 尾集分支模块

### C. 智能与运营支撑层

- AI 与管理后台模块

## 5. 推荐开发顺序

建议后续按以下顺序逐模块推进：

1. 内容资源模块
2. 数据与配置模块
3. 本地服务端模块
4. AI 与管理后台模块中的后台子模块
5. Android 播放与基础体验模块
6. 高光互动模块
7. 历史互动沉淀模块
8. 尾集分支模块
9. AI 与管理后台模块中的离线 AI 子模块

这个顺序的好处是：

- 先把基础支撑层打稳
- 再先落地后台管理能力
- 再打通用户主链路
- 最后补离线 AI 能力

## 6. 当前建议

如果后续要按模块推进开发，建议优先从以下 3 个模块入手：

- 内容资源模块
- 数据与配置模块
- 本地服务端模块

原因是这三块决定了后面 Android、后台和 AI 脚本能否共享统一的数据基础。
