# Drama Pulse 高光人工复核后台方案

## 1. 文档目的

本文档用于冻结 Drama Pulse 项目中“高光候选人工复核后台”的页面结构、后端接口、数据字段、状态流转和实现顺序。

目标不是做一个泛化内容运营系统，而是做一个：

- 能查看 AI 候选高光
- 能播放对应片段
- 能精确修正时间点
- 能调整类型、模板、互动选项
- 能完成 `candidate -> confirmed / disabled`

的可视化审核工作台。

本文档优先服务：

- 高光人工冻结
- 比赛演示稳定性
- 后续 Agent 并行开发时的低歧义实现

---

## 2. 当前结论

当前项目已经具备：

- 高光候选离线识别链路
- 管理后台基础页面
- `GET /admin/highlights`
- `PATCH /admin/highlights/:highlightId`
- `POST /admin/highlights/:highlightId/enable`
- `POST /admin/highlights/:highlightId/disable`

但**还不具备完整人工复核体验**，目前缺失的关键能力包括：

- 候选高光详情页 / 复核工作台
- 视频预览与时间轴定位
- transcript 上下文展示
- 时间微调快捷操作
- 审核动作的可视化闭环
- review 相关字段的完整保存

所以，当前后台的“高光管理页”还只是管理列表，不足以承担正式的人工冻结流程。

---

## 3. 总体目标

本模块最终目标定义为：

`AI 候选高光 -> 人工可视化复核 -> 修正并冻结 -> confirmed 高光下发客户端`

人工复核时，审核者应能够：

1. 快速看到候选列表
2. 打开单条候选进入复核页
3. 自动播放候选时间点附近的片段
4. 查看 transcript 片段上下文
5. 微调高光起止时间
6. 修改类型、模板、文案、互动选项
7. 执行 `确认` 或 `禁用`

---

## 4. 第一版边界

### 4.1 要做的

- 候选高光复核工作台
- 视频预览
- transcript 上下文展示
- 时间点修正
- 类型与模板修正
- 互动选项修正
- `confirm / disable`

### 4.2 不做的

- 多人同时协作审核
- 复杂审批流
- 历史版本 diff 可视化
- 自动合并重复候选
- 复杂角色图谱

### 4.3 允许保留的轻量能力

- `speakerGuess / targetCharacterGuess` 可以先展示为只读或可编辑文本
- transcript 可先做只读高亮，不强求字幕编辑器
- 播放器可以先用浏览器原生 `<video>` 承担，不强求重型自定义播放器

---

## 5. 页面结构方案

第一版建议采用**独立复核页**，而不是只在列表页弹一个小弹窗。

推荐路由：

- `/highlights`
- `/highlights/:highlightId/review`

其中：

- `/highlights` 负责筛选和列表
- `/highlights/:highlightId/review` 负责单条复核

原因：

- 候选高光的审核信息密度高
- 需要视频、表单、上下文同时存在
- 弹窗容易拥挤，影响精细操作

---

## 6. 复核页布局

推荐三栏布局：

```text
┌──────────────────────────────────────────────────────────────┐
│ 顶部工具栏：返回 / 剧名集数 / 当前状态 / 上一条 / 下一条      │
├───────────────┬──────────────────────────┬───────────────────┤
│ 左侧候选列表   │ 中间视频与时间轴区        │ 右侧审核表单区     │
│               │                          │                   │
│ - 候选标题     │ - 视频播放器              │ - 类型            │
│ - 类型         │ - 当前候选时间高亮        │ - 标题            │
│ - 置信度       │ - 自动跳转到候选前 2-3 秒 │ - 描述            │
│ - 状态         │ - transcript 片段         │ - 模板            │
│               │                          │ - 互动选项        │
│               │                          │ - 强度/置信度      │
│               │                          │ - 时间修正         │
│               │                          │ - 审核动作         │
└───────────────┴──────────────────────────┴───────────────────┘
```

### 6.1 左侧候选列表

显示：

- 候选标题
- 类型
- 状态
- `start-end`
- 置信度

能力：

- 点击切换到该候选
- 当前候选高亮
- 支持上一条 / 下一条导航

### 6.2 中间视频区

必须支持：

- 播放/暂停
- 拖动进度条
- 自动跳转到 `candidate.startTimeMs - 3000ms`
- 在时间轴上高亮候选区间
- 可选高亮 transcript 对应片段

推荐规则：

- 初次进入默认从候选开始前 `2-3 秒` 开始
- 候选结束后继续播放 `1-2 秒`
- 让审核者看到“进入点 + 峰值 + 收尾”

### 6.3 右侧审核表单区

必须可编辑字段：

- `startTimeMs`
- `endTimeMs`
- `type`
- `title`
- `description`
- `templateId`
- `interactionOptions`
- `intensity`
- `confidence`

建议展示但可选编辑字段：

- `speakerGuess`
- `targetCharacterGuess`
- `mentionedCharacters`
- `characterGuessConfidence`
- `reason`
- `supportingSegmentIds`

## 7. 审核动作定义

第一版只保留以下动作：

- `保存修改`
- `确认高光`
- `禁用候选`

### 7.1 保存修改

含义：

- 只更新字段
- 不改变状态

适用于：

- 候选先改好，稍后再决定是否确认

### 7.2 确认高光

含义：

- 当前候选进入 `confirmed`
- 推荐同时把 `source` 更新为 `ai_manual`

适用于：

- 时间点和文案已确认

### 7.3 禁用候选

含义：

- 状态改为 `disabled`

适用于：

- 非高光
- 重复候选
- 粒度不合适且不值得保留

## 8. 时间修正交互细节

这部分最容易出错，必须定死。

### 8.1 表单输入

提供两个输入方式：

- 直接输入毫秒值
- 以 `mm:ss.mmm` 形式显示和编辑

建议前端内部统一使用毫秒存储。

### 8.2 快捷按钮

推荐在时间输入旁边提供：

- `开始 -1000ms`
- `开始 -500ms`
- `开始 +500ms`
- `开始 +1000ms`
- `结束 -1000ms`
- `结束 -500ms`
- `结束 +500ms`
- `结束 +1000ms`

作用：

- 让审核者快速对齐爆点

### 8.3 时间合法性校验

必须校验：

- `startTimeMs >= 0`
- `endTimeMs > startTimeMs`
- `endTimeMs - startTimeMs <= 30000`

第一版建议：

- 普通高光推荐 `3-12 秒`
- 超过 `20 秒` 弹出警告
- 超过 `30 秒` 禁止提交

### 8.4 一键同步到当前播放器位置

推荐补两个按钮：

- `用当前播放位置覆盖开始时间`
- `用当前播放位置覆盖结束时间`

这样审核者边看边改会很快。

## 9. Transcript 上下文展示

复核页必须提供候选对应 transcript 上下文。

建议显示范围：

- 当前候选命中的 segment
- 前 2-3 条
- 后 2-3 条

每条显示：

- `segmentId`
- `start-end`
- `text`

高亮规则：

- 命中的 `supportingSegmentIds` 高亮
- 落在候选时间区间内的片段次级高亮

### 9.1 transcript 数据来源

推荐第一版使用**标准化 transcript JSON 文件**作为上下文来源。

建议固定路径规范：

```text
data/exports/highlight-demo/{episodeId}-transcript.json
```

更正式的推荐规范：

```text
data/exports/highlights/{dramaId}/{episodeId}.transcript.json
```

后端读取策略：

- 根据 `episodeId` 或 `dramaId + episodeId` 解析 transcript 文件路径
- 读取 JSON 后返回前端需要的上下文片段

如果当前没有 transcript 文件：

- 页面允许只显示视频和基础字段
- transcript 区显示“暂无 transcript 上下文”

## 10. 后端接口设计

当前已有：

- `GET /admin/highlights`
- `PATCH /admin/highlights/:highlightId`
- `POST /admin/highlights/:highlightId/enable`
- `POST /admin/highlights/:highlightId/disable`

第一版建议新增 2 个接口。

### 10.0 本轮后端落地文件

为了避免实现时口径漂移，后端本轮默认修改这些文件：

- `server/prisma/schema.prisma`
- `server/src/shared/schemas/index.ts`
- `server/src/shared/response/index.ts`（若需要补通用响应类型）
- `server/src/modules/admin/routes.ts`
- `server/tests/integration/admin.test.ts`

如果后端已有更合适的拆分方式，也应保证功能最终仍然落在：

- Prisma schema
- admin route
- zod request/response schema
- integration tests

这四层上。

### 10.1 GET /admin/highlights/:highlightId

用途：

- 获取单条高光完整详情

返回至少包含：

- highlight 本身字段
- episode 基础信息
- drama 基础信息

建议响应：

```json
{
  "id": "hl_xxx",
  "episodeId": "episode_001",
  "type": "conflict",
  "title": "...",
  "description": "...",
  "startTimeMs": 77700,
  "endTimeMs": 85740,
  "templateId": "vote_side",
  "interactionOptionsJson": "[\"支持她\",\"开怼\"]",
  "confidence": 0.95,
  "status": "candidate",
  "episode": {
    "id": "episode_001",
    "episodeNo": 1,
    "title": "第1集",
    "videoPath": "...",
    "videoUrl": "http://..."
  },
  "drama": {
    "id": "drama_001",
    "title": "荒年全村啃树皮，我有系统满仓肉"
  }
}
```

实现要求：

- 查询必须 `include episode`
- `episode` 必须继续 `include drama`
- `episode.videoPath` 需经 `pathToUrl()` 转成 `videoUrl`
- 若高光不存在，返回 `404`
- 不要在前端自行拼接 video url

### 10.2 GET /admin/highlights/:highlightId/review-context

用途：

- 获取复核页所需上下文

返回建议包含：

- 当前高光
- 关联 episode / drama
- transcript 片段上下文
- 相邻候选列表（可选）

建议响应：

```json
{
  "highlight": { "...": "..." },
  "episode": { "...": "..." },
  "drama": { "...": "..." },
  "transcriptContext": [
    {
      "segmentId": "seg_0029",
      "startTimeMs": 77700,
      "endTimeMs": 81120,
      "text": "之前我丈夫战死的22体恤银全都给你们了"
    }
  ]
}
```

实现要求：

- transcriptContext 按当前候选所在区间前后各取 `2-3` 条 segment
- 若 transcript 文件不存在：
  - `transcriptContext: []`
  - 额外返回 `transcriptAvailable: false`
- 若 transcript 文件存在：
  - 返回 `transcriptAvailable: true`

推荐补充响应字段：

```json
{
  "transcriptAvailable": true,
  "candidateNeighbors": [
    {
      "id": "hl_xxx",
      "title": "...",
      "startTimeMs": 77700,
      "endTimeMs": 85740,
      "type": "conflict",
      "status": "candidate"
    }
  ]
}
```

其中：

- `candidateNeighbors` 用于左侧列表的上一条 / 下一条切换
- 第一版只需要当前集内候选，不需要跨集导航

### 10.3 PATCH /admin/highlights/:highlightId

建议增强：

- 支持 `source` 更新
- 支持审核相关字段

推荐新增可选字段：

- `reason`
- `supportingSegmentIdsJson`
- `speakerGuess`
- `targetCharacterGuess`
- `mentionedCharactersJson`
- `characterGuessConfidence`

推荐请求体 schema：

```json
{
  "startTimeMs": 77700,
  "endTimeMs": 85740,
  "type": "conflict",
  "title": "娘家吞恤银翻脸赶人",
  "description": "女主求粮被拒并被赶回去，适合站队互动",
  "templateId": "vote_side",
  "interactionOptionsJson": "[\"支持她\",\"怼回去\"]",
  "intensity": 4,
  "confidence": 0.93,
  "reason": "求助被拒绝，情绪峰值明确",
  "supportingSegmentIdsJson": "[\"seg_0029\",\"seg_0030\",\"seg_0031\"]",
  "speakerGuess": "女主",
  "targetCharacterGuess": "娘家人",
  "mentionedCharactersJson": "[\"女主\",\"娘家\"]",
  "characterGuessConfidence": 0.72
}
```

约束要求：

- `templateId` 必须限制在：
  - `emotion_button`
  - `vote_side`
  - `suspense_lock`
- `type` 必须限制在：
  - `feel_good`
  - `reversal`
  - `conflict`
  - `sweet`
  - `suspense`
- `interactionOptionsJson` 必须是合法 JSON 字符串数组
- `supportingSegmentIdsJson` 必须是合法 JSON 字符串数组
- `mentionedCharactersJson` 必须是合法 JSON 字符串数组
- `characterGuessConfidence` 取值范围建议 `0-1`

### 10.4 POST /admin/highlights/:highlightId/confirm

建议新增专门确认接口，而不是继续复用 `/enable`。

原因：

- `enable` 语义偏通用
- `confirm` 更符合人工冻结流程
- 确认时可顺便更新 `source=ai_manual`

建议逻辑：

- `status -> confirmed`
- 若当前 `source == ai`，则自动改为 `ai_manual`

推荐响应：

```json
{
  "id": "hl_xxx",
  "status": "confirmed",
  "source": "ai_manual"
}
```

补充说明：

- 第一版 `confirm` 不需要额外 body
- 第一版保留原有 `/enable` 兼容旧页面，但新复核页只调用 `/confirm`

## 11. 数据字段建议

为了支撑高光复核，建议在 `Highlight` 上新增或预留这些字段：

- `reason`
- `supportingSegmentIdsJson`
- `speakerGuess`
- `targetCharacterGuess`
- `mentionedCharactersJson`
- `characterGuessConfidence`

如果暂时不想改主表结构，也可以用兼容方案：

- 第一版先只把这些字段保存在候选 JSON 文件中
- 后台优先复核核心字段

但从长期看，**推荐入库**，不然后面做审核详情页会越来越别扭。

### 11.1 Prisma 建议结构

建议在 `Highlight` 上新增以下字段：

```prisma
reason                   String  @default("")
supportingSegmentIdsJson String  @default("[]") @map("supporting_segment_ids_json")
speakerGuess             String  @default("")   @map("speaker_guess")
targetCharacterGuess     String  @default("")   @map("target_character_guess")
mentionedCharactersJson  String  @default("[]") @map("mentioned_characters_json")
characterGuessConfidence Float?  @map("character_guess_confidence")
```

说明：

- 保持和当前已有 `interactionOptionsJson` 一样的 JSON-string 存储风格，减少本轮 schema 改动成本
- `Float?` 可空，兼容当前 transcript 中大量 `null`

### 11.2 当前真实 Highlight 表结构差异

当前 `server/prisma/schema.prisma` 中的 `Highlight` 还没有这些审核增强字段，因此这部分不能只写前端页面，必须先补 migration。

当前已有字段：

- `startTimeMs`
- `endTimeMs`
- `type`
- `title`
- `description`
- `intensity`
- `templateId`
- `interactionOptionsJson`
- `visualEffectType`
- `source`
- `confidence`
- `status`

所以本轮是**在现有基础上增量扩展**，不是重做主表。

### 11.3 前端共享类型需要同步补齐

`admin-web/src/shared/types/index.ts` 中的 `Highlight` 也需要同步新增：

- `reason: string`
- `supportingSegmentIdsJson: string`
- `speakerGuess: string`
- `targetCharacterGuess: string`
- `mentionedCharactersJson: string`
- `characterGuessConfidence: number | null`

- `admin-web/src/pages/highlights/components/TimeAdjustControls.tsx`（新增）
- `admin-web/src/pages/highlights/components/ReviewActionBar.tsx`（新增）

### 12.2 页面组件拆分建议

推荐组件：

- `HighlightReviewPage`
- `CandidateSidebar`
- `HighlightVideoPanel`
- `HighlightTranscriptPanel`
- `HighlightReviewForm`
- `TimeAdjustControls`
- `ReviewActionBar`

### 12.2.1 列表页跳转规则

现有 `HighlightsPage` 中，操作列除了 `编辑` 外，应新增：

- `复核`

点击后跳转：

- `/highlights/:highlightId/review`

第一版保留旧 `编辑` 弹窗，便于兼容；但后续正式流程应逐步转向 `复核页`。

### 12.3 视频实现建议

第一版使用浏览器原生 `<video>` 即可。

原因：

- 本地后台复核不需要复杂播放器特效
- 只需要：
  - seek
  - play/pause
  - currentTime
  - timeupdate

### 12.4 视频初始化规则

进入复核页时：

- 视频自动 seek 到 `max(0, startTimeMs - 3000)`
- 默认不自动播放，避免审核者一进来被打断
- 提供 `播放候选片段` 按钮：
  - 点击后从预设位置开始播放
  - 播到 `endTimeMs + 1500` 可自动暂停

### 12.5 transcript 联动

播放时可选做：

- 当前播放时间所在 segment 高亮
- 当前候选 `supportingSegmentIds` 固定高亮

第一版不是必须，但强烈建议做。

### 12.6 表单策略

使用本地表单状态，不要每次输入都直接写接口。

提交时机：

- 点“保存修改”
- 点“确认高光”
- 点“禁用候选”

推荐页面本地 state 结构：

```ts
type ReviewFormState = {
  startTimeMs: number;
  endTimeMs: number;
  type: "feel_good" | "reversal" | "conflict" | "sweet" | "suspense";
  title: string;
  description: string;
  templateId: "emotion_button" | "vote_side" | "suspense_lock";
  interactionOptions: string[];
  intensity: number;
  confidence: number;
  reason: string;
  supportingSegmentIds: string[];
  speakerGuess: string;
  targetCharacterGuess: string;
  mentionedCharacters: string[];
  characterGuessConfidence: number | null;
};
```

前端落地要求：

- 页面读接口时把 `interactionOptionsJson / supportingSegmentIdsJson / mentionedCharactersJson` 解成数组
- 提交接口时再序列化成 JSON string
- 表单提交前做一次本地校验，避免无意义请求

### 12.6.1 React Query 约定

建议新增 query key：

```ts
highlightDetail: (id: string) => ["admin", "highlight", id]
highlightReviewContext: (id: string) => ["admin", "highlight-review", id]
```

mutation 成功后至少失效：

- `["admin", "highlights"]`
- `["admin", "highlight", highlightId]`
- `["admin", "highlight-review", highlightId]`

### 12.7 离开保护

若表单有未保存修改：

- 跳转页面前弹确认
- 防止审核者误丢修改

## 13. 现有高光管理页需要修正的点

当前已有高光管理页有一些地方与冻结方案不完全一致，后续要一并修正：

- `TYPE_OPTIONS` 里还保留了 `funny`、`emotion_burst`
- `TEMPLATE_OPTIONS` 里还保留了 `ending_branch`

而当前高光冻结方案第一版应收敛为：

- 类型：
  - `feel_good`
  - `reversal`
  - `conflict`
  - `sweet`
  - `suspense`

- 模板：
  - `emotion_button`
  - `vote_side`
  - `suspense_lock`

因此后台复核页和列表页都应统一到这一口径。

此外还需要修正：

- 列表页当前只有 `编辑 / 启用 / 禁用`，没有 `复核` 入口
- 当前列表页只展示基础字段，没有展示 `title / confidence / source`
- 当前 `PATCH /admin/highlights/:highlightId` 的表单字段也还未覆盖 review 增强字段

---

## 14. 状态流转

复核状态流转建议固定为：

```text
candidate
  ├─ 保存修改 -> candidate
  ├─ 确认高光 -> confirmed
  └─ 禁用候选 -> disabled
```

若后续要支持“重新启用”：

```text
disabled -> candidate
disabled -> confirmed
```

但第一版不建议过度复杂化。

## 15. 异常与兜底

必须考虑以下异常：

### 15.1 视频资源不存在

处理：

- 页面展示“视频资源不可用”
- 仍允许编辑文本字段和时间字段

### 15.2 transcript 不存在

处理：

- transcript 面板显示空态
- 不阻塞审核

### 15.3 高光不存在

处理：

- 404 页面或返回列表

### 15.4 保存失败

处理：

- toast 提示
- 保留本地未保存表单

### 15.5 状态变更冲突

处理：

- 若高光已被他处修改，可提示刷新重试

## 16. 验收标准

这一模块完成后，至少要满足：

1. 能从候选列表进入单条复核页
2. 能看到本集视频并定位到候选前后时间
3. 能播放回看候选片段
4. 能看到 transcript 上下文
5. 能编辑起止时间、类型、标题、模板、互动选项
6. 能保存修改
7. 能确认候选为 `confirmed`
8. 能禁用候选为 `disabled`
9. 返回列表后能看到状态已更新

## 17. 实现顺序建议

推荐实现顺序：

1. 后端补 `GET /admin/highlights/:highlightId`
2. 后端补 `GET /admin/highlights/:highlightId/review-context`
3. 后台新增复核页路由
4. 接视频预览
5. 接 transcript 上下文
6. 接表单编辑
7. 接确认/禁用动作
8. 最后再做快捷时间微调和键盘增强

### 17.1 更细的开发拆解

后端：

1. Prisma schema 增字段并生成 migration
2. 扩展 zod schema
3. 新增 `GET /admin/highlights/:highlightId`
4. 新增 `GET /admin/highlights/:highlightId/review-context`
5. 新增 `POST /admin/highlights/:highlightId/confirm`
6. 补 integration tests

前端：

1. 扩展 `Highlight` 类型
2. 路由新增 review 页
3. 列表页增加 `复核` 按钮
4. review 页先接 detail/context
5. 接 `<video>` 与 transcript
6. 接表单与时间修正
7. 接 confirm/disable/save
8. 最后做未保存离开保护

### 17.2 测试最小集合

后端至少覆盖：

- 获取单条详情成功
- transcript 存在时 review-context 返回上下文
- transcript 缺失时 review-context 返回空数组且不报错
- confirm 后 `status=confirmed`
- confirm 后 `source=ai_manual`（当原始 source=ai）

前端至少验证：

- 列表可跳 review 页
- review 页加载 detail/context 成功
- 点击“播放候选片段”可 seek 到预设时间
- 保存后列表状态更新
- confirm 后状态显示 confirmed
- disable 后状态显示 disabled

## 18. 当前冻结结论

高光人工复核后台第一版最终冻结为：

- 采用独立复核页，而不是小弹窗
- 采用三栏结构：候选列表 / 视频与 transcript / 审核表单
- 后端新增 review context 接口
- 前端使用原生 `<video>` 完成片段回看
- 审核动作只保留：保存、确认、禁用
- 时间修正必须有快捷微调能力
- transcript 上下文是强推荐能力，不可长期缺席

这版方案已经足够支撑后续直接开发，不建议在实现前继续发散后台形态。
