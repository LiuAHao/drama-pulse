# Drama Pulse 技术选型与落地说明

## 1. 文档目的

本文档用于回答两个问题：

1. 我们整个项目为什么选择当前这套技术方案
2. 这套方案如何真正落地，而不是停留在概念初稿

本文档定位不是泛泛而谈的“技术可选项列表”，而是面向当前比赛项目的第一版可执行技术基线，用于：

- 技术方案记录
- 开发前口径确认
- 后续实施依据
- 答辩时解释“为什么这么选”

## 2. 项目技术目标

Drama Pulse 不是纯前端 Demo，也不是纯 AI 实验，而是一个包含以下部分的完整项目：

- Android 客户端
- 本地服务端
- 本地 Web 管理后台
- 离线 AI 处理链路
- 本地视频与结构化数据管理

因此，技术选型必须同时满足以下目标：

- 单人可控
- 本地部署稳定
- 联调成本低
- 对 AI 能力友好
- 前后端结构清晰
- 答辩时容易讲明白

## 3. 技术选型原则

我们当前的技术选型遵循以下原则：

### 3.1 稳定优先于炫技

本项目是比赛项目，不是线上生产系统。第一目标是完整闭环和稳定演示，而不是追求最复杂或最“高级”的技术。

### 3.2 效率优先于绝对性能

当前业务规模很小：

- 短剧数量有限
- 剧集数量有限
- 互动数据规模有限
- AI 任务量有限

因此，开发效率、联调效率、排查效率比极致性能更重要。

### 3.3 主链路轻，AI 链路异步

播放过程中的高光互动必须轻量，不能依赖实时 AI 推理。重型任务如高光识别、剧情扩写、分支生成，必须尽量离线化或异步化。

### 3.4 单人维护友好

技术方案不能建立在“团队协作足够大、分工足够细”的前提上。所有核心层都需要单人能理解、能修改、能调试。

## 5. 为什么选 Android 客户端这套技术

### 5.1 为什么选 Android

因为比赛要求客户端三选一，而 Android 有以下优势：

- 工具链成熟
- 视频播放能力成熟
- 真机调试方便
- 本地服务联调成本低
- 答辩时演示可控

### 5.2 为什么选 Kotlin

原因：

- Android 当前主流语言
- 生态成熟
- 与 Jetpack 体系结合最好
- 比 Java 更适合快速开发和状态管理

### 5.3 为什么选 Jetpack Compose

原因：

- UI 开发效率高
- 状态驱动更适合互动播放器场景
- 做弹层、浮层、动画、贴纸效果更灵活
- 后期页面迭代快

不优先选 XML 的原因：

- 页面层级和交互层会比较复杂
- 高光互动、弹幕、分支入口、动效叠层更适合 Compose

### 5.4 为什么选 Media3 / ExoPlayer

原因：

- Android 视频播放事实标准
- 对进度监听和播放状态监听支持成熟
- 便于实现高光点时间轴触发
- 便于检测最后一集结束事件

### 5.5 为什么选 ViewModel + StateFlow

原因：

- 播放状态、高光状态、互动状态、分支状态都适合状态流管理
- 与 Compose 配合自然
- 比回调和散乱状态更稳

### 5.6 为什么选 Retrofit + OkHttp

原因：

- 简洁稳定
- 文档和生态成熟
- 联调成本低
- 适合本地 REST API 场景

### 5.7 为什么选 Room

原因：

- 可做本地缓存
- 可缓存剧集、高光、历史结果
- 在接口失败时可做基础降级

## 6. 为什么选本地服务端这套技术

### 6.1 为什么服务端要独立存在

因为本项目不是单纯离线播放器。我们需要：

- 下发高光点
- 写入互动事件
- 聚合历史互动
- 管理分支任务
- 提供后台管理能力
- 调用 AI 能力

这些都要求独立服务层存在。

### 6.2 为什么选 Node.js

原因：

- 本项目服务端主要是 I/O 型场景
- 包括数据库读写、文件读取、HTTP 接口、模型调用编排
- Node.js 在这类场景下足够轻、足够快、足够成熟
- 对单人项目非常友好

### 6.3 为什么选 TypeScript

原因：

- 本项目数据结构多，字段复杂
- 需要保证客户端、服务端、后台、AI 输出结构理解一致
- TypeScript 可以显著减少字段不统一、接口错位、JSON 解析混乱的问题

TypeScript 在这个项目里的价值不在于“高级”，而在于：

- 让结构化数据更稳
- 让联调更顺
- 让后期返工更少

### 6.4 为什么选 Fastify

原因：

- 比较轻
- 插件机制清晰
- 性能足够
- TypeScript 体验较好

为什么不优先选 Express：

- Express 也能做，而且资料更多
- 但 Fastify 在类型体验和结构约束上更适合当前这类新项目

结论是：

- `Fastify` 是当前冻结方案
- 当前阶段不建议在项目中途切换为 `Express`

### 6.5 为什么选 SQLite

原因：

- 当前部署方式是本地部署，不是云数据库
- SQLite 零运维
- 单文件数据库，适合备份和迁移
- 数据量完全足够承载当前项目

为什么不优先选 MySQL / PostgreSQL：

- 运维成本更高
- 本地部署更重
- 对当前比赛规模没有必要

### 6.6 为什么选 Prisma

原因：

- 表结构定义清晰
- migration 管理方便
- TypeScript 类型联动更自然
- 适合单人快速维护

为什么不直接手写 SQL：

- 后期表一多，维护成本上升
- 容易出现字段和类型不一致

### 6.7 为什么选 Zod

原因：

- 用于接口入参和 AI 输出结构校验
- 当前项目 JSON 数据很多，尤其 AI 输出必须做结构校验
- Zod 足够轻，也很适合配合 TypeScript 使用

## 7. 为什么选本地 Web 管理后台这套技术

### 7.1 为什么后台必须存在

后台不是附属品，而是项目闭环的一部分。因为我们需要：

- 配视频资源路径
- 管短剧和剧集信息
- 查看和修正高光标签
- 查看互动统计
- 管理分支任务
- 初始化演示数据

如果没有后台，这些工作会转成大量手工脚本和临时数据修改，效率和展示效果都会变差。

### 7.2 为什么选 React + TypeScript

原因：

- 当前前端生态最成熟
- 与 Node.js + TypeScript 服务端配合最好
- 组件化管理后台效率高
- 单人开发成本可控

### 7.3 为什么选 Vite

原因：

- 初始化快
- 启动快
- 本地开发体验好
- 对当前规模的后台项目足够

为什么不优先选 Next.js：

- 但本项目后台没有强 SSR 诉求
- Vite 更轻，更适合本地工具型后台

### 7.4 为什么选 React Router + TanStack Query

原因：

- React Router 适合后台多页面切换
- TanStack Query 适合后台查询列表、筛选、刷新、状态管理

### 7.5 为什么选 Tailwind CSS

原因：

- 开发快
- 适合后台工具型页面
- 能快速搭管理布局和表格操作区

## 8. 为什么 AI 与离线脚本层建议用 Python

这里要注意，我们不是说“整个后端都必须用 Python”，而是：

- 业务服务层优先用 `Node.js + TypeScript`
- 离线 AI 脚本层可以使用 `Python`

### 原因

- Python 在音视频处理、文本处理、离线脚本上生态成熟
- 后续如果要做 ASR、文本清洗、批处理导入，Python 更顺手
- 可以和主服务层解耦，不必把所有 AI 处理逻辑塞进 Node 服务里

### 最合理的分工

- `Node.js / TypeScript`：业务 API、任务管理、后台接口
- `Python`：离线高光识别脚本、批处理脚本、辅助数据导入

这是一种“主服务稳定 + AI 脚本灵活”的组合。

## 9. 为什么资源和数据这样组织

### 9.1 为什么视频不用直接进数据库

原因：

- 视频文件体积大
- 本地文件路径读取更直接
- 本地部署环境下，文件系统比把视频写入数据库更合理

### 9.2 为什么结构化数据放 SQLite

原因：

- 高光、互动、分支、评论都是典型结构化数据
- 查询、统计、过滤都更适合数据库

### 9.3 为什么目录要分开

建议继续保持：

- `videos/`：原始短剧视频
- `assets/dramas/covers/`：封面
- `assets/dramas/branches/`：旧的固定分支视频或兼容资源路径
- `assets/generated/fixed-branches/`：当前固定分支图文结果 sidecar 与 manifest
- `data/`：结构化数据

这样后续资源扫描、后台配置和服务端读取都清晰。

### 9.4 为什么数据库存 path，而接口返回 url

原因：

- 数据库存相对路径更稳定，不依赖当前机器的局域网 IP 和端口
- 客户端和后台真正访问资源时需要的是可访问地址，而不是本地文件路径
- 服务端统一做 `path -> url` 映射后，客户端和后台都能共享一致规则

## 10. 为什么这套方案最适合当前项目

归纳起来，这套方案适合当前项目的原因是：

### 10.1 它是闭环导向的

不是为了追求单点最强，而是为了确保：

- 客户端能跑
- 服务端能支撑
- 后台能管理
- AI 能接入
- 本地部署能稳定演示

### 10.2 它是单人友好的

当前项目不是大团队并行开发，所以：

- 学习成本
- 维护成本
- 调试成本

都要尽量低。

### 10.3 它足够工程化

虽然是比赛项目，但这套方案不是一次性 Demo，而是具备：

- 清晰模块边界
- 清晰数据边界
- 可扩展的服务结构
- 后续可迁移到云端的可能性

### 10.4 它适合答辩表达

这套技术方案在答辩时很好讲：

- Android 客户端负责体验
- 本地服务端负责闭环
- 本地后台负责管理
- AI 链路负责内容理解与分支生成

逻辑清楚，不混乱。

## 11. 当前真正可落地的冻结方案

如果我们要从“技术初稿”进入“可开发基线”，当前建议冻结如下：

### 客户端

- Kotlin
- Jetpack Compose
- Media3 / ExoPlayer
- ViewModel + StateFlow
- Retrofit + OkHttp
- Room
- Coil

### 服务端

- Node.js
- TypeScript
- Fastify
- Prisma
- SQLite
- Zod

### 管理后台

- React
- TypeScript
- Vite
- React Router
- TanStack Query
- Tailwind CSS

### AI 与脚本

- Python
- ASR
- LLM 调用

### 部署

- 本地单机部署
- Android 真机联调
- 服务端与后台运行在同一台电脑
- 管理后台通过本地 `ADMIN_TOKEN` 控制访问

### 数据与资源规则

- 数据库存 `path`
- 客户端接口返回 `url`
- `videos/` 存原始剧集视频
- `assets/dramas/branches/` 保留旧分支资源路径
- `assets/generated/fixed-branches/` 存当前固定分支图文结果
- `assets/dramas/covers/` 存封面
- 自定义分支第一版输出文本剧情与分镜结构，不依赖实时视频生成

这就是当前第一版真正可以开始开发的技术方案。

## 12. 当前仍然保留弹性的部分

虽然整体方案建议冻结，但以下点仍可保留小范围弹性：

- `Room` 的使用深度可以前期保持轻量，只承担基础缓存
- `Python` 脚本层可以延后实现，先手工导入高光
- AI 结果的视频化可以后置，第一版以文本剧情和分镜结构为主

这些调整不会影响系统主结构。

## 13. 当前结论

当前建议不是“讨论所有可能性”，而是确定一套真正适合当前项目的落地方案。

最终结论是：

- 整个项目应采用“Android + 本地 Node 服务端 + 本地 React 后台 + Python AI 脚本”的组合
- 这套方案在开发效率、结构清晰度、本地部署稳定性和答辩表达上最平衡
- 它既不是最重的方案，也不是最草率的方案，而是当前比赛项目最合适的第一版工程基线
