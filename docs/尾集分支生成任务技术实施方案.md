# Drama Pulse 尾集分支生成任务技术实施方案

> 说明：本文档是 `尾集自定义分支生成任务` 的独立实施方案，服务于后续数据库迁移、服务端开发、管理后台扩展、联调与验收。若与 [产品方案初稿.md](/Users/a0000/Desktop/项目文件/drama-pulse/docs/产品方案初稿.md)、[技术方案初稿.md](/Users/a0000/Desktop/项目文件/drama-pulse/docs/技术方案初稿.md) 存在冲突，以本方案中对 `BranchTask` 图文分镜链路的最新定义为准。

## 1. 文档目标

本文档只解决一件事：

`如何把尾集自定义分支，从“用户输入一句话 -> 异步生成文本结果”的骨架，升级为“有上下文、有阶段状态、有分镜可视化图片、有图文 manifest、有可选 TTS / 视频扩展位”的完整任务系统。`

本文档用于指导以下工作：

- `BranchTask` 数据结构扩展
- 固定分支候选生成链路复用
- 服务端异步任务编排实现
- 分支生成链路的阶段拆分
- 后台任务详情与重试能力扩展
- Android 端结果消费的兼容策略
- 测试与验收标准制定

## 2. 设计边界

### 2.1 本方案覆盖

- 尾集 `自定义分支任务` 的创建、执行、查询、重试、落库、回显
- 与剧情上下文、角色资料、场景资料、参考素材相关的任务输入准备
- 与文本剧情、分镜、分镜配图、图文 manifest、可选 TTS / 视频增强任务相关的任务输出组织
- 管理后台对任务执行过程的查看与干预

补充约定：

- 播放完最后一集后，客户端先弹出尾集分支弹窗
- 弹窗内固定展示 `2 个固定分支 + 1 个自定义选项`
- 两个固定分支对应预生成好的图文分镜结果
- 用户选择自定义选项后，才创建 `BranchTask`
- 自定义任务创建后进入分支列表进行异步生成和状态查看
- 固定分支和自定义分支共享同一套“尾集后继续参与剧情”的产品入口，但只有自定义分支进入任务链路

### 2.2 本方案不覆盖

- 固定分支资源制作流程
- Android 分支结果页的视觉细节
- 具体视频生成供应商接入细节
- 全量内容审核系统

说明：

- 固定分支的“候选内容生成”可以复用本方案中的上下文准备、剧情扩写、分镜生产能力
- 但固定分支不通过 `BranchTask` 面向用户逐条创建，而是由离线生产任务生成候选后写入 `BranchOption`

### 2.3 实施原则

- 不推翻当前已跑通的 `BranchTask` 基础链路
- 优先兼容现有 Android 客户端字段消费方式
- 先稳住文本、分镜、分镜配图与图文 manifest 结果，再增量挂接 TTS 或视频增强
- 所有重任务异步执行，不阻塞 API 主线程

### 3.5 当前问题

当前实现已经能完成第一阶段 Demo，但还存在 5 个明显缺口：

- 任务输入只有 `user_prompt`，缺少稳定的剧情上下文包
- 任务执行只有一个大步骤，无法知道失败卡在哪一段
- 文本结果、分镜图片结果和图文结果没有分层，后续很难扩展
- 后台虽然能重试，但看不到“为什么失败、失败在哪一步”
- 若后续接角色参考图、场景参考图、TTS 或视频增强，当前结构无法承载
- 当前分支结果缺少稳定的图文产物，即使没有视频也不够适合展示

## 4. 目标实施形态

目标链路定义如下：

`尾集播放结束 -> 弹出尾集分支弹窗 -> 展示 2 个固定分支和 1 个自定义选项 -> 用户选择自定义 -> 输入提示词 -> 创建 BranchTask -> 进入分支列表 -> 准备剧情上下文 -> 解析分支方向 -> 生成结构化剧情 -> 准备角色/场景素材 -> 生成分镜 -> 生成分镜配图 -> 组装图文 manifest -> 可选补 TTS / 视频增强 -> 返回文本/分镜/配图结果 -> 后台可查看和重试`

这个目标链路拆成两层：

### 4.1 对外任务状态

继续保留现有客户端可消费状态：

- `pending`
- `running`
- `success`
- `failed`
- `timeout`
- `blocked`

### 4.2 对内阶段状态

新增任务内部阶段：

- `queued`
- `context_prepared`
- `prompt_interpreted`
- `story_generated`
- `assets_prepared`
- `storyboard_generated`
- `storyboard_images_generating`
- `storyboard_images_generated`
- `manifest_generated`
- `review_pending`
- `completed`
- `failed`

这样做的目的：

- Android 端不需要一次性改状态枚举
- 服务端和后台能知道任务执行到哪一步
- 以后可针对阶段做局部复跑，而不是每次整任务重跑

### 4.3 具体分支方案基线

为了让实现层不只停留在抽象字段，本方案约定首批尾集分支按下面的具体结构落地：

- `固定分支`
  - 目标：基于剧集上下文自动产出两条“合理且互相区分”的分支结局
  - 结果介质：预生成图文分镜结果
  - 方向来源：不由文档或实现层人工硬编码，而是复用已有分支生成链路，基于 `story_context_package + tail_state_snapshot` 先生成多条候选，再选出 `2` 条写入固定分支
  - 建议长度：每条 `8 到 12` 张分镜卡
  - 结构要求：都必须承接尾集结束状态，并且两条固定分支在情绪落点或叙事目标上有明显差异
- `自定义分支`
  - 目标：在不推翻原剧既有设定的前提下，根据用户提示词产出一条更短、更聚焦的替代结局
  - 结果介质：异步生成图文分镜结果
  - 建议长度：`5 到 7` 张分镜卡
  - 结构要求：第 1 张承接尾集结束瞬间，第 2 到 4 张推进冲突，第 5 到 6 张完成反转或情绪兑现，最后一张做结局卡

统一展示约束：

- 每个分镜卡由 `分镜图 + 图下长旁白文案 + 可选短对白` 组成
- 固定分支与自定义分支尽量复用同一套生产流水线，只是固定分支离线预生成
- 当前“解说”先按纯文字旁白处理，同时预留后续 TTS 字段

固定分支候选生成建议：

- 先由模型基于完整剧情上下文生成 `3 到 5` 条固定分支候选摘要
- 每条候选至少包含 `title / hook / summary / ending / emotion_tags`
- 再按“和原结局区分度、剧情合理性、情绪完成度、分镜可视化可行性”筛掉不合适候选
- 最终选出 `2` 条进入固定分支生产流水线

建议在 `BranchTask` 上新增以下字段：

- `branch_type`
- `pipeline_stage`
- `story_context_json`
- `tail_state_snapshot_json`
- `character_bible_json`

- `narration_payload_json`
- `image_task_status`
- `image_task_payload_json`
- `review_notes`

### 字段说明

`branch_type`

- 含义：本次分支任务归类后的方向
- 建议取值：
  - `revenge`
  - `truth_reveal`
  - `misunderstanding_fix`
  - `hidden_line`
  - `daily_extra`
  - `custom`

`pipeline_stage`

- 含义：任务当前执行阶段
- 用途：后台排障、重试定位、阶段耗时统计

`story_context_json`

- 含义：本次任务使用的全剧资料包摘要
- 内容建议：
  - 全剧大纲
  - 分集摘要
  - 关键事件时间线
  - 主要人物关系

`tail_state_snapshot_json`

- 含义：尾集结束那一刻的世界状态
- 内容建议：
  - 谁活着
  - 谁受伤
  - 谁知道真相
  - 谁和谁关系如何
  - 当前主要矛盾是否解决

`character_bible_json`

- 含义：本次任务使用的角色资料包
- 内容建议：
  - 姓名
  - 年龄感
  - 外形特征
  - 身份气质
  - 角色关系
  - 禁止偏移点

`reference_assets_json`

- 含义：角色图、场景图、封面图等参考素材清单
- 内容建议：
  - 资产类型
  - 角色名或场景名
  - 来源 URL
  - 本地缓存路径
  - 是否命中成功

`prompt_package_json`

- 含义：任务执行统一输入包
- 内容建议：
  - `user_prompt_raw`
  - `user_prompt_normalized`
  - `branch_type`
  - `world_rules`
  - `negative_constraints`
  - `output_contract`
  - `asset_requirements`

`story_expansion_json`

- 含义：结构化剧情扩写结果
- 内容建议：
  - `title`
  - `hook`
  - `summary`
  - `conflict`
  - `twist`
  - `ending`
  - `emotion_tags`
  - `cast`

补充要求：

- 这部分输出不能只是松散文字，必须保持层级清晰，方便后续直接拆分镜
- `summary` 建议按 `开端 / 推进 / 反转 / 结局` 四段组织
- `cast` 不能只列名字，建议同时带上 `roleFunction`，例如 `主视角 / 对手 / 关键证人 / 情绪承接`

推荐结构：

`shot_prompt_json`

- 含义：镜头级分镜 prompt 包
- 内容建议：

  - `storyTitle`
  - `storyHook`
  - `readingMode`
  - `visualStyle`
  - `globalCharacterConsistencyNotes`
  - `globalSceneConsistencyNotes`
  - `shots`

补充约束：

- 当前主链路不是“单条 15 秒视频片段”，而是“图文分镜结果”
- 固定分支建议产出 `8 到 12` 张分镜图，自定义分支建议产出 `5 到 7` 张分镜图
- 每张图都必须能承载一段图下长旁白文案
- 每张图都要显式声明依赖的角色参考图、场景参考图和连续性约束

`shots` 内建议至少包含：

- 镜头序号

## 5.2 字段落地原则

- 新增 JSON 字段统一使用字符串存储合法 JSON
- Android 与管理后台当前版本都应直接消费 `storyboard_manifest_json` 与 `storyboard_images_json`
- `result_title / result_hook / result_story / storyboard_json` 继续作为客户端主结果层
- `storyboard_images_json` 与 `storyboard_manifest_json` 共同作为第一版分支结果的核心可视化层
- 视频能力不属于当前主链路，如后续恢复视频增强，再以可选扩展字段单独补充

## 6. 服务端模块拆分方案

## 6.1 模块目标

当前 `taskQueue/index.ts` 里只有一个 `taskGenerator(task)`。后续不要继续把所有逻辑塞进这个函数里，而是拆成专门的分支任务服务。

建议目录：

`server/src/services/branchTask/`

建议子模块：

- `constants.ts`
- `types.ts`
- `contextBuilder.ts`
- `promptInterpreter.ts`
- `storyGenerator.ts`
- `assetCollector.ts`
- `storyboardGenerator.ts`
- `storyboardImageGenerator.ts`
- `storyboardManifestBuilder.ts`
- `videoOrchestrator.ts`
- `resultProjector.ts`
- `stageUpdater.ts`

## 6.2 模块职责

`contextBuilder`

- 根据 `episodeId` 组装剧情上下文
- 读取全剧摘要、尾集状态、人物资料

`promptInterpreter`

- 规范化用户提示词
- 归类 `branch_type`
- 生成 `prompt_package_json`

`storyGenerator`

- 调用模型生成结构化剧情扩写
- 输出 `story_expansion_json`
- 同步回填 `result_title / result_hook / result_story`

`assetCollector`

- 根据出场角色和场景需求收集参考素材
- 输出 `reference_assets_json`

`storyboardGenerator`

- 根据剧情结果和素材结果生成镜头级分镜
- 输出 `shot_prompt_json`
- 同步投影为 `storyboard_json`

`storyboardImageGenerator`

- 根据分镜结果和参考素材生成分镜配图
- 输出 `storyboard_images_json`
- 管理 `image_task_status`
- 记录 `image_task_payload_json`

`storyboardManifestBuilder`

- 把分镜图、图下旁白、对白和顺序结构组装成客户端可直接消费的图文 manifest
- 输出 `storyboard_manifest_json`
- 同步整理 `narration_payload_json`

`videoOrchestrator`

- 当前阶段不进入主实现
- 若后续恢复 TTS 或视频增强，再独立接入可选编排器

`resultProjector`

- 把结构化内部结果映射成客户端稳定字段

`stageUpdater`

- 统一更新 `pipeline_stage`
- 记录阶段耗时
- 处理失败阶段写入

## 6.3 与 taskQueue 的关系

`server/src/services/taskQueue/index.ts` 继续保留为：

- 轮询器
- 调度器
- 超时保护器

不要让它承担：

- 提示词理解逻辑
- 剧情生成逻辑
- 素材拼装逻辑
- 分镜配图逻辑
- 视频任务编排逻辑
- 图文 manifest 组装逻辑

推荐关系如下：

`taskQueue.processTask(task) -> branchTaskExecutor.execute(task)`

## 7. 任务执行状态机

## 7.1 标准流程

```text
pending
  -> running + queued
  -> context_prepared
  -> prompt_interpreted
  -> story_generated
  -> assets_prepared
  -> storyboard_generated
  -> storyboard_images_generating
  -> storyboard_images_generated
  -> manifest_generated
  -> review_pending
  -> success + completed
```

## 7.2 简化文本模式

如果当前版本只跑图文主链路，流程收敛为：

```text
pending
  -> running + queued
  -> context_prepared
  -> prompt_interpreted
  -> story_generated
  -> assets_prepared
  -> storyboard_generated
  -> storyboard_images_generating
  -> storyboard_images_generated
  -> manifest_generated
  -> success + completed
```

此时：

- `image_task_status = success | partial_failed | failed`
- 文本、分镜、分镜配图与图文 manifest 结果即可进入客户端展示

## 7.3 失败规则

### 失败

- 模型输出非合法 JSON
- 素材收集器抛异常
- 分镜生成结构不符合约束
- 分镜配图请求结构不合法

处理：

- 外层 `status = failed`
- 内层 `pipeline_stage = failed`
- 写入 `fail_reason`

### 超时

- 单任务执行时间超出阈值

处理：

- 外层 `status = timeout`
- 内层保留最后阶段
- 写入 `fail_reason`

### 阻断

- 提示词违规
- 内容安全命中
- 结果不适合展示

处理：

- 外层 `status = blocked`
- 保留中间结果与审核备注

### 配图部分失败

- 文本与分镜成功
- 部分镜头图片生成失败

处理：

- 外层可保持 `status = success`
- `image_task_status = partial_failed`
- 后台支持单独补图或整任务重试

### 9. 当前首批角色三视图准备清单

当前分支剧情已经基本收敛，图文分镜阶段不建议继续“全角色都做三视图”。
更合理的做法是按 `必须做 / 建议做 / 可用模板替代` 三层准备。

统一原则：

- `必须做`：固定双分支和大部分自定义延展都会反复出现，且需要稳定脸和服装连续性
- `建议做`：不是每条分支都上镜，但一旦进入剧情，会明显影响情绪对手戏
- `模板替代`：可先用通用群演、亲戚、邻居模板，不单独投入三视图制作

三视图素材建议至少包含：

- 正面清晰图
- 左或右 45 度侧面图
- 半身或全身定装图

若角色在尾集分支里存在明显“状态切换”，建议额外补一套常用造型：

- 日常态
- 对峙态 / 情绪高压态

- 后台重试后阶段归零
- 阻断任务不可继续点赞评论的规则
- 分镜图列表可被后台正确读取

## 12.4 回归重点

每次改动都要回归：

- Android 分支结果页能否继续正常展示
- 当前 `GET /branch-tasks/:taskId` 是否仍兼容现有 DTO
- 后台任务列表是否分页正常
- 超时任务能否正常重试

## 13. 风险与兜底

## 13.1 主要风险

- 模型输出不稳定，字段缺失
- 剧情上下文包过大，导致执行变慢
- 参考素材匹配失败，分镜图风格不稳定
- 可选增强任务耗时过长，影响演示

## 13.2 兜底策略

- 所有模型输出先做 JSON 校验
- 长上下文只保留摘要，不传全文
- 素材收集失败不阻断文本结果
- 分镜图失败时保留文本与分镜结果
- manifest 组装失败时保留文本、分镜与分镜配图结果
- 可选增强失败时保留文本、分镜、分镜配图与图文 manifest 结果
- 后台保留整任务重试入口

## 14. 验收标准

本专项方案完成后，至少应满足以下标准：

### P0

- 用户能在尾集创建自定义分支任务
- 任务能异步生成标题、hook、剧情、分镜
- 任务能生成分镜配图
- 客户端能继续查看结果
- 后台能查看任务详情和失败原因
- 后台能重试失败/超时任务

### P1

- 任务能记录剧情上下文包
- 任务能记录提示词输入包
- 后台能看到当前阶段
- 测试覆盖核心成功/失败链路

### P2

- 任务能记录角色/场景参考素材
- 任务能挂接 TTS / 视频增强
- 可选增强失败不影响图文主链路可展示

## 15. 结论

尾集分支生成任务不应该继续停留在“一个 prompt 直接吐一段故事”的 Demo 形态，而应该升级成：

`有上下文输入、有阶段状态、有结构化产物、有分镜可视化图片、有图文 manifest、有后台可观测性、有失败兜底、有可选增强扩展位的异步任务系统。`

这套方案的关键不是一次做完所有能力，而是：

- 先保住当前骨架
- 再分阶段加字段、加阶段、加分镜图、加 manifest 和后台观察能力
- 最后再挂 TTS 或视频增强

这样最适合当前比赛项目的节奏，也最利于后续逐步实施。
