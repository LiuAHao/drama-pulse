# Drama Pulse 高光识别与打标实现方案

## 1. 文档目的

本文档用于冻结 Drama Pulse 项目中“高光识别、人工打标、客户端互动触发”这一条链路的实现方向。

本文档重点不是研究“最强识别模型”，而是定义一条：

- 可快速产出候选
- 可人工校准
- 可稳定进入客户端播放链路
- 可支撑比赛演示

的高光处理方案。

---

## 2. 核心原则

本模块整体原则定义为：

`AI 先提候选，人工做冻结，客户端只消费已确认标签`

具体含义：

- AI 负责从剧集文本中发现潜在高光点
- 管理后台负责人工修正和最终确认
- Android 客户端不做高光判断，只按标签触发互动模板

这条原则的目标是：

- 降低全自动误判风险
- 保证答辩演示时的高光质量
- 让客户端逻辑简单、稳定、可控

---

## 3. 总体链路

高光处理链路建议固定为：

`视频/字幕 -> 文本提取 -> 文本结构化 -> LLM 候选识别 -> 候选高光入库 -> 后台人工审核 -> 确认标签下发 -> 客户端时间轴触发互动`

更细化为：

1. 获取剧集原始文本
2. 生成带时间戳的结构化文本片段
3. 使用大模型识别高光候选
4. 生成候选高光 JSON
5. 候选高光写入数据库，状态为 `candidate`
6. 管理后台人工审核，状态改为 `confirmed` 或 `disabled`
7. 客户端仅拉取 `confirmed` 高光
8. 播放命中时间轴时触发互动组件

---

## 4. 模块拆分

本模块建议拆成 3 层：

## 4.1 高光识别层

职责：

- 获取原始文本
- 做时间戳对齐
- 基于 LLM 识别高光候选

输出：

- `candidate` 状态的高光记录

## 4.2 高光冻结层

职责：

- 在后台查看候选高光
- 人工修正时间戳、类型、标题、模板、互动选项
- 将候选高光冻结为可下发标签

输出：

- `confirmed` 状态的高光记录

## 4.3 客户端消费层

职责：

- 拉取 `confirmed` 高光
- 按 `startTimeMs ~ endTimeMs` 触发互动
- 上报用户互动结果

输出：

- 高光互动体验
- 历史互动统计数据

---

## 5. 高光来源策略

## 5.1 优先级建议

建议高光文本来源优先级为：

1. 原始字幕文件
2. 现成逐字稿
3. ASR 转写结果

原因：

- 直接字幕通常最稳定
- ASR 作为补充即可
- 不建议一开始就上“纯视频理解”

## 5.2 第一版建议

第一版建议优先使用：

- 若有字幕：直接导入字幕文本
- 若无字幕：使用 ASR 生成字幕文本

然后统一整理成同一结构：

- 文本内容
- 起止时间
- 所属剧集
- 片段序号
- 前后文窗口

## 5.3 媒体预处理链路细化

结合当前项目实际情况，媒体预处理链路建议进一步冻结为：

`mp4 -> 音频抽取 -> ASR 转写 / 字幕对照表 -> 统一结构化字幕 JSON`

具体说明：

1. 输入源以剧集原始 `mp4` 为准
2. 若已有外挂字幕，可直接跳过音频抽取和 ASR
3. 若无字幕，则先从视频中抽取音频
4. 将音频送入 ASR 工具，生成带时间戳的字幕对照表
5. 将字幕对照表统一转换为 Drama Pulse 内部 transcript JSON

这一层的重点不是“生成最好看的字幕文件”，而是：

- 稳定拿到带时间戳的文本
- 保证后续 LLM 可以回到时间轴
- 保证同一剧集可以重复处理和复核

## 5.4 Transcript 中间结果格式

无论原始来源是：

- 外挂字幕
- ASR 结果
- 其他逐字稿

都建议统一整理成如下中间格式：

```json
{
  "dramaId": "drama_001",
  "episodeId": "episode_001",
  "source": "asr",
  "language": "zh",
  "segments": [
    {
      "segmentId": "seg_0001",
      "startTimeMs": 1230,
      "endTimeMs": 4560,
      "text": "你终于来了。",
      "speakerGuess": null,
      "targetCharacterGuess": null,
      "mentionedCharacters": [],
      "characterGuessConfidence": null
    }
  ]
}
```

说明：

- `source` 用于标记来源，例如 `subtitle`、`asr`
- `segments` 是后续送给 LLM 的基础素材
- `segmentId` 便于后续人工复核和日志排查

## 5.5 AsrTools 参考接入策略

当前本地已有参考项目：

- `/Users/a0000/Desktop/项目文件/开源项目/AsrTools`

对该项目的评估结论如下：

### 可借鉴的部分

- 已具备“音频/视频 -> 带时间戳文本”的基础能力
- `BaseASR` / `ASRData` 抽象清晰，便于理解输出结构
- `WhisperASR` 已经支持通过 OpenAI 兼容接口输出 `verbose_json`
- 支持直接生成 `srt / txt / json / ass`

### 不建议直接整包并入主仓库的原因

- 该项目本质上是通用 ASR 工具项目，包含 GUI 和多种外部平台适配，不是围绕 Drama Pulse 业务设计
- 仓库内包含与本项目无关的 GUI、演示资源和多后端实现，直接引入会增加维护负担
- 许可证为 `GPL v3`，如果整包合并进主仓库，需要额外谨慎处理授权边界

### 当前推荐策略

推荐将 `AsrTools` 作为：

- 本地参考实现
- 可单独调用的外部 ASR 工具
- 或可选择性封装的独立脚本依赖

而不是直接把整个项目源码复制进 Drama Pulse 主仓库。

第一版更稳的做法是：

1. 保持 `AsrTools` 作为独立本地工具存在
2. 在 `drama-pulse/scripts/` 下编写我们自己的包装脚本
3. 包装脚本只消费其输出结果，统一转成 Drama Pulse 的 transcript JSON
4. 后续如需完全自研或替换其他 ASR，只需更换包装层，不影响后面的高光识别链路

### 第一版接入建议

建议优先参考和复用其以下思路：

- `ASRDataSeg(text, start_time, end_time)` 的时间片表示
- `ASRData.to_srt()` / `to_json()` 的字幕序列化方式
- `WhisperASR` 的 `verbose_json -> segments` 解析逻辑

但我们自己的主实现仍建议放在：

- `drama-pulse/scripts/highlight/`

以保证：

- 目录清晰
- 授权边界清晰
- 后续更换 ASR 工具时影响最小

### 本地验证结论

针对 `AsrTools` 当前已做过一次本地接入验证，结论如下：

- `BcutASR`：
  - 能返回带 `utterances` 的结构化结果
  - 每个 `utterance` 包含：
    - `start_time`
    - `end_time`
    - `transcript`
    - `words`
  - 其中 `words` 还能提供更细的字级时间戳
  - 这种结构非常适合后续转成 Drama Pulse transcript JSON

- `KuaiShouASR`：
  - 当前接口返回 `code=501`
  - 错误信息为“效果 subtitle_generate 禁用”
  - 不适合作为第一版正式后端

因此当前更推荐：

- 将 `BcutASR` 的输出格式作为参考样板
- 在我们自己的包装脚本里统一转成内部 transcript 结构
- 不把某一个单独第三方后端视为不可替代的正式实现

## 6. 文本结构化方案

在送入 LLM 之前，需要把文本整理成结构化片段。

建议每个片段包含：

- `episodeId`
- `segmentId`
- `startTimeMs`
- `endTimeMs`
- `text`
- `speakerGuess`（可选）
- `targetCharacterGuess`（可选）
- `previousContext`
- `nextContext`

建议切分粒度：

- 按字幕句子切
- 或按 3-10 秒一个文本块切

原则：

- 不宜太碎，否则上下文不足
- 不宜太长，否则模型定位不准

## 6.1 角色区分策略

这类 ASR 工具通常只能稳定输出：

- 文本内容
- 起止时间

通常并不能天然区分：

- 具体是谁说的
- 说话对象是谁
- 某句台词属于哪一个角色

因此本项目对“角色区分”的策略建议定义为：

`角色推断是增强项，不是第一阶段阻塞项`

### 第一版建议

第一版高光识别链路可以先不强依赖角色识别，优先保证：

- 时间点可用
- 文本可读
- 高光类型可判断
- 模板可映射

### 推荐实现方式

第一版推荐做“文本级角色弱标注”，而不是做复杂的音频说话人分离。

建议新增可选字段：

- `speakerGuess`
- `targetCharacterGuess`
- `mentionedCharacters`
- `characterGuessConfidence`

示例：

```json
{
  "segmentId": "seg_0012",
  "startTimeMs": 52340,
  "endTimeMs": 56120,
  "text": "你居然一直在骗我？",
  "speakerGuess": "女主",
  "targetCharacterGuess": "男主",
  "mentionedCharacters": ["女主", "男主"],
  "characterGuessConfidence": 0.62
}
```

### 推荐来源

角色弱标注可以来自：

- 字幕文本中的人名或称呼
- 剧情上下文中的角色名
- 大模型根据前后文做文本推断

### 不建议第一版采用的方式

第一版不建议直接做：

- 说话人分离（speaker diarization）
- 声纹识别
- 音频角色身份映射

原因：

- 实现复杂度明显提高
- 准确率不一定足够稳定
- 对比赛项目来说投入产出比偏低

### 与后台审核的关系

若后续把角色推断加入到高光候选中，后台也应允许人工修正这些字段。

也就是说：

- AI 负责猜测角色
- 人工负责最终确认

## 7. 高光候选识别方案

## 7.1 识别目标

识别层的目标不是直接给出“最终正确高光”，而是快速给出：

- 候选时间点
- 候选类型
- 候选互动模板建议

## 7.2 候选输出字段

建议 LLM 输出至少包含以下字段：

- `episodeId`
- `startTimeMs`
- `endTimeMs`
- `highlightType`
- `title`
- `description`
- `intensity`
- `templateId`
- `interactionOptions`
- `reason`
- `confidence`

## 7.3 候选状态

候选高光初始写入数据库时：

- `status = candidate`

后台审核后再改为：

- `confirmed`
- 或 `disabled`

---

## 8. 高光类型冻结建议

第一版不建议类型过多，建议先冻结为 5 类：

- `feel_good`：爽点
- `reversal`：反转
- `conflict`：冲突
- `sweet`：甜蜜
- `suspense`：悬念

原因：

- 便于客户端做清晰映射
- 便于后台审核
- 便于答辩讲解

---

## 9. 模板映射建议

建议一版先固定“类型 -> 模板”的简单映射关系：

- `feel_good -> emotion_button`
- `reversal -> emotion_button`
- `conflict -> vote_side`
- `sweet -> emotion_button`
- `suspense -> suspense_lock`

后续可扩展更多模板，但第一版要控制数量。

原则：

- 不同类型可以共用模板
- 客户端模板数要少
- 映射规则尽量固定，避免每部剧都重新定义

---

## 10. 互动选项建议

LLM 产出的 `interactionOptions` 只需要给出 2-4 个短选项。

要求：

- 单个选项尽量 2-6 字
- 优先情绪表达，不要长句解释

示例：

### 爽点

- 爽了
- 打脸
- 继续反杀

### 反转

- 啊？
- 没想到
- 细思极恐

### 冲突

- 别忍
- 开怼
- 站她

### 甜蜜

- 磕到了
- 太甜了
- 锁死

### 悬念

- 不对劲
- 有坑
- 继续看

---

## 11. LLM 输入输出建议

## 11.1 输入内容

建议输入：

- 当前片段文本
- 当前片段时间范围
- 前后文摘要或上下文片段
- 角色名（如可提取）
- 剧种风格提示

## 11.2 输出格式

必须要求模型输出结构化 JSON。

建议输出为数组形式：

```json
[
  {
    "episodeId": "ep_001_03",
    "startTimeMs": 82400,
    "endTimeMs": 90500,
    "highlightType": "reversal",
    "title": "身份反转曝光",
    "description": "主角隐藏身份在众人面前被揭开，剧情出现明显反转。",
    "intensity": 4,
    "templateId": "emotion_button",
    "interactionOptions": ["啊？", "没想到", "细思极恐"],
    "reason": "情绪冲击强，观众表达欲高",
    "confidence": 0.86
  }
]
```

## 11.3 服务端要求

服务端或脚本层必须：

- 校验 JSON 合法性
- 校验字段完整性
- 不合法时重试一次
- 仍失败时标记本集识别失败

## 12. 后台审核流程

后台建议提供一个最小高光审核工作流：

1. 查看某集候选高光列表
2. 点击高光可查看：
   - 时间戳
   - 标题
   - 类型
   - 模板
   - 推荐按钮
   - LLM 理由
3. 播放对应片段进行人工确认
4. 可修改：
   - 开始/结束时间
   - 高光类型
   - 标题文案
   - 模板
   - 互动选项
5. 操作：
   - `确认`
   - `禁用`

后台审核结果定义：

- `candidate -> confirmed`
- `candidate -> disabled`

  - `ai`

- `status` 建议取值：
  - `candidate`
  - `confirmed`
  - `disabled`

## 14. 客户端消费规则

客户端不负责识别高光，只负责消费确认后的标签。

客户端规则建议固定为：

1. 只请求 `confirmed` 高光
2. 基于时间轴判断命中
3. 命中后按 `templateId` 触发组件
4. 点击互动后上报事件
5. 根据服务端返回统计显示热度或历史互动

客户端不要做：

- 类型重判
- 模板改写
- 本地自动增删高光

---

## 15. 与互动数据回流的关系

互动数据可以作为后续高光优化依据，但第一版不建议直接影响已确认标签。

推荐策略：

- 某时间点互动数异常集中
- 进入 `候选池`
- 后续由后台或离线脚本决定是否新增高光

即：

`互动数据 -> 候选线索`

而不是：

`互动数据 -> 直接修改线上高光`

---

## 16. 实现顺序建议

建议按以下顺序实现：

### 第一步：定义高光 schema

输出：

- 高光字段结构
- 类型枚举
- 模板枚举

### 第二步：完成文本提取

输出：

- 统一结构化字幕/逐字稿

建议拆成 3 个可执行小步骤：

1. `mp4 -> audio`
2. `audio -> subtitle / transcript`
3. `subtitle / transcript -> Drama Pulse transcript JSON`

对应建议脚本：

- `scripts/highlight/extract_audio.py`
- `scripts/highlight/transcribe_episode.py`
- `scripts/highlight/normalize_transcript.py`

### 第三步：完成 LLM 候选识别脚本

输出：

- 候选高光 JSON

对应建议脚本：

- `scripts/highlight/detect_highlights.py`
- `scripts/highlight/build_prompt.py`

### 第四步：候选高光入库

输出：

- `candidate` 状态高光记录

对应建议脚本或接口：

- `scripts/highlight/validate_candidates.py`
- `scripts/highlight/import_candidates.py`

### 第五步：后台审核页

输出：

- 高光确认/禁用流程

### 第六步：客户端消费已确认高光

输出：

- 时间轴触发
- 模板互动

### 第七步：互动数据反哺候选池

输出：

- 热点回流机制

## 17. 风险点

## 17.1 全自动误判风险

风险：

- 模型会把普通剧情误判为高光
- 模型会漏掉真正高光

控制：

- 必须经过人工冻结

## 17.2 时间戳不准

风险：

- 字幕时间和实际情绪峰值存在偏移

控制：

- 后台支持微调起止时间

## 17.3 模板过多导致客户端复杂

风险：

- 模板越多，客户端越容易乱

控制：

- 第一版模板严格收敛

## 17.4 文本来源质量不稳定

风险：

- ASR 错字、断句问题影响候选质量

控制：

- 优先原字幕，其次 ASR

## 17.5 外部 ASR 工具授权与维护边界

风险：

- 直接复制第三方开源工具源码，可能带来许可证和维护负担

控制：

- 第一版优先做“外部工具调用 + 内部结构化转换”
- 不把第三方通用 GUI 工具直接并入主业务代码

## 18. 代码落地建议

这一模块不应只停留在文档层，建议至少补齐以下代码层能力：

### 18.1 媒体预处理脚本

负责：

- 从 `mp4` 抽取音频
- 调用 ASR 工具
- 生成原始字幕结果

建议文件：

- `scripts/highlight/extract_audio.py`
- `scripts/highlight/transcribe_episode.py`

### 18.2 Transcript 归一化脚本

负责：

- 将 `srt / json / txt` 等结果统一整理成内部 transcript JSON

建议文件：

- `scripts/highlight/normalize_transcript.py`

### 18.3 LLM 高光识别脚本

负责：

- 读取 transcript JSON
- 生成 prompt
- 调用模型
- 输出候选高光 JSON

建议文件：

- `scripts/highlight/detect_highlights.py`

### 18.4 结构校验脚本

负责：

- 校验模型输出 JSON 是否合法
- 检查字段完整性、枚举值、时间范围

建议文件：

- `scripts/highlight/validate_candidates.py`
