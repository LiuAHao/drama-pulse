# Drama Pulse 高光识别与打标实现方案

## 1. 文档目的

本文档用于冻结 Drama Pulse 项目中“高光识别、人工打标、客户端互动触发”这一条链路的实现方向。

本文档重点不是研究“最强识别模型”，而是定义一条：

- 可快速产出候选
- 可人工校准
- 可稳定进入客户端播放链路
- 可支撑比赛演示

的高光处理方案。

---

## 2. 核心原则

本模块整体原则定义为：

`AI 先提候选，人工做冻结，客户端只消费已确认标签`

具体含义：

- AI 负责从剧集文本中发现潜在高光点
- 管理后台负责人工修正和最终确认
- Android 客户端不做高光判断，只按已确认标签触发对应互动表现

这条原则的目标是：

- 降低全自动误判风险
- 保证答辩演示时的高光质量
- 让客户端逻辑简单、稳定、可控

---

## 3. 总体链路

高光处理链路建议固定为：

`视频/字幕 -> 文本提取 -> 文本结构化 -> Seedance 初筛候选 -> DeepSeek 二次复核与时间修正 -> 候选高光入库 -> 后台抽检/少量人工审核 -> 确认标签下发 -> 客户端时间轴触发互动`

更细化为：

1. 获取剧集原始文本
2. 生成带时间戳的结构化文本片段
3. 生成本集剧情上下文包（story context）
4. 使用 Seedance 对 transcript 做初步高光识别
5. 使用 DeepSeek 对 Seedance 候选做二次复核与时间区间修正
6. 生成候选高光 JSON
7. 候选高光写入数据库，状态为 `candidate`
8. 管理后台只做抽检、最终确认和少量微调，状态改为 `confirmed` 或 `disabled`
9. 客户端仅拉取 `confirmed` 高光
10. 播放命中时间轴时触发互动组件

补充冻结：

- `startTimeMs / endTimeMs` 定义高光内容本体区间
- `interactionStartMs / interactionEndMs` 定义用户可重复点击的交互窗口
- `interactionAppearMs` 定义交互组件真正出现的时间点

这三者不要混用。

---

## 4. 模块拆分

本模块建议拆成 3 层：

## 4.1 高光识别层

职责：

- 获取原始文本
- 做时间戳对齐
- 基于 Seedance 做初步候选识别
- 基于 DeepSeek 做候选复核与时间修正

输出：

- `candidate` 状态的高光记录
- 其中必须包含 `interactionAppearMs`

## 4.2 高光冻结层

职责：

- 在后台查看候选高光
- 人工修正时间戳、类型、标题、互动选项和交互时间窗口
- 将候选高光冻结为可下发标签

输出：

- `confirmed` 状态的高光记录

## 4.3 客户端消费层

职责：

- 拉取 `confirmed` 高光
- 先按 `interactionAppearMs` 让组件出现
- 在 `interactionStartMs ~ interactionEndMs` 窗口内允许用户重复点击触发
- 上报用户互动结果

输出：

- 高光互动体验
- 历史互动统计数据

## 5. 高光来源策略

## 5.1 优先级建议

建议高光文本来源优先级为：

1. 原始字幕文件
2. 现成逐字稿
3. ASR 转写结果

原因：

- 直接字幕通常最稳定
- ASR 作为补充即可
- 不建议一开始就上“纯视频理解”

## 5.2 第一版建议

第一版建议优先使用：

- 若有字幕：直接导入字幕文本
- 若无字幕：使用 ASR 生成字幕文本

然后统一整理成同一结构：

- 文本内容
- 起止时间
- 所属剧集
- 片段序号
- 前后文窗口

## 5.3 媒体预处理链路细化

结合当前项目实际情况，媒体预处理链路建议进一步冻结为：

`mp4 -> 音频抽取 -> ASR 转写 / 字幕对照表 -> 统一结构化字幕 JSON`

具体说明：

1. 输入源以剧集原始 `mp4` 为准
2. 若已有外挂字幕，可直接跳过音频抽取和 ASR
3. 若无字幕，则先从视频中抽取音频
4. 将音频送入 ASR 工具，生成带时间戳的字幕对照表
5. 将字幕对照表统一转换为 Drama Pulse 内部 transcript JSON

这一层的重点不是“生成最好看的字幕文件”，而是：

- 稳定拿到带时间戳的文本
- 保证后续 LLM 可以回到时间轴
- 保证同一剧集可以重复处理和复核

## 5.4 Transcript 中间结果格式

无论原始来源是：

- 外挂字幕
- ASR 结果
- 其他逐字稿

都建议统一整理成如下中间格式：

```json
{
  "dramaId": "drama_001",
  "episodeId": "episode_001",
  "dramaTitle": "荒年全村啃树皮，我有系统满仓肉",
  "episodeTitle": "第2集",
  "episodeSummary": "女主醒来后彻底看清原身在荒年中的恶名和家庭困境……",
  "mainGenre": "爽文",
  "characterRelationshipSummary": "女主与几个孩子的关系正从恐惧疏离转向试探式靠近……",
  "source": "asr",
  "language": "zh",
  "segments": [
    {
      "segmentId": "seg_0001",
      "startTimeMs": 1230,
      "endTimeMs": 4560,
      "text": "你终于来了。",
      "speakerGuess": null,
      "targetCharacterGuess": null,
      "mentionedCharacters": [],
      "characterGuessConfidence": null
    }
  ]
}
```

说明：

- `source` 用于标记来源，例如 `subtitle`、`asr`
- `segments` 是后续送给 LLM 的基础素材
- `segmentId` 便于后续人工复核和日志排查
- `episodeSummary` / `mainGenre` / `characterRelationshipSummary` 是模型理解人物立场与互动语气的重要上下文，不应留空

## 5.4.1 剧情上下文补充规则

标准化 transcript 时，必须补齐以下上下文：

- `episodeTitle`
- `episodeSummary`
- `mainGenre`
- `characterRelationshipSummary`

推荐来源优先级：

1. `episodes` / `dramas` 表中的基础元数据
2. `data/seed/story_context.json` 中的补充剧情梗概与角色关系摘要
3. 命令行参数临时覆盖

角色关系摘要的目标不是写人物小传，而是帮助模型快速理解：

- 当前主角和家人/对立方处于什么关系
- 当前矛盾更容易让观众站队谁
- 当前温情/反转到底发生在谁和谁之间

## 5.4.2 交互时间字段约定

高光链路中，时间字段统一拆为三层：

- `startTimeMs / endTimeMs`
  - 表示高光内容本体
  - 用于字幕证据、高光抽检、统计落点
- `interactionStartMs / interactionEndMs`
  - 表示交互窗口
  - 用户在这个窗口内可以重复点击、重复触发互动
- `interactionAppearMs`
  - 表示组件真正露出时间
  - 应尽量对齐“观众已经看懂高光成立”的时刻，而不是机械等于高光开始

默认策略：

- `interactionStartMs = startTimeMs`
- `interactionAppearMs = startTimeMs + 600ms`
- 强反转、情绪爆点、立场站队类高光允许后移到 `+900ms ~ +1500ms`
- `interactionEndMs = endTimeMs + 1500ms`

校验约束：

- `interactionStartMs <= interactionAppearMs < interactionEndMs`
- `interactionStartMs` 不应比 `startTimeMs` 晚超过 3 秒
- `interactionAppearMs` 不应明显晚于 `endTimeMs`
- `interactionEndMs` 至少覆盖到高光尾部，并可略向后延续

## 5.5 AsrTools 参考接入策略

当前本地已有参考项目：

- `/Users/a0000/Desktop/项目文件/开源项目/AsrTools`

对该项目的评估结论如下：

### 可借鉴的部分

- 已具备“音频/视频 -> 带时间戳文本”的基础能力
- `BaseASR` / `ASRData` 抽象清晰，便于理解输出结构
- `WhisperASR` 已经支持通过 OpenAI 兼容接口输出 `verbose_json`
- 支持直接生成 `srt / txt / json / ass`

### 不建议直接整包并入主仓库的原因

- 该项目本质上是通用 ASR 工具项目，包含 GUI 和多种外部平台适配，不是围绕 Drama Pulse 业务设计
- 仓库内包含与本项目无关的 GUI、演示资源和多后端实现，直接引入会增加维护负担
- 许可证为 `GPL v3`，如果整包合并进主仓库，需要额外谨慎处理授权边界

### 当前推荐策略

推荐将 `AsrTools` 作为：

- 本地参考实现
- 可单独调用的外部 ASR 工具
- 或可选择性封装的独立脚本依赖

而不是直接把整个项目源码复制进 Drama Pulse 主仓库。

第一版更稳的做法是：

1. 保持 `AsrTools` 作为独立本地工具存在
2. 在 `drama-pulse/scripts/` 下编写我们自己的包装脚本
3. 包装脚本只消费其输出结果，统一转成 Drama Pulse 的 transcript JSON
4. 后续如需完全自研或替换其他 ASR，只需更换包装层，不影响后面的高光识别链路

### 第一版接入建议

建议优先参考和复用其以下思路：

- `ASRDataSeg(text, start_time, end_time)` 的时间片表示
- `ASRData.to_srt()` / `to_json()` 的字幕序列化方式
- `WhisperASR` 的 `verbose_json -> segments` 解析逻辑

但我们自己的主实现仍建议放在：

- `drama-pulse/scripts/highlight/`

以保证：

- 目录清晰
- 授权边界清晰
- 后续更换 ASR 工具时影响最小

### 本地验证结论

针对 `AsrTools` 当前已做过一次本地接入验证，结论如下：

- `BcutASR`：
  - 能返回带 `utterances` 的结构化结果
  - 每个 `utterance` 包含：
    - `start_time`
    - `end_time`
    - `transcript`
    - `words`
  - 其中 `words` 还能提供更细的字级时间戳
  - 这种结构非常适合后续转成 Drama Pulse transcript JSON

- `KuaiShouASR`：
  - 当前接口返回 `code=501`
  - 错误信息为“效果 subtitle_generate 禁用”
  - 不适合作为第一版正式后端

因此当前更推荐：

- 将 `BcutASR` 的输出格式作为参考样板
- 在我们自己的包装脚本里统一转成内部 transcript 结构
- 不把某一个单独第三方后端视为不可替代的正式实现

## 6. 文本结构化方案

在送入 LLM 之前，需要把文本整理成结构化片段。

建议每个片段包含：

- `episodeId`
- `segmentId`
- `startTimeMs`
- `endTimeMs`
- `text`
- `speakerGuess`（可选）
- `targetCharacterGuess`（可选）
- `previousContext`
- `nextContext`

建议切分粒度：

- 按字幕句子切
- 或按 3-10 秒一个文本块切

原则：

- 不宜太碎，否则上下文不足
- 不宜太长，否则模型定位不准

## 6.1 角色区分策略

这类 ASR 工具通常只能稳定输出：

- 文本内容
- 起止时间

通常并不能天然区分：

- 具体是谁说的
- 说话对象是谁
- 某句台词属于哪一个角色

因此本项目对“角色区分”的策略建议定义为：

`角色推断是增强项，不是第一阶段阻塞项`

### 第一版建议

第一版高光识别链路可以先不强依赖角色识别，优先保证：

- 时间点可用
- 文本可读
- 高光类型可判断
- 可稳定映射到固定互动表现

### 推荐实现方式

第一版推荐做“文本级角色弱标注”，而不是做复杂的音频说话人分离。

建议新增可选字段：

- `speakerGuess`
- `targetCharacterGuess`
- `mentionedCharacters`
- `characterGuessConfidence`

示例：

```json
{
  "segmentId": "seg_0012",
  "startTimeMs": 52340,
  "endTimeMs": 56120,
  "text": "你居然一直在骗我？",
  "speakerGuess": "女主",
  "targetCharacterGuess": "男主",
  "mentionedCharacters": ["女主", "男主"],
  "characterGuessConfidence": 0.62
}
```

### 推荐来源

角色弱标注可以来自：

- 字幕文本中的人名或称呼
- 剧情上下文中的角色名
- 大模型根据前后文做文本推断

### 不建议第一版采用的方式

第一版不建议直接做：

- 说话人分离（speaker diarization）
- 声纹识别
- 音频角色身份映射

原因：

- 实现复杂度明显提高
- 准确率不一定足够稳定
- 对比赛项目来说投入产出比偏低

### 与后台审核的关系

若后续把角色推断加入到高光候选中，后台也应允许人工修正这些字段。

也就是说：

- AI 负责猜测角色
- 人工负责最终确认

## 7. 高光候选识别方案

## 7.1 识别目标

识别层的目标不是直接给出“最终正确高光”，而是快速给出：

- 候选时间点
- 候选类型
- 候选互动表现建议

## 7.2 候选输出字段

建议 LLM 输出至少包含以下字段：

- `episodeId`
- `startTimeMs`
- `endTimeMs`
- `highlightType`
- `title`
- `description`
- `intensity`
- `interactionOptions`
- `reason`
- `confidence`

## 7.3 候选状态

候选高光初始写入数据库时：

- `status = candidate`

后台审核后再改为：

- `confirmed`
- 或 `disabled`

## 7.4 双阶段识别与复核策略

为了降低人工复核页上逐条拖时间轴、调起止时间和手改类型的成本，当前建议从“单次识别 + 人工冻结”升级为：

`Seedance 初筛 -> DeepSeek 二次复核 -> 候选入库 -> 后台抽检`

两阶段职责分工如下：

### Seedance 初筛

职责：

- 从结构化 transcript 小窗口中找出“疑似高光”
- 给出较高召回率的候选
- 输出候选的初始时间段、类型、标题和互动建议

要求：

- 宁可略有冗余，也不要漏掉真正高光
- 单条时间段优先控制在 `3s ~ 12s`
- 同一窗口内尽量只报最强的 `1 ~ 2` 个点
- 禁止把整段背景铺垫、世界观说明打成高光

### DeepSeek 二次复核

职责：

- 判断 Seedance 候选是否真的值得进入高光池
- 调整开始时间、结束时间
- 缩短过长区间
- 去掉重复候选
- 修正类型、标题、描述和互动选项
- 直接给出更适合客户端的交互时间窗口

要求：

- 以“精准命中情绪峰值”为目标
- 可以直接拒绝某条候选
- 输出必须带 `reviewDecision`

### 为什么这样拆

原因：

- Seedance 更适合做第一轮“广撒网”
- DeepSeek 更适合做第二轮“精修与结构化”
- 这样能显著降低人工复核工作量
- 后台页面从“逐条手工修时间”变成“抽检和少量微调”

## 7.5 DeepSeek 复核后的决策类型

建议固定为：

- `approve`
- `reject`
- `merge`
- `revise`

含义：

- `approve`：直接认可该候选，可入候选池
- `reject`：不认为这是值得保留的高光
- `merge`：该候选应和别的候选合并
- `revise`：保留，但必须调整时间区间或字段

## 8. 高光类型冻结建议

第一版不建议类型过多，建议先冻结为 4 类：

- `feel_good`：爽点
- `reversal`：反转
- `conflict`：冲突
- `sweet`：甜蜜

原因：

- 便于客户端做清晰映射
- 便于后台审核
- 便于答辩讲解

---

## 9. 类型与交互表现映射建议

这一版不再把 `templateId` 作为前台可编辑、可配置的主概念。

冻结规则改为：

- Web 后台只展示和编辑 `highlightType`
- 后端根据 `highlightType` 做一一映射
- 再根据 `intensity` 决定使用“轻量选项态”还是“强视觉组件态”

固定映射如下：

- `feel_good -> shuang_burst`
  - 视觉意象：`爽` 字组件
- `reversal -> woc_flip`
  - 视觉意象：`卧槽` 字组件
- `conflict -> flame_rage`
  - 视觉意象：火焰燃烧组件
- `sweet -> heart_bloom`
  - 视觉意象：爱心组件

原则：

- 类型和表现一一对应，不再让前端单独选模板
- 模板是内部实现概念，不是运营配置概念
- 如果后续扩展，只允许在类型稳定后增加同类型的强弱态，不再横向新增一堆通用模板

## 10. 互动选项建议

LLM 产出的 `interactionOptions` 只需要给出 2-4 个短选项。

要求：

- 单个选项尽量 2-6 字
- 优先情绪表达，不要长句解释
- 优先像观众会点的一键弹幕，而不是像策划说明或理性建议
- 避免书面化、过度正常、过度完整的表达
- 尽量带一点网感、共鸣感、站队感
- 需要和剧情立场一致，不要出现“中立理性吃瓜”这类削弱情绪峰值的话术

不推荐：

- `从长计议`
- `围观后续发展`
- `期待后续逆袭`
- `中立理性吃瓜`

更推荐：

- `别忍了`
- `太敢说了`
- `给我还钱`
- `护住孩子`
- `磕到了`
- `这波要逆天`

## 10.1 强度与交互形态映射

为了让互动既有层次感又不挡剧情，当前冻结规则为：

- `intensity 1-2`
  - 形态：轻量选项态
  - 目标：轻吐槽、轻共鸣、小反应
  - 表现：以一键选项、轻浮层、轻弹幕为主，不强推大组件
- `intensity 3`
  - 形态：标准组件态
  - 目标：让用户明确感知到这一类高光已到来
  - 表现：进入对应类别的标准组件
- `intensity 4-5`
  - 形态：强视觉组件态
  - 目标：在真正情绪峰值时制造强反馈
  - 映射示例：
    - `feel_good -> shuang_burst`
    - `reversal -> woc_flip`
    - `conflict -> flame_rage`
    - `sweet -> heart_bloom`

这层映射由后端和客户端统一决定，不要求模型自由决定，也不要求后台运营单独选模板。

示例：

### 爽点

- 爽了
- 打脸
- 继续反杀

### 反转

- 啊？
- 没想到
- 细思极恐

### 冲突

- 别忍
- 开怼
- 站她

### 甜蜜

- 磕到了
- 太甜了
- 锁死

## 11. LLM 输入输出建议

## 11.1 输入内容

建议输入：

- 当前片段文本
- 当前片段时间范围
- 前后文摘要或上下文片段
- 角色名（如可提取）
- 剧种风格提示

如果使用双阶段模型，则需要拆成两套上下文。

### Seedance 初筛上下文

Seedance 的上下文建议按“窗口级 transcript”组织，而不是把整集一次性塞进去。

推荐输入单元：

- `dramaId`
- `episodeId`
- `episodeTitle`
- `episodeSummary`
- `mainGenre`
- `chunkStartMs`
- `chunkEndMs`
- 当前窗口的 `segments`
- 窗口前 `2 ~ 4` 条上下文片段
- 窗口后 `2 ~ 4` 条上下文片段

Seedance 的主要任务是“从这一小段里找高光”，因此上下文要满足：

- 能理解当前冲突/反转/甜蜜是否成立
- 但又不能大到让模型失焦

建议窗口大小：

- 主窗口：`18 ~ 28` 个 segment
- overlap：`3 ~ 5` 个 segment

### DeepSeek 复核上下文

DeepSeek 的上下文不是整段 transcript，而是：

- 当前候选本身
- 候选命中的 supporting segments
- 候选前后各 `2 ~ 5` 条 segment
- 当前集 metadata
- 该集已通过候选列表（用于避免重复）
- 类型与固定互动表现映射规则

DeepSeek 的任务不是“再找一次高光”，而是：

- 判断候选是否成立
- 修正时间
- 缩短区间
- 拒绝噪声候选

因此它的上下文组织必须更聚焦、更结构化。

## 11.2 输出格式

必须要求模型输出结构化 JSON。

建议输出为数组形式：

```json
[
  {
    "episodeId": "ep_001_03",
    "startTimeMs": 82400,
    "endTimeMs": 90500,
    "highlightType": "reversal",
    "title": "身份反转曝光",
    "description": "主角隐藏身份在众人面前被揭开，剧情出现明显反转。",
    "intensity": 4,
    "interactionOptions": ["啊？", "没想到", "细思极恐"],
    "reason": "情绪冲击强，观众表达欲高",
    "confidence": 0.86
  }
]
```

## 11.3 Seedance 初筛输出建议

Seedance 初筛输出建议字段如下：

```json
[
  {
    "episodeId": "ep_001_02",
    "startTimeMs": 77700,
    "endTimeMs": 85740,
    "highlightType": "conflict",
    "title": "娘家吞烈士恤银翻脸驱赶求助女主",
    "description": "女主求助娘家却被翻脸驱赶，冲突强烈。",
    "intensity": 4,
    "interactionOptions": ["太过分了", "别再求了", "直接翻脸"],
    "reason": "该段有明确利益冲突和情绪爆发，适合站队互动。",
    "confidence": 0.88,
    "supportingSegmentIds": ["seg_0029", "seg_0030"]
  }
]
```

Seedance 阶段要求：

- 允许略有冗余
- 不要求完全精准
- 但禁止输出明显铺垫段或超长段

## 11.4 DeepSeek 二次复核输出建议

DeepSeek 的输出建议字段如下：

```json
[
  {
    "candidateIndex": 0,
    "reviewDecision": "revise",
    "approved": true,
    "startTimeMs": 78120,
    "endTimeMs": 84200,
    "interactionStartMs": 78120,
    "interactionAppearMs": 78720,
    "interactionEndMs": 87200,
    "highlightType": "conflict",
    "title": "娘家翻脸赶走上门求助的女主",
    "description": "女主带着孩子求助娘家，却被翻脸驱赶，情绪峰值明确。",
    "interactionOptions": ["太过分了", "别再求了", "直接翻脸"],
    "reason": "该候选成立，适合进入高光池。",
    "reviewReason": "原候选前段包含铺垫，已压缩到真正翻脸爆发点。",
    "confidence": 0.93,
    "supportingSegmentIds": ["seg_0029", "seg_0030"]
  }
]
```

说明：

- `candidateIndex` 用于对应 Seedance 输入候选
- `reviewDecision` 用于标记是批准、拒绝还是修正
- `interactionStartMs / interactionEndMs` 是客户端允许重复点击的互动窗口
- `interactionAppearMs` 是互动组件真正展示出来的时间点

## 11.5 初步高光识别上下文如何组织

这是当前这条链路里最关键的问题之一，建议固定为下面的结构。

### 不建议的组织方式

不要这样做：

- 整集 transcript 一次性全部丢给 Seedance
- 只给纯文本，不给时间戳
- 只给一句台词，不给前后文

这些方式会导致：

- 模型容易漏高光
- 时间点不准
- 类型判断飘

### 推荐的组织方式

每次调用 Seedance 时，传入一个“窗口级上下文包”：

```json
{
  "dramaId": "drama_001",
  "episodeId": "ep_001_02",
  "episodeTitle": "第2集",
  "episodeSummary": "女主刚穿越醒来，面对孩子和灾荒处境，冲突逐步升级。",
  "mainGenre": "爽文",
  "chunkStartMs": 72000,
  "chunkEndMs": 92000,
  "segments": [
    {
      "segmentId": "seg_0026",
      "startTimeMs": 72100,
      "endTimeMs": 74120,
      "text": "……"
    }
  ],
  "previousContext": [
    {
      "segmentId": "seg_0024",
      "text": "……"
    }
  ],
  "nextContext": [
    {
      "segmentId": "seg_0032",
      "text": "……"
    }
  ]
}
```

组织原则：

- `segments` 是主要识别材料
- `previousContext / nextContext` 只做辅助理解
- 时间戳必须保留，便于回写高光区间
- `episodeSummary` 帮助模型理解这一段的戏剧目标
- `mainGenre` 帮助模型理解爽点/反转的风格偏好

### 推荐的切片策略

- transcript 先按自然 segment 切开
- 每次送 `18 ~ 28` 个 segment
- 相邻窗口保留 `3 ~ 5` 个 overlap
- 每个窗口独立识别
- 最后合并候选并去重

这样组织的好处是：

- 精度更稳定
- 能回到时间轴
- 后续 DeepSeek 复核也容易接

## 11.3 服务端要求

服务端或脚本层必须：

- 校验 JSON 合法性
- 校验字段完整性
- 不合法时重试一次
- 仍失败时标记本集识别失败

## 12. 后台审核流程

后台建议提供一个最小高光审核工作流：

1. 查看某集候选高光列表
2. 点击高光可查看：
   - 时间戳
   - 标题
   - 类型
   - 互动表现
   - 推荐按钮
   - LLM 理由
3. 播放对应片段进行人工确认
4. 可修改：
   - 开始/结束时间
   - 高光类型
   - 标题文案
   - 互动表现
   - 互动选项
5. 操作：
   - `确认`
   - `禁用`

后台审核结果定义：

- `candidate -> confirmed`
- `candidate -> disabled`

- `startTimeMs`
- `endTimeMs`

- `createdAt`

其中：

  - `ai`
  - `manual`
  - `ai_manual`

- `status` 建议取值：
  - `candidate`
  - `confirmed`
  - `disabled`

## 14. 客户端消费规则

客户端不负责识别高光，只负责消费确认后的标签。

客户端规则建议固定为：

1. 只请求 `confirmed` 高光
2. 基于时间轴判断命中
3. 命中后按 `highlightType` 选择对应类别的互动表现
4. 再按 `intensity / visualEffectType` 决定是：
   - 轻量选项态
   - 标准组件态
   - 还是强视觉冲击型互动组件
5. 点击互动后上报事件
6. 根据服务端返回统计显示热度或历史互动

客户端不要做：

- 类型重判
- 表现改写
- 本地自动增删高光

---

## 15. 与互动数据回流的关系

互动数据可以作为后续高光优化依据，但第一版不建议直接影响已确认标签。

推荐策略：

- 某时间点互动数异常集中
- 进入 `候选池`
- 后续由后台或离线脚本决定是否新增高光

即：

`互动数据 -> 候选线索`

而不是：

`互动数据 -> 直接修改线上高光`

---

## 16. 实现顺序建议

建议按以下顺序实现：

### 第一步：定义高光 schema

输出：

- 高光字段结构

### 第二步：完成文本提取

输出：

- 统一结构化字幕/逐字稿

建议拆成 3 个可执行小步骤：

1. `mp4 -> audio`

3. `subtitle / transcript -> Drama Pulse transcript JSON`

- `scripts/highlight/extract_audio.py`

输出：

### 第七步：互动数据反哺候选池

输出：

- 热点回流机制

## 17. 风险点

## 17.1 全自动误判风险

风险：

- 模型会把普通剧情误判为高光
- 模型会漏掉真正高光

控制：

- 必须经过人工冻结

## 17.2 时间戳不准

风险：

- 字幕时间和实际情绪峰值存在偏移

控制：

- 后台支持微调起止时间

## 17.3 交互表现过多导致客户端复杂

风险：

- 表现种类越多，客户端越容易乱

控制：

- 第一版严格收敛到 4 类高光对应 4 类核心表现

## 17.4 文本来源质量不稳定

风险：

- ASR 错字、断句问题影响候选质量

控制：

- 优先原字幕，其次 ASR

## 17.5 外部 ASR 工具授权与维护边界

风险：

- 直接复制第三方开源工具源码，可能带来许可证和维护负担

控制：

- 第一版优先做“外部工具调用 + 内部结构化转换”
- 不把第三方通用 GUI 工具直接并入主业务代码

### 18.1 媒体预处理脚本

### 18.2 Transcript 归一化脚本

负责：

- 将 `srt / json / txt` 等结果统一整理成内部 transcript JSON

建议文件：

- `scripts/highlight/normalize_transcript.py`

### 18.3 LLM 高光识别脚本

负责：

- `detect_highlights.py`
  - 读取 transcript JSON
  - 生成 Seedance 初筛 prompt
  - 调用 Seedance
  - 输出初筛候选 JSON
- `review_highlights.py`
  - 读取初筛候选和 transcript
  - 生成 DeepSeek 复核 prompt
  - 调用 DeepSeek
  - 输出复核后的候选 JSON

建议文件：

- `scripts/highlight/detect_highlights.py`
- `scripts/highlight/review_highlights.py`

### 18.4 结构校验脚本

负责：

- 校验模型输出 JSON 是否合法
- 检查字段完整性、枚举值、时间范围

建议文件：

- `scripts/highlight/validate_candidates.py`

### 18.5 候选入库脚本

负责：

- 将校验通过的候选高光写入数据库
- 默认状态为 `candidate`

建议文件：

- `scripts/highlight/import_candidates.py`

### 18.6 后台审核页面

负责：

- 候选高光列表
- 片段回看
- 时间调整
- 类型、选项修正
- 确认 / 禁用

也就是说：

本方案后续必须配合具体代码实现，不建议只停留在文档和口头流程层。

## 19. 当前冻结结论

本项目高光方案最终冻结为：

- 使用文本驱动识别，不走端上实时视频理解
- 以 `AI 候选 + 人工冻结` 为主
- 客户端只消费 `confirmed` 高光
- 第一版先冻结 4 种高光类型
- 第一版不把模板暴露给后台单独编辑，只在内部保留实现映射
- 互动数据只做后续候选线索，不自动改线上标签
