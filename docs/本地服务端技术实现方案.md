# Drama Pulse 本地服务端技术实现方案

## 1. 文档目的

本文档基于 [技术方案初稿.md](/Users/a0000/Desktop/项目文件/drama-pulse/docs/技术方案初稿.md) 与 [开发顺序与任务拆解初稿.md](/Users/a0000/Desktop/项目文件/drama-pulse/docs/开发顺序与任务拆解初稿.md)，单独拆出 `server/` 的技术实现方案，作为服务端开发、数据库设计、接口实现和联调验收的直接依据。

本文档目标：

- 明确本地服务端的职责边界

- 明确每个服务模块的输入、输出与依赖

- 明确异步任务、静态资源、鉴权、错误处理与测试策略
- 支撑其他 agent 并行开发服务端模块

## 2. 服务端范围与边界

### 2.1 服务端负责的内容

- 提供短剧、剧集、继续观看摘要查询
- 提供高光点下发和高光统计查询
- 接收互动事件并聚合历史互动数据
- 提供尾集固定分支配置
- 提供自定义分支任务创建、查询、点赞和评论
- 提供本地 Web 管理后台所需的管理接口
- 提供静态资源 `path -> url` 映射
- 编排 AI 高光导入与分支生成任务

### 2.2 服务端不负责的内容

- Android 端播放器逻辑
- Web 管理后台页面渲染
- 实时多人互动
- 实时视频理解
- 自定义分支实时视频生成

### 2.3 第一版必须坚持的边界

- 只支持本地部署，不做公网账户体系
- 只支持尾集分支，不支持中途关键节点分支
- 只消费 `confirmed` 高光，不在播放期间动态改标签
- 自定义分支第一版输出文本剧情和分镜结构，不依赖实时视频生成
- 评论能力仅存在于分支结果页

## 3. 冻结技术栈

- 运行时：`Node.js 20 LTS`
- 语言：`TypeScript`
- HTTP 框架：`Fastify`
- 数据库：`SQLite`
- ORM：`Prisma`
- 请求与输出校验：`Zod`
- 包管理：`pnpm`

冻结原因：

- 本地部署轻量
- 单人开发效率高
- 与后台前端类型口径更容易统一
- 足够承载当前 I/O 密集型服务场景

## 4. 运行形态

```text
Android Client
   |
   | HTTP
   v
Fastify Server
   |-- SQLite (结构化数据)
   |-- Local File Storage (videos/assets/data/exports)
   |-- In-process Task Queue (BranchTask)
   |-- Admin API
   |-- Static File Routes
```

运行要求：

- 服务端监听 `0.0.0.0`
- 默认端口建议 `8787`
- 管理后台与服务端运行在同一台电脑
- Android 真机通过局域网 IP 访问服务端

│   │   ├── highlight/
│   │   ├── interaction/

│   │   ├── admin/
│   │   ├── assets/
│   │   └── progress/
│   ├── services/
│   │   ├── userIdentity/
│   │   ├── stats/
│   │   ├── taskQueue/
│   │   ├── ai/

目录原则：

- `modules/` 负责按业务领域组织路由、控制器、服务
- `services/` 放跨模块共享能力，不承载 HTTP 层
- `shared/` 放通用能力和基础设施
- `scripts/` 放服务端直接执行的本地维护脚本

## 6. 分层与职责

### 6.1 推荐分层

- `route` 层：定义 Fastify 路由、入参校验、响应转换
- `controller` 层：负责流程编排和调用模块 service
- `service` 层：负责业务规则
- `repository` 层：负责 Prisma 读写
- `shared service` 层：负责聚合统计、资源映射、任务队列等通用能力

### 6.2 约束

- 路由层不直接写 Prisma 查询
- repository 不直接处理 HTTP 请求对象
- service 不直接依赖 Fastify reply
- 所有接口输出统一走响应包装器

## 7. 模块拆解

## 7.1 Content Module

职责：

- 提供短剧列表
- 提供某部短剧的剧集列表
- 提供单集详情
- 提供首页继续观看摘要

读表：

- `Drama`
- `Episode`
- `WatchProgress`

核心接口：

- `GET /dramas`
- `GET /dramas/:dramaId/episodes`
- `GET /episodes/:episodeId`

实现要点：

- `GET /dramas` 返回主打剧、备选剧和继续观看摘要
- 按 `display_order` 排序
- 剧封面、视频地址统一由资源映射服务输出 `url`

## 7.2 Highlight Module

职责：

- 查询指定剧集的高光列表
- 查询指定高光的统计快照
- 后台高光启停、修正和模板调整

读写表：

- `Highlight`
- `HighlightStats`

核心接口：

- `GET /episodes/:episodeId/highlights`
- `GET /highlights/:highlightId/stats`
- `GET /admin/highlights`
- `PATCH /admin/highlights/:highlightId`
- `POST /admin/highlights/:highlightId/enable`
- `POST /admin/highlights/:highlightId/disable`

实现要点：

- 客户端只返回 `status=confirmed`
- 后台返回全部状态，支持筛选 `candidate/confirmed/disabled`
- 后台修改高光时必须允许改：
  - 起止时间
  - 类型
  - 强度
  - 模板
  - 互动选项
  - 状态

## 7.3 Interaction Module

职责：

- 接收高光互动事件
- 更新聚合统计
- 返回历史互动统计快照
- 提供后台互动记录查询

读写表：

- `InteractionEvent`
- `HighlightStats`

核心接口：

- `POST /interactions`
- `GET /admin/interactions`

实现要点：

- 服务端根据 `device_id` 生成稳定匿名 `user_id`
- 接口成功后返回该高光的最新聚合统计
- 聚合规则：
  - `total_count` 按事件数累计
  - `unique_device_count` 按设备去重
  - `top_option` 根据计数排序
  - `heat_level` 根据阈值映射

## 7.4 Branch Module

职责：

- 返回尾集固定分支配置
- 创建与查询自定义分支任务
- 提供分支点赞和评论

读写表：

- `BranchOption`
- `BranchTask`
- `BranchComment`
- `BranchLike`
- `Episode`

核心接口：

- `GET /episodes/:episodeId/branch-options`
- `POST /branch-tasks`
- `GET /branch-tasks/:taskId`
- `GET /users/:userId/branch-tasks`
- `POST /branch-tasks/:taskId/likes`
- `POST /branch-tasks/:taskId/comments`
- `GET /branch-tasks/:taskId/comments`

实现要点：

- 只允许在 `is_final_episode=true` 的剧集上创建分支
- 固定分支第一版固定返回 2 条 `active` 记录
- 点赞按 `user_id + branch_task_id` 唯一约束控制
- 评论分页按 `created_at DESC`

## 7.5 Progress Module

职责：

- 记录与查询继续观看进度

读写表：

- `WatchProgress`

核心接口：

- `GET /users/:userId/watch-progress`
- `POST /users/:userId/watch-progress`

实现要点：

- 以 `user_id + drama_id` 为唯一键
- 每次播放退出、切集或手动上报时更新

## 7.6 Admin Module

职责：

- 封装后台管理能力
- 校验 `ADMIN_TOKEN`
- 提供资源配置、数据初始化、任务重试

核心接口：

- `GET /admin/dramas`
- `GET /admin/episodes`
- `GET /admin/highlights`
- `PATCH /admin/highlights/:highlightId`
- `GET /admin/interactions`
- `GET /admin/branch-tasks`
- `POST /admin/branch-tasks/:taskId/retry`
- `POST /admin/demo/reset`
- `POST /admin/assets/config`

实现要点：

- 所有 `/admin/*` 接口统一挂鉴权钩子
- 同时限制来源为 `localhost` 或局域网
- `demo/reset` 仅重置 `runtime data`

## 7.7 Assets Module

职责：

- 管理静态资源对外访问
- 统一做 `path -> url`
- 校验资源是否存在

暴露规则：

- `/static/videos/*`
- `/static/assets/*`
- `/static/exports/*`

映射规则：

- `videos/<drama_title>/第N集.mp4` -> `http://<host>:8787/static/videos/...`
- `assets/dramas/covers/<drama_id>.jpg` -> `http://<host>:8787/static/assets/...`
- `assets/dramas/branches/<episode_id>/<option_id>.mp4` -> `http://<host>:8787/static/assets/...`

## 8. 数据库实现方案

### 8.1 数据源

- SQLite 文件：`data/app.db`
- Prisma schema：`server/prisma/schema.prisma`
- 种子数据脚本：`server/prisma/seed.ts`

### 8.2 核心表

- `Drama`
- `Episode`
- `Highlight`
- `InteractionEvent`
- `HighlightStats`
- `BranchOption`
- `BranchTask`
- `BranchComment`
- `BranchLike`
- `WatchProgress`

### 8.3 关键索引

- `Episode(drama_id, episode_no)` 唯一索引
- `Highlight(episode_id, start_time_ms)` 普通索引
- `InteractionEvent(highlight_id, server_timestamp)` 普通索引
- `BranchTask(episode_id, created_at)` 普通索引
- `BranchLike(branch_task_id, user_id)` 唯一索引
- `WatchProgress(user_id, drama_id)` 唯一索引

### 8.4 表关系要求

- 删除 `Drama` 时默认不做级联物理删除，避免误删演示数据
- 删除 `Episode` 时需要保护关联高光和分支，建议后台只做 `status=hidden`
- `HighlightStats` 可作为聚合快照表独立更新

## 9. 接口实现规范

### 9.1 统一响应结构

```json
{
  "code": 0,
  "message": "ok",
  "data": {}
}
```

### 9.2 错误码建议

- `0`：成功
- `40001`：请求参数错误
- `40004`：资源不存在
- `40009`：状态不允许
- `40101`：后台鉴权失败
- `40301`：来源不允许
- `50001`：服务内部错误
- `50002`：AI 任务执行失败

### 9.3 客户端上下文约定

为了让继续观看、互动去重、分支点赞评论等能力都能关联到稳定匿名用户，客户端请求上下文统一约定如下：

- Android 客户端请求统一携带 `x-device-id` 请求头
- 写接口 body 中若同时存在 `deviceId`，服务端优先校验其与请求头一致
- 服务端根据 `x-device-id` 生成或查找稳定匿名 `user_id`
- `GET /dramas` 依赖 `x-device-id` 返回继续观看摘要
- 后台接口不使用 `x-device-id`，只走 `ADMIN_TOKEN`

### 9.4 请求校验

所有接口入参统一使用 Zod：

- `params schema`
- `query schema`
- `body schema`

所有后台出参和 AI 结果出参同样经过 Zod 结构校验。

## 10. 异步任务实现方案

### 10.1 第一版策略

- 采用进程内单任务队列
- 默认并发度 `1`
- 任务状态：
  - `pending`
  - `running`
  - `success`
  - `failed`
  - `timeout`
  - `blocked`

### 10.2 队列职责

- 拉取 `pending` 的 `BranchTask`
- 更新为 `running`
- 调用 AI 编排器生成结果
- 结果通过 Zod 校验
- 成功写回 `result_title/result_hook/result_story/storyboard_json`
- 失败写回 `fail_reason`

### 10.3 重试规则

- 仅允许后台对 `failed/timeout` 任务触发重试
- 重试前清空旧的 `started_at/finished_at/fail_reason`
- 重试次数第一版建议限制为 `3`

## 11. 管理后台鉴权与访问控制

### 11.1 鉴权方式

- 使用固定 `ADMIN_TOKEN`
- 后台请求头：`Authorization: Bearer <ADMIN_TOKEN>`

### 11.2 来源限制

- 允许 `localhost`
- 允许局域网来源
- 禁止默认暴露到公网

### 11.3 风险控制

- `ADMIN_TOKEN` 只放 `.env`
- `.env` 不进入仓库
- 演示时避免在日志和截图中暴露 token

## 12. 资源配置持久化方案

### 12.1 配置落盘位置

后台资源配置不直接改数据库，第一版建议写入：

- `config/resource-paths.local.json`

建议结构：

```json
{
  "videosRoot": "../videos",
  "assetsRoot": "../assets",
  "exportsRoot": "../data/exports"
}
```

### 12.2 生效策略

- 服务端启动时先读 `.env`
- 若存在 `config/resource-paths.local.json`，则以该文件覆盖对应根路径
- `POST /admin/assets/config` 成功后写回该文件
- 写回成功后刷新服务端内存中的资源映射配置

## 13. 配置与环境变量

建议最少包含以下配置：

```env
PORT=8787
HOST=0.0.0.0
DATABASE_URL=file:../data/app.db
ADMIN_TOKEN=replace-me
STATIC_ROOT=..
VIDEOS_ROOT=../videos
ASSETS_ROOT=../assets
EXPORTS_ROOT=../data/exports
ALLOWED_ORIGINS=http://localhost:5173,http://127.0.0.1:5173
```

说明：

- `STATIC_ROOT` 用于统一拼接相对路径
- `ALLOWED_ORIGINS` 控制本地后台跨域访问

## 14. 日志、错误处理与可观测性

### 13.1 日志分层

- `info`：启动、路由注册、任务状态流转
- `warn`：参数异常、资源缺失、非法状态请求
- `error`：数据库错误、AI 调用失败、未捕获异常

### 13.2 错误处理

- Fastify 全局错误处理器统一包装错误响应
- 业务错误走自定义错误类
- Prisma 错误统一映射为业务可读错误

### 13.3 关键日志点

- `POST /interactions`
- `POST /branch-tasks`
- 分支任务执行开始/结束
- `POST /admin/demo/reset`
- `POST /admin/branch-tasks/:taskId/retry`

## 15. 测试策略

### 14.1 测试分层

- 单元测试：service、resource mapper、stats calculator
- 集成测试：Fastify 路由 + SQLite 测试库
- 脚本测试：seed、reset、importHighlights

### 14.2 必测场景

- `GET /dramas` 返回主打剧、备选剧、继续观看摘要
- `GET /episodes/:id/highlights` 只返回 `confirmed`
- `POST /interactions` 能写入事件并返回统计
- 非尾集 `POST /branch-tasks` 被拒绝
- `POST /branch-tasks/:taskId/likes` 幂等控制生效
- `/admin/*` 无 token 时拒绝
- `demo/reset` 只清空运行期数据

### 14.3 联调前验收命令

建议提供：

```bash
pnpm lint
pnpm test
pnpm prisma migrate dev
pnpm prisma db seed
pnpm dev
```

## 16. 开发顺序建议

建议其他 agent 按如下顺序接力：

1. 初始化 `server/` 工程和 `prisma/schema.prisma`
2. 完成 `Drama/Episode/WatchProgress` 数据表和内容接口
3. 完成 `Highlight/HighlightStats` 表与高光接口
4. 完成 `InteractionEvent` 表、聚合逻辑与互动接口
5. 完成 `BranchOption/BranchTask/BranchComment/BranchLike` 及分支接口
6. 完成 `Admin` 鉴权与管理接口
7. 完成静态资源映射和本地脚本
8. 完成异步任务与测试

## 17. Agent 分工建议

如果分给多个 agent，建议这样切：

- Agent A：Prisma schema + seed + content/progress 模块
- Agent B：highlight + interaction 模块
- Agent C：branch + taskQueue + comment/like 模块
- Agent D：admin 鉴权、admin 接口、demo reset、assets config

要求：

- 所有 agent 都必须遵守本文档的字段口径
- 不允许自行更改主键命名、状态枚举和路径规则

## 18. 完成标准

当满足以下条件时，本地服务端模块算完成：

- `server/` 可独立启动
- SQLite 建表、迁移、种子数据可运行
- Android 客户端所需接口全部可调
- Web 管理后台所需接口全部可调
- 固定分支、自定义分支、点赞评论闭环可跑通
- 本地静态资源可通过服务端 URL 访问
- 管理后台鉴权与演示数据重置能力可用
