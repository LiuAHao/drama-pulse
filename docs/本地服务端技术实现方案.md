# Drama Pulse 本地服务端技术实现方案

## 1. 文档目的

本文档基于 [技术方案初稿.md](/Users/a0000/Desktop/项目文件/drama-pulse/docs/技术方案初稿.md) 与 [开发顺序与任务拆解初稿.md](/Users/a0000/Desktop/项目文件/drama-pulse/docs/开发顺序与任务拆解初稿.md)，单独拆出 `server/` 的技术实现方案，作为服务端开发、数据库设计、接口实现和联调验收的直接依据。

说明：

- 文中涉及固定分支资源时，请以当前实现中的 `assets/generated/fixed-branches/` 为主消费结果路径
- `assets/dramas/branches/` 仍可能作为旧资源路径或兼容字段保留在部分 seed/历史数据中

本文档目标：

- 明确本地服务端的职责边界
- 冻结服务端目录结构和分层方案
- 明确每个服务模块的输入、输出与依赖
- 明确数据库表和接口的映射关系
- 明确异步任务、静态资源、鉴权、错误处理与测试策略
- 支撑其他 agent 并行开发服务端模块

## 2. 服务端范围与边界

### 2.1 服务端负责的内容

- 提供短剧、剧集、继续观看摘要查询
- 提供高光点下发和高光统计查询
- 接收互动事件并聚合历史互动数据
- 提供尾集固定分支配置
- 提供自定义分支任务创建、查询、点赞和评论
- 提供本地 Web 管理后台所需的管理接口
- 提供静态资源 `path -> url` 映射
- 编排 AI 高光导入与分支生成任务

### 2.2 服务端不负责的内容

- Android 端播放器逻辑
- Web 管理后台页面渲染
- 实时多人互动
- 实时视频理解
- 自定义分支实时视频生成

### 2.3 第一版必须坚持的边界

- 只支持本地部署，不做公网账户体系
- 只支持尾集分支，不支持中途关键节点分支
- 只消费 `confirmed` 高光，不在播放期间动态改标签
- 自定义分支第一版输出文本剧情和分镜结构，不依赖实时视频生成
- 评论能力仅存在于分支结果页

## 3. 冻结技术栈

- 运行时：`Node.js 20 LTS`
- 语言：`TypeScript`
- HTTP 框架：`Fastify`
- 数据库：`SQLite`
- ORM：`Prisma`
- 请求与输出校验：`Zod`
- 包管理：`pnpm`

冻结原因：

- 本地部署轻量
- 单人开发效率高
- 与后台前端类型口径更容易统一
- 足够承载当前 I/O 密集型服务场景

## 4. 运行形态

```text
Android Client
   |
   | HTTP
   v
Fastify Server
   |-- SQLite (结构化数据)
   |-- Local File Storage (videos/assets/data/exports)
   |-- In-process Task Queue (BranchTask)
   |-- Admin API
   |-- Static File Routes
```

运行要求：

- 服务端监听 `0.0.0.0`
- 默认端口建议 `8787`
- 管理后台与服务端运行在同一台电脑
- Android 真机通过局域网 IP 访问服务端

## 6. 分层与职责

### 6.1 推荐分层

- `route` 层：定义 Fastify 路由、入参校验、响应转换
- `controller` 层：负责流程编排和调用模块 service
- `service` 层：负责业务规则
- `repository` 层：负责 Prisma 读写
- `shared service` 层：负责聚合统计、资源映射、任务队列等通用能力

### 6.2 约束

- 路由层不直接写 Prisma 查询
- repository 不直接处理 HTTP 请求对象
- service 不直接依赖 Fastify reply
- 所有接口输出统一走响应包装器

## 7. 模块拆解

## 7.1 Content Module

职责：

- 提供短剧列表
- 提供某部短剧的剧集列表
- 提供单集详情
- 提供首页继续观看摘要

读表：

- `Drama`
- `Episode`
- `WatchProgress`

核心接口：

- `GET /dramas`
- `GET /dramas/:dramaId/episodes`
- `GET /episodes/:episodeId`

实现要点：

- `GET /dramas` 返回主打剧、备选剧和继续观看摘要
- 按 `display_order` 排序
- 剧封面、视频地址统一由资源映射服务输出 `url`

## 7.2 Highlight Module

职责：

- 查询指定剧集的高光列表
- 查询指定高光的统计快照
- 后台高光启停、修正和类别/强度调整

读写表：

- `Highlight`
- `HighlightStats`

核心接口：

- `GET /episodes/:episodeId/highlights`
- `GET /highlights/:highlightId/stats`
- `GET /admin/highlights`
- `PATCH /admin/highlights/:highlightId`
- `POST /admin/highlights/:highlightId/confirm`
- `POST /admin/highlights/:highlightId/disable`

实现要点：

- 客户端只返回 `status=confirmed`
- 后台返回全部状态，支持筛选 `candidate/confirmed/disabled`
- 后台修改高光时必须允许改：
  - 起止时间
  - 交互窗口开始/结束
  - 交互真正出现时间
  - 类型
  - 强度
  - 互动选项
  - 状态
  - 角色与支撑片段等复核字段

## 7.3 Interaction Module

职责：

- 接收高光互动事件
- 更新聚合统计
- 返回历史互动统计快照
- 提供后台互动记录查询

读写表：

- `InteractionEvent`
- `HighlightStats`

核心接口：

- `POST /interactions`
- `GET /admin/interactions`

实现要点：

- 服务端根据 `device_id` 生成稳定匿名 `user_id`
- 接口成功后返回该高光的最新聚合统计
- 聚合规则：
  - `total_count` 按事件数累计
  - `unique_device_count` 按设备去重
  - `top_option` 根据计数排序
  - `heat_level` 根据阈值映射

## 7.4 Branch Module

职责：

- 返回尾集固定分支配置
- 创建与查询自定义分支任务
- 提供分支点赞和评论

读写表：

- `BranchOption`
- `BranchTask`
- `BranchComment`
- `BranchLike`
- `Episode`

核心接口：

- `GET /episodes/:episodeId/branch-options`
- `POST /branch-tasks`
- `GET /branch-tasks/:taskId`
- `GET /users/:userId/branch-tasks`
- `POST /branch-tasks/:taskId/likes`
- `POST /branch-tasks/:taskId/comments`
- `GET /branch-tasks/:taskId/comments`

实现要点：

- 只允许在 `is_final_episode=true` 的剧集上创建分支
- 固定分支第一版固定返回 2 条 `active` 记录
- 点赞按 `user_id + branch_task_id` 唯一约束控制
- 评论分页按 `created_at DESC`

## 7.5 Progress Module

职责：

- 记录与查询继续观看进度

读写表：

- `WatchProgress`

核心接口：

- `GET /users/:userId/watch-progress`
- `POST /users/:userId/watch-progress`

实现要点：

- 以 `user_id + drama_id` 为唯一键
- 每次播放退出、切集或手动上报时更新

## 7.6 User Module

职责：

- 承载用户资料基础信息
- 为客户端资料页提供服务端对齐能力

读写表：

- `UserProfile`

核心接口：

- `GET /users/:userId/profile`
- `PUT /users/:userId/profile`

实现要点：

- 仍然以 `x-device-id -> user_id` 的匿名稳定映射作为身份基础
- `GET` 在资料尚未写入时也应返回默认资料，而不是 404
- 第一版至少承载 `nickname / bio / avatarUrl`

## 7.7 Admin Module

职责：

- 封装后台管理能力
- 校验 `ADMIN_TOKEN`
- 提供资源配置、数据初始化、任务重试

核心接口：

- `GET /admin/dramas`
- `GET /admin/episodes`
- `GET /admin/highlights`
- `PATCH /admin/highlights/:highlightId`
- `GET /admin/interactions`
- `GET /admin/branch-tasks`
- `POST /admin/branch-tasks/:taskId/retry`
- `POST /admin/demo/reset`
- `POST /admin/assets/config`

实现要点：

- 所有 `/admin/*` 接口统一挂鉴权钩子
- 同时限制来源为 `localhost` 或局域网
- `demo/reset` 仅重置 `runtime data`

## 7.8 Assets Module

职责：

- 管理静态资源对外访问
- 统一做 `path -> url`
- 校验资源是否存在

暴露规则：

- `/static/videos/*`
- `/static/assets/*`
- `/static/exports/*`

映射规则：

- `videos/<drama_title>/第N集.mp4` -> `http://<host>:8787/static/videos/...`
- `assets/dramas/covers/<drama_id>.jpg` -> `http://<host>:8787/static/assets/...`
- `assets/dramas/branches/<episode_id>/<option_id>.mp4` -> `http://<host>:8787/static/assets/...`
- `assets/generated/fixed-branches/<episode_id>/<option_id>.json` -> `http://<host>:8787/static/assets/...`
- `assets/generated/fixed-branches/<episode_id>/manifest.json` -> `http://<host>:8787/static/assets/...`

## 8. 数据库实现方案

### 8.1 数据源

- SQLite 文件：`server/data/app.db`
- Prisma schema：`server/prisma/schema.prisma`
- 种子数据脚本：`server/prisma/seed.ts`

### 8.2 核心表

- `Drama`
- `Episode`
- `Highlight`
- `InteractionEvent`
- `HighlightStats`
- `BranchOption`
- `BranchTask`
- `BranchComment`
- `BranchLike`
- `WatchProgress`

### 8.3 关键索引

- `Episode(drama_id, episode_no)` 唯一索引
- `Highlight(episode_id, start_time_ms)` 普通索引
- `InteractionEvent(highlight_id, server_timestamp)` 普通索引
- `BranchTask(episode_id, created_at)` 普通索引
- `BranchLike(branch_task_id, user_id)` 唯一索引
- `WatchProgress(user_id, drama_id)` 唯一索引

### 8.4 表关系要求

- 删除 `Drama` 时默认不做级联物理删除，避免误删演示数据
- 删除 `Episode` 时需要保护关联高光和分支，建议后台只做 `status=hidden`
- `HighlightStats` 可作为聚合快照表独立更新

## 9. 接口实现规范

### 9.1 统一响应结构

```json
{
  "code": 0,
  "message": "ok",
  "data": {}
}
```

### 9.2 错误码建议

- `0`：成功
- `40001`：请求参数错误
- `40004`：资源不存在
- `40009`：状态不允许
- `40101`：后台鉴权失败
- `40301`：来源不允许
- `50001`：服务内部错误
- `50002`：AI 任务执行失败

### 9.3 客户端上下文约定

为了让继续观看、互动去重、分支点赞评论等能力都能关联到稳定匿名用户，客户端请求上下文统一约定如下：

- Android 客户端请求统一携带 `x-device-id` 请求头
- 写接口 body 中若同时存在 `deviceId`，服务端优先校验其与请求头一致
- 服务端根据 `x-device-id` 生成或查找稳定匿名 `user_id`
- `GET /dramas` 依赖 `x-device-id` 返回继续观看摘要
- 后台接口不使用 `x-device-id`，只走 `ADMIN_TOKEN`

### 9.4 请求校验

所有接口入参统一使用 Zod：

- `params schema`
- `query schema`
- `body schema`

所有后台出参和 AI 结果出参同样经过 Zod 结构校验。

### 9.5 Android 客户端联调返回契约补充

为了避免后端“接口已通”但 Android 端仍因字段不全、路径未映射、结构不一致而联调失败，客户端消费接口的返回契约在本文档中进一步冻结如下。

这部分是对前文模块设计的补充约束，优先级高于“只要能返回数据即可”的宽松实现。

#### 9.5.1 总原则

- 所有客户端接口继续统一返回：
  - `code`
  - `message`
  - `data`
- 所有资源类路径对 Android 客户端都必须返回“可直接请求”的 URL，不允许只返回本地相对路径后让客户端自行拼接
- 服务端若同时保留原始相对路径与 URL，允许双字段并存，但必须保证客户端当前依赖字段一定可用
- 同一类实体在不同接口中的核心字段命名必须保持一致，避免列表页和详情页口径不一致
- 只要 Android 端当前 DTO 已声明某字段，服务端就不应随意缺省该字段，即使值为空也要稳定返回

#### 9.5.2 `GET /dramas`

用途：

- 首页剧列表
- 首页 featured / alternatives
- 首页 continue watching 摘要
- 调试播放器入口的默认剧集来源

`data` 必须返回：

```json
{
  "featured": [
    {
      "id": "drama_001",
      "title": "荒年全村啃树皮，我有系统满仓肉",
      "description": "...",
      "coverPath": "http://<host>:8787/static/assets/...",
      "tagsJson": "[\"爽文\",\"穿越\"]",
      "mainGenre": "爽文",
      "isFeatured": true,
      "displayOrder": 1,
      "status": "active",
      "createdAt": "...",
      "updatedAt": "..."
    }
  ],
  "alternatives": [],
  "continueWatching": {
    "drama": {},
    "episode": {},
    "progressMs": 54179
  }
}
```

强约束：

- `featured` / `alternatives` 中的 `coverPath` 必须是完整 URL
- `continueWatching.drama.coverPath` 同样必须是完整 URL
- `continueWatching.episode` 必须至少包含 Android `EpisodeDto` 当前依赖的全部字段：
  - `id`
  - `dramaId`
  - `episodeNo`
  - `title`
  - `videoPath` 或 `videoUrl`
  - `durationMs`
  - `summary`
  - `isFinalEpisode`
  - `hasBranch`
  - `status`
  - `createdAt`
  - `updatedAt`
- 推荐继续保留 `videoPath`，并额外返回 `videoUrl`
  - 原因：Android 当前 mapper 采用 `videoUrl.ifEmpty { videoPath }`
  - 这样可兼容旧端与新端

#### 9.5.3 `GET /dramas/:dramaId/episodes`

用途：

- 进入播放页时获取某剧的剧集列表
- 选集面板
- 调试播放器 route 入口

返回数组中每一项必须包含：

- `id`
- `dramaId`
- `episodeNo`
- `title`
- `videoPath`
- `videoUrl`
- `durationMs`
- `summary`
- `isFinalEpisode`
- `hasBranch`
- `status`
- `createdAt`
- `updatedAt`

强约束：

- `videoPath` 必须是完整可访问 URL，不能是 `videos/...` 相对路径
- 推荐同时冗余返回 `videoUrl`，值与 URL 化后的 `videoPath` 一致
- 必须按 `episodeNo ASC` 排序

#### 9.5.4 `GET /episodes/:episodeId`

用途：

- 播放页首进详情加载
- 切集详情刷新

必须返回：

- `EpisodeDto` 全字段
- `videoPath` 必须 URL 化
- `videoUrl` 推荐同时返回

补充要求：

- 若服务端使用 `include: { drama: true }` 带出 `drama`，允许保留，但客户端当前不依赖，不得因此牺牲 `EpisodeDto` 主字段稳定性
- `summary` 必须稳定返回，不得缺省为 `undefined`

#### 9.5.5 `GET /episodes/:episodeId/highlights`

用途：

- 播放页高光触发
- 高光互动组件渲染
- 历史热度提示

客户端当前至少依赖以下字段：

- `id`
- `episodeId`
- `startTimeMs`
- `endTimeMs`
- `interactionStartMs`
- `interactionAppearMs`
- `interactionEndMs`
- `type`
- `title`
- `description`
- `intensity`
- `interactionOptionsJson`
- `stats`

为兼容当前 Android DTO，建议完整返回：

- `id`
- `episodeId`
- `startTimeMs`
- `endTimeMs`
- `interactionStartMs`
- `interactionAppearMs`
- `interactionEndMs`
- `type`
- `title`
- `description`
- `intensity`
- `templateId`
- `interactionOptionsJson`
- `visualEffectType`
- `source`
- `confidence`
- `status`
- `createdAt`
- `updatedAt`
- `stats`

其中 `stats` 必须稳定返回对象，不建议返回 `null`：

```json
{
  "totalCount": 0,
  "uniqueDeviceCount": 0,
  "heatLevel": 0,
  "topOption": ""
}
```

强约束：

- 客户端接口只返回 `status=confirmed`
- `interactionStartMs <= interactionAppearMs < interactionEndMs`
- 若库内为空，服务端必须在返回层兜底补齐默认时间
- `interactionOptionsJson` 必须始终是合法 JSON 字符串数组，即使为空也返回 `"[]"`

联调补充建议：

- 第一版继续保留 `templateId`，兼容旧端与后台
- 但客户端实际渲染以 `type + intensity + interaction*Ms` 为主
- 后端后续若去模板化，必须先更新 Android DTO 和后台类型口径

#### 9.5.6 `GET /highlights/:highlightId/stats`

用途：

- 高光单点热度刷新
- 互动后局部统计回读

必须返回：

- `totalCount`
- `uniqueDeviceCount`
- `heatLevel`
- `topOption`

建议与 `/episodes/:episodeId/highlights[].stats` 完全同结构。

#### 9.5.7 `POST /interactions`

用途：

- 点击高光组件后的统计写入

请求成功后返回的 `data` 必须直接是最新统计快照：

- `totalCount`
- `uniqueDeviceCount`
- `heatLevel`
- `topOption`

联调补充要求：

- 服务端当前 `interactionType` 即使仍使用旧枚举，也必须允许客户端稳定上报一个受控值并成功写入
- 第一版不要求返回整个高光对象，只返回统计即可
- 后续若要支持“服务端返回最新 recent texts / option stats”，应新增字段，不要破坏当前统计结构

#### 9.5.8 `GET /users/:userId/watch-progress`

用途：

- 恢复继续观看
- 调试播放器默认选集

每条记录必须返回：

- `id`
- `userId`
- `deviceId`
- `dramaId`
- `episodeId`
- `progressMs`
- `updatedAt`
- `drama`
- `episode`

其中：

- `drama` 必须满足 `DramaDto` 全字段返回
- `drama.coverPath` 必须 URL 化
- `episode` 必须满足 `EpisodeDto` 全字段返回
- `episode.videoPath` 必须 URL 化
- 推荐同时返回 `episode.videoUrl`

这一点是当前实现最容易遗漏的联调点：

- 不能只 `include drama/episode` 原样返回数据库字段
- 必须在响应层对嵌套的 `coverPath/videoPath` 一并做 `path -> url` 映射

#### 9.5.9 `POST /users/:userId/watch-progress`

用途：

- 播放退出
- 切集
- 周期性进度保存

建议返回：

- `id`
- `userId`
- `deviceId`
- `dramaId`
- `episodeId`
- `progressMs`
- `updatedAt`

第一版不强制要求回带嵌套 `drama/episode`，因为 Android 当前只依赖保存成功，不依赖返回体刷新 UI。

#### 9.5.10 `GET /users/:userId/profile`

用途：

- 恢复资料页默认展示
- 对齐昵称与简介的服务端状态

必须返回：

- `userId`
- `nickname`
- `bio`
- `avatarUrl`

联调要求：

- 未写入时返回服务端默认资料，不返回 404
- 若请求头带 `x-device-id`，必须校验其与 `userId` 匹配

#### 9.5.11 `PUT /users/:userId/profile`

用途：

- 保存昵称与简介

请求体至少支持：

- `nickname`
- `bio`
- `avatarUrl`

联调要求：

- `nickname` 不能为空
- `bio` 允许为空字符串
- 更新成功后直接返回最新资料快照

#### 9.5.12 `GET /episodes/:episodeId/branch-options`

用途：

- 尾集固定分支入口

每个 option 必须返回：

- `id`
- `episodeId`
- `title`
- `description`
- `resultType`
- `resultContentPath`
- `coverPath`
- `sortIndex`
- `status`

强约束：

- `resultContentPath` 必须 URL 化
- `coverPath` 必须 URL 化
- 非尾集直接返回空数组，不报错

#### 9.5.11 `POST /branch-tasks`

用途：

- 创建自定义分支任务

建议返回完整 `BranchTaskDto`：

- `id`
- `userId`
- `deviceId`
- `episodeId`
- `userPrompt`
- `status`
- `resultTitle`
- `resultHook`
- `resultStory`
- `storyboardJson`
- `resultTagsJson`
- `resultInteractionOptionsJson`
- `resultSource`
- `failReason`
- `retryCount`
- `createdAt`
- `startedAt`
- `finishedAt`

第一版创建成功后不强制要求带 `episode/drama/count`，因为客户端会继续轮询详情。

补充约定：

- `resultSource` 第一版固定写 `doubao`
- 高光识别链路使用 `DeepSeek`，不复用这里的分支任务来源字段

#### 9.5.12 `GET /branch-tasks/:taskId`

用途：

- 分支任务轮询详情
- 分支结果页展示

必须返回完整 `BranchTaskDto`，并满足：

- `episode` 至少返回 `EpisodeDto` 口径
- 若返回 `episode.videoPath`，必须 URL 化
- `drama` 若返回，必须保证 `coverPath` URL 化
- 必须补齐 `count`
  - `likes`
  - `comments`

注意：

- 当前 Prisma `_count` 与 Android `count` 字段名并不天然一致
- 返回层必须显式转换成：

```json
{
  "count": {
    "likes": 12,
    "comments": 5
  }
}
```

不能把 `_count` 原样透给客户端。

#### 9.5.13 `GET /users/:userId/branch-tasks`

用途：

- 个人页分支记录

建议返回 `BranchTaskDto[]`，至少包含：

- 任务基础字段
- `episode`
- `count`

若当前第一版暂不返回 `count`，需在文档中明确 Android 端个人页不可依赖该值；否则建议一次补齐，与详情页保持一致。

#### 9.5.14 `POST /branch-tasks/:taskId/likes`

必须返回：

```json
{
  "likeCount": 12
}
```

要求：

- 幂等
- 重复点赞不报错

#### 9.5.15 `POST /branch-tasks/:taskId/comments`

必须返回完整评论对象：

- `id`
- `branchTaskId`
- `userId`
- `deviceId`
- `content`
- `status`
- `createdAt`
- `updatedAt`

#### 9.5.16 `GET /branch-tasks/:taskId/comments`

必须返回分页结构：

- `items`
- `total`
- `page`
- `pageSize`
- `totalPages`

其中 `items[]` 每项满足 `BranchCommentDto`。

### 9.6 Android 联调补充约束

#### 9.6.1 资源 URL 化范围

以下字段凡出现在客户端接口中，都必须在服务端返回层完成 URL 化：

- `Drama.coverPath`
- `ContinueWatching.drama.coverPath`
- `ContinueWatching.episode.videoPath`
- `Episode.videoPath`
- `Episode.videoUrl`
- `BranchOption.resultContentPath`
- `BranchOption.coverPath`
- `BranchTask.episode.videoPath`
- `BranchTask.drama.coverPath`
- 未来若高光挂载视觉资源，也应在服务端完成 URL 化

#### 9.6.2 结构稳定性要求

- 不允许同一路由在“有数据”和“无数据”时返回不同字段形态
- 空数组返回 `[]`
- 空统计返回默认统计对象
- 空字符串字段返回 `""`
- JSON 字符串字段返回合法 JSON 文本，不返回 `null`

#### 9.6.3 向后兼容策略

在 Android 端与管理后台都还未完全去掉旧口径前，服务端建议暂时保留这些兼容字段：

- `Highlight.templateId`
- `Highlight.visualEffectType`
- `Episode.videoPath`
- `Episode.videoUrl`

兼容原则：

- 新端优先消费新口径
- 旧字段不立刻删除
- 等 Android / admin-web 同步完成后，再统一清理旧字段

#### 9.6.4 联调验收新增标准

除前文验收项外，新增以下联调通过标准：

- `GET /dramas` 中 featured / continueWatching 的封面与视频字段均可直接播放或展示
- `GET /users/:userId/watch-progress` 返回的嵌套 `drama/episode` 字段已完成 URL 化
- `GET /users/:userId/profile` / `PUT /users/:userId/profile` 已覆盖昵称与简介资料
- `GET /episodes/:episodeId/highlights` 返回的交互时间字段完整且满足顺序约束
- `GET /episodes/:episodeId/highlights` 返回的 `interactionOptionsJson` 已统一归一化为对象数组字符串，兼容旧库存量字符串数组
- `POST /interactions` 返回统计结构与高光列表中的 `stats` 完全同构
- `GET /branch-tasks/:taskId` 返回的点赞/评论计数已映射为 `count.likes/count.comments`
- Android 客户端无需自行拼接任何本地资源路径，即可直接联调所有主流程

## 10. 异步任务实现方案

### 10.1 第一版策略

- 采用进程内单任务队列
- 默认并发度 `1`
- 任务状态：
  - `pending`
  - `running`
  - `success`
  - `failed`
  - `timeout`
  - `blocked`

### 10.2 队列职责

- 拉取 `pending` 的 `BranchTask`
- 更新为 `running`
- 调用 AI 编排器生成结果
- 结果通过 Zod 校验
- 成功写回 `result_title/result_hook/result_story/storyboard_json`
- 失败写回 `fail_reason`

### 10.3 重试规则

- 仅允许后台对 `failed/timeout` 任务触发重试
- 重试前清空旧的 `started_at/finished_at/fail_reason`
- 重试次数第一版建议限制为 `3`

## 11. 管理后台鉴权与访问控制

### 11.1 鉴权方式

- 使用固定 `ADMIN_TOKEN`
- 后台请求头：`Authorization: Bearer <ADMIN_TOKEN>`

### 11.2 来源限制

- 允许 `localhost`
- 允许局域网来源
- 禁止默认暴露到公网

### 11.3 风险控制

- `ADMIN_TOKEN` 只放 `.env`
- `.env` 不进入仓库
- 演示时避免在日志和截图中暴露 token

## 12. 资源配置持久化方案

### 12.1 配置落盘位置

后台资源配置不直接改数据库，第一版建议写入：

- `config/resource-paths.local.json`

建议结构：

```json
{
  "videosRoot": "../videos",
  "assetsRoot": "../assets",
  "exportsRoot": "../data/exports"
}
```

### 12.2 生效策略

- 服务端启动时先读 `.env`
- 若存在 `config/resource-paths.local.json`，则以该文件覆盖对应根路径
- `POST /admin/assets/config` 成功后写回该文件
- 写回成功后刷新服务端内存中的资源映射配置

## 14. 日志、错误处理与可观测性

### 13.1 日志分层

- `info`：启动、路由注册、任务状态流转
- `warn`：参数异常、资源缺失、非法状态请求
- `error`：数据库错误、AI 调用失败、未捕获异常

### 13.2 错误处理

- Fastify 全局错误处理器统一包装错误响应
- 业务错误走自定义错误类
- Prisma 错误统一映射为业务可读错误

### 13.3 关键日志点

- `POST /interactions`
- `POST /branch-tasks`
- 分支任务执行开始/结束
- `POST /admin/demo/reset`
- `POST /admin/branch-tasks/:taskId/retry`
